# 2026-05-31 — Control flow, the no-GC data-type model, and value structs

A long session spanning three arcs: finishing the control-flow surface, a
research detour into how Beef's core data types and optional GC actually work
(which set our memory-management direction), and then the value-struct layout
sprint. Recording the reasoning, not the diffs.

---

## 1. Finishing control flow (calls → loops → switch)

### Intra-program call resolution: the two guards (commit 3e04c58)

**Why:** bare `Foo(args)` calls were lowering to a defaulted external `i64`
call. To get recursion/mutual-recursion, a call must resolve to the sibling
method's real signature. So I pre-collect a per-type `MethodSig` table *before*
lowering any body (forward refs + recursion resolve for free).

**The challenge:** the naive version regressed the LLVM verify gate from 152/152
to **127/152**. Two distinct causes, two guards:

1. **Arity gate.** Beef has overloads; my table keyed by name (first wins). A
   call whose arg count didn't match the resolved method's arity emitted a call
   with the wrong number of arguments → verifier error. Fix: resolve *only* when
   `arg_count == sig.params.len()`. Coercion then makes every arg exactly the
   declared param type, so the call always matches the signature; an arity
   mismatch falls back to the external varargs call. Got us to 146/152.
2. **Body-less skip.** The remaining 6: I was indexing *every* same-named method
   including body-less ones (extern/abstract). A `Foo(int,int)` extern could win
   the name over the body-having `Foo(int)`, so a 2-arg call resolved to a symbol
   that was never emitted (or had a different arity). Fix: only index methods
   that actually get lowered (skip `MethodBody::None`). Back to 152/152.

**Insight worth keeping:** when lowering with *partial* information (no real
overload resolution yet), the safe move is to resolve only when the resolution
is unambiguous *and* provably type-correct (exact arity → coercion guarantees a
match), and otherwise degrade. "Index only what you emit" is the corollary — a
resolution table must mirror what's actually in the module.

### The loop family + ++/-- (commit 97b18ac)

Added `for`, do-while, `break`, `continue` on a shared **loop-target stack**:
`Vec<(continue_target, break_target)>`, innermost last. `for`'s `continue` must
run the update before re-testing, so it targets a dedicated `for.cont` block,
not the head.

`++`/`--` initially lowered to `undef` — which meant a `for` update of `i++`
silently did nothing and would spin forever. Realisation: inc/dec are
*lvalue* operations (load-modify-store against a slot), not value ops. Wrote an
`incdec` helper; postfix returns the old value, prefix the new. The corpus test
`incr_decr → 577` exists specifically to pin that old-vs-new distinction.

### switch (commit 25c0d17)

Lowered to a cmp/cond-br chain (Beef arms don't fall through → each branches to
one exit). The subtlety: inside a switch arm, `break` exits the *switch* but
`continue` targets the *enclosing loop*. Handled by pushing
`(enclosing_continue, switch_exit)` onto the loop stack for the arm bodies — so
the two keywords correctly diverge.

## 2. Why we stopped to research the data types

The user asked the right question: **with no GC, how are strings and arrays even
supposed to work?** Rather than guess, I fanned out agents over the actual Beef
corlib + compiler source at `E:\beef` and the official docs.

**The finding that reframed everything:** `String`/`T[]`/`List<T>`/`Span<T>` are
**not compiler built-ins** — they're ordinary stdlib classes/structs resting on
**four compiler primitives**: the object header, `new`/`delete`, `scope`, and
**append allocation** (`[AllowAppend]`). This cleanly splits the work into
*compiler/runtime* (our Rust) vs *stdlib* (Beef we compile). No GC anywhere —
ownership is deterministic. Captured in `docs/CORETYPES.md`.

The genuinely clever bit is `String`'s small-string optimisation: one field
`mPtrOrBuffer` is *either* an inline char buffer (chars live at the field's own
address) *or* a real heap pointer, selected by a flag bit. And append allocation
(object header + variable payload in one allocation, via a synthesized
`__CalcAppend` size function + an `__appendIdx` cursor) is the hard primitive —
it's what makes `String` SSO and inline-element `T[]` possible.

## 3. The GC "escape hatch" — and the direction it set

The user was intrigued by Beef's optional-collector hook, having been burned
before by "declaring precise root safepoints all over the code." Researched
Beef's actual GC, then how our own `NewGC` and its two existing clients (NOD,
NCL) integrate. Captured in `docs/GC.md`.

**The asymmetry that dissolves the safepoint pain:** people conflate "precise
GC" with "safepoints everywhere." They're separable. Safepoints buy precise
*roots* (what's live in registers/stack at a suspension point). Precise *heap*
tracing (which fields of an object are pointers) is **static type info — free,
no safepoints**. Beef exploits exactly this: conservative *roots* (scan the
stack/registers word-by-word, validate before trusting), precise *heap* (a
compiler-emitted per-type `GCMarkMembers`). A grep of the entire Beef compiler
for `gcroot`/`statepoint`/`stackmap` returns **zero** hits.

The price of conservative roots is you can't *move* objects (you can't safely
rewrite an ambiguous root) — hence Beef is non-moving.

**Reconciling our NCL scar** ("conservative scanning fails non-deterministically;
precise roots are the answer"): that was right *for a moving collector*. The
fork is upstream of roots — *moving ⇒ precise roots ⇒ safepoints; non-moving (or
pin-only) ⇒ conservative roots suffice.* The safepoint nightmare is a consequence
of choosing to move, not a tax on having a GC.

**Direction decided (recorded in GC.md):** NewBF stays no-GC by default; the
*optional* GC plugs into **NewGC's `conservative-pin` mode** — conservative
roots + precise heap + mostly-copying — which is its *tested default*, built for
"JITs without statepoint stack maps." This **deletes NOD's entire precise-root
apparatus** (liveness fixpoint, spill slab, per-site safepoint tables, reload
plumbing, loop-header polls). NewBF would be NewGC's first pure-conservative
client. The gating constraint: NewGC's pinner rejects interior pointers, and Beef
code holds them freely (`&array[i]`, `Span.mPtr`) — but NewGC already has the
`object_start_at_or_before` function to fix this, just unwired. NCL's deepest
lesson, banked: the *manual* root contract rots (one missed `push_root` =
use-after-GC far from the cause); conservative roots remove that whole class.

## 4. The value-struct layout sprint (commit 81dd1fe)

The IR was scalar-only by design ("aggregate types arrive with the layout
sprint"). This was that sprint: `IrType::Struct` + a field-address instruction
(IR), LLVM struct lowering, and nominal-type tracking + member access (sema).

### Decisions

- **Value structs first**, deferring the heap/`new`/memory-model fork. It's the
  smallest self-contained slice that builds the layout machinery everything else
  needs, and it's fully JIT-testable without an allocator.
- **`StructId` is a `u32` handle**, not an inline layout, so `IrType` stays
  `Copy`. The field list lives in a `Module`-level registry. This kept the
  pervasive `IrType: Copy` assumption intact across the codebase.
- **`ObjectLayout` is one contiguous pointer range, not a bitmap** (this is
  NewGC's model too). For value structs the field list is just ordered fields;
  the constraint bites later when a `BeefLayout` must group reference fields —
  and since *we* control layout, the answer is field reordering, not a bitmap.

### Challenges + solutions

- **Nested / forward / self struct references in LLVM.** A struct field whose
  type is another struct needs both LLVM types to exist. Solution: two passes —
  create all *opaque* struct shells first, then set each body. Standard, but easy
  to get wrong if you build bodies eagerly.
- **The keystone risk: a global verify regression.** Registering structs made
  `lower_ty` map *every* struct-typed reference in the whole corpus from `Ptr`
  to `Struct(id)` — params, locals, returns, fields. That's a huge IR change
  across 152 modules, and ill-typed aggregate IR would fail the verifier. This
  was the moment that could cascade. It **held at 152/152** because: (a) `coerce`
  *always* yields a value of the target type (struct↔scalar with no real cast
  degrades to a typed `undef`), and (b) the "skipped/unsupported → typed undef"
  discipline keeps everything well-typed even when semantically lossy. The
  graceful-degrade design we'd been maintaining all along is what made a
  high-blast-radius change safe.
- **Distinguishing `struct` from `class`.** Critical: only value `struct`s
  become aggregates; a `class` must stay `Ptr` (reference), or class-typed
  locals/params across the corpus would flip type and likely break calls. Keyed
  on `TypeKind::Struct` + non-generic.
- **Borrow-checker choreography in `lvalue`.** Resolving `obj.field` reads the
  struct def (`&self.structs`) *and* emits a `field_addr` (`&mut self.fb`). Had
  to copy the field index + type out of the immutable borrow in an inner block
  before the mutable emit. Small thing, but the kind that stalls you if you
  fight it instead of restructuring.

### Insight

Refactoring `assign` and `incdec` onto a single `lvalue` resolver was the unlock:
once "what's a storable place?" lives in one function (local slot *or* chained
struct field via `fieldaddr`), `p.x = v`, `p.x += i`, `p.x++`, and nested
`o.inner.v` all fell out together. `lvalue` is recursive, so arbitrary chains
work. Proven by `struct_nested → 15` (two-level GEP) and `struct_swap → 21`.

The strategic bonus: the per-type field layout this sprint computes is *exactly*
the precise heap map a future `BeefLayout` hands NewGC. The layout sprint quietly
advanced the GC direction too.

## 5. The heap + object model sprint (commit 2c6708a)

Reference types (`class`), `new`/`delete`, and member access through references —
the no-GC manual-memory core. Came together same-day, on top of the value-struct
machinery.

### The key decision: `IrType::Ref(StructId)`

The carrying-forward note (above, now realised) predicted the hard part: member
access on a class reference needs to recover *which class* a pointer points at,
but we'd kept class refs as opaque `Ptr`. The clean fix was a new IR type:
**`Ref(StructId)`** — a pointer that *remembers its pointee's layout*. It lowers
to a plain LLVM `ptr` (so nothing changes at the LLVM level), but carries the
nominal class so the lowerer can `fieldaddr` through the heap body.

Why this was almost free on the verify gate: because `Ref` → `ptr`, every
class-typed parameter/local/return that used to be `Ptr` is *still* `ptr` in
LLVM. Registering all the corpus's classes (previously `Ptr`, now `Ref(id)`)
produced **zero LLVM-signature change** — verify stayed 152/152. The only new IR
is where member access on a reference now *resolves* (previously degraded to
`undef`). Same trick as the value-struct sprint: a change with huge surface but
no type-level blast radius.

### The unification that paid off: `struct_base`

A value struct and a class reference differ in *where the body is*:
- value struct local `p`: its lvalue place **is** the struct body (an alloca).
- class ref local `obj`: its lvalue place holds a *pointer*; the body is one
  load away.

Rather than branch member access on this everywhere, one helper `struct_base`
resolves any base to `(body_pointer, struct_id)` — returning the place directly
for `Struct`, loading it for `Ref`. Because it's built on the recursive
`lvalue`, **chained access through reference fields just works**: `a.next.val`
loads `a`'s ref → GEPs `next` (a `Ref` field) → loads that ref → GEPs `val`.
Proven by `heap_list → 42` (a self-referential `Node { int val; Node next; }`).
This is the second time collapsing a distinction into one resolver
(`lvalue` for places, now `struct_base` for member bases) unlocked several
features at once. Worth internalising as a pattern.

### Object model + what's deferred

- A class's instance body is a registered struct with a synthetic `$header`
  (`ClassVData*`) field at index 0; real fields follow. User field lookups skip
  the header naturally (name mismatch), so member indices come out right.
- `new C()` = `malloc(SizeOf body)` + zero the header, yielding `Ref(C)`. Added
  `InstKind::SizeOf` so the *backend* computes the ABI size
  (`StructType::size_of`) — the IR must not guess struct layout/padding.
- **Deferred on purpose:** the constructor call (`new` just allocates + zeroes
  the header for now) and the destructor (`delete` just `free`s). Both need the
  method-call + ctor-resolution machinery; fields are set explicitly in the
  corpus tests meanwhile. Also deferred: inheritance (a class body has only its
  *own* fields, so base-class field access degrades).

### Why the header is `Ptr`, and the GC tie-in

The header is the `ClassVData*` slot from `CORETYPES.md`. It's null for now
(no vtables/type-data yet). When we add the optional GC, this is also where a
type-id / mark word would live, and the per-class field layout we now compute
*is* the `BeefLayout::header_layout` NewGC consumes (`GC.md`). The object model
and the GC direction keep converging.

## 6. Constructors + destructors (commit e6b17fb)

Objects initialize themselves now: `new C(args)` runs the ctor, `delete x` runs
the dtor before freeing.

### The scoping insight that made it small

I'd flagged this as needing "instance-method calls (`this` + dispatch)" — which
sounded like the big ABI-changing refactor (every instance method gains a
`this`; `obj.Method()` needs cross-type dispatch). But **a constructor doesn't
need dispatch**: `new C(args)` names the exact type `C`, so it calls `C`'s ctor
*directly* by mangled name. That collapsed the sprint to **sema-only, no
ABI-wide change**: a ctor is just a function with a leading `this` param, `this`
is a param, and `new`/`delete` already existed. Regular instance methods were
left untouched (`this_ty = None`) — I explicitly did *not* add `this` to all of
them, dodging the risky cross-corpus signature change. Lesson: re-examine
whether the "obvious prerequisite" is actually required for the slice in hand.

### `this.field` came for free

The only "member access" a ctor body needs is `this.field`. And `this` is just a
`Ref` param → so `struct_base` (built last sprint for `obj.field` on references)
already handles it: `lvalue(This)` returns the `this` slot, `struct_base` sees
`Ref(id)`, loads it to the body, GEPs the field. **Zero new member-access code.**
Third sprint running where the `lvalue`/`struct_base` resolvers paid forward —
the pattern of routing everything through one place-resolver keeps compounding.

### The one fiddly detail: symbol names must line up

`new` resolves the ctor by a *name* recorded in the pre-pass (`{prefix}$ctor`),
but the ctor *function* is emitted later by `lower_type` under its own prefix.
Those two prefixes have to match exactly, so I threaded the same prefix-building
(`format!("{prefix}{name}.")`, types-only, namespaces pass through) into the
pre-pass and stored it per id. A mismatch would be a silent "ctor never runs"
(the call would target a nonexistent symbol → external varargs degrade). Worth a
test if this grows.

### Deferred (and why it's fine)

Ctor `: this(...)/base(...)` chaining, implicit-`this` bare field names
(`v = 5` meaning `this.v = 5`), `obj.Method()` dispatch, and inheritance. Simple
ctors that set/compute fields from args cover the common case and unblock the
core types; the rest are their own sprints. verify held at 152/152 (ctors/dtors
are well-typed `void` functions; `Ref this` lowers to `ptr`).

## 7. Instance methods + `obj.Method()` dispatch (commit eb7467e)

The "bigger refactor" I'd twice deferred — and it came in cleaner than feared.

### What made it tractable

The pre-pass already walked every type collecting ctor/dtor signatures; adding a
**global per-type method table** (name → this-aware signature) was a few more
lines in the same loop. That replaced the old *per-type-local* `sigs` (built
inline in `lower_type` for sibling calls only) with one table usable both for
same-type bare calls *and* cross-type `obj.Method()`. The dispatch itself is by
static type — `obj`'s `Ref(id)`/`Struct(id)` gives the owning type — so no
vtable yet, no runtime dispatch.

### The arity gate paid a dividend I didn't plan

Worry: now that instance methods carry a leading `this`, would a bare `Foo()`
call (no receiver) wrongly resolve to an instance method and call it without a
`this`? No — and for free. Instance sigs now have `params = [this, …explicit]`,
so a bare call with only explicit args **arity-mismatches** and falls back to
the external path. The arity gate built way back in the call-resolution sprint
(08d) already does exactly the right thing. Bare calls resolve only to static
siblings; instance calls must go through a receiver. Zero special-casing.

### `struct_base` paid forward a *fourth* time

`obj.Method()` and `this.Method()` both resolve the receiver through the same
`struct_base` — value struct (place is the body) or class ref (load to the
body) — that powered `obj.field`, `this.field`, and chained access. One
receiver-resolver, every member/method form. The "route everything through one
resolver" discipline has now paid off in four consecutive sprints; it's the
single highest-leverage shape decision of the object-model work.

Verify held at 152/152 again — same reason every object-model sprint did:
`Ref` lowers to `ptr`, so adding `this` changed no LLVM signature.

Deferred: overload-by-type (first-name-wins today), inheritance + virtual
dispatch, and implicit-`this` bare field/method names (`v` for `this.v`).

## 8. Pointer indexing — `p[i]` (commit fa1126a)

The journaled gate to `String`. The wrinkle the carrying-forward flagged turned
out to be the whole sprint: indexing needs the **element type** behind a
pointer, but our pointers are opaque (`Ptr`) — `char8*` and `int32*` both lower
to the same `ptr`, yet they index at different widths/strides.

### The decision: track element types in sema, not in `IrType`

The tempting move — a `TypedPtr(elem)` IR type — would ripple through every
exhaustive `match IrType` in three crates (`bit_width`, `mnemonic`, `coerce`,
`zero_of`, LLVM `basic_type_of`/`undef_of`, …) and `IrType` is `Copy`, so the
element couldn't even be a `Box`. Instead I attached the element type as
**binding metadata in the lowerer**: the scope entry became
`(slot, ty, Option<elem>)`, set from a pointer local/param's `AstType::Pointer`
inner. **Zero IR/LLVM type-system change** — the only IR addition is one
instruction (`ElemAddr`, a base+index GEP). The element type lives exactly where
it's introduced and is read back at the index site. Much smaller blast radius
than a new pointer type.

### `lvalue` paid off a fifth time

`Expr::Index` lowered as one `lvalue` arm — compute `elem_addr(base, elem, idx)`,
return `(addr, elem)`. Reads `load` it; assignment and `++`/`--` already route
through `lvalue`, so `p[i]`, `p[i] = x`, `p[i] += d`, `p[i]++` *all* fell out of
that single arm. Five sprints now where the place-resolver discipline turned "add
a feature" into "add one match arm."

### Wrinkles

- **Stride correctness is the point.** `index_bytes` (char8* → byte stride →
  131) and `index_ints` (int32* → 4-byte stride, variable index in loops → 60)
  exist specifically to prove the GEP scales by `sizeof(elem)`, not a fixed 1.
- **Testing needed a real pointer.** No arrays and no `&addr-of` yet, so the
  corpus programs `malloc` a buffer (CRT `malloc`/`free` resolve through the
  JIT's process-symbol generator) and `free` it. Convenient, and it exercises
  the FFI path too.
- **Limitation:** v1 resolves the element type for pointer *locals/params* only.
  Indexing a pointer *field* (`this.buf[i]`) needs field element types — and
  that's exactly the `String` shape, so it's the immediate follow-up.

verify-corpus held at 152/152 (`ElemAddr` is a standard GEP; the binding change
is sema-internal).

## 9. Field-pointer indexing — `obj.buf[i]` (commit 54a0ef9)

The one gap the last sprint flagged, closed — and it brought the
**String-buffer pattern** to life.

### Same shape as before, one level deeper

Locals/params tracked their pointer element type as binding metadata; fields now
do too, via a `field_elems` side-table parallel to `defs[id].fields`. Keeping it
*beside* the IR `StructDef` (rather than adding `elem` to the IR `FieldDef`)
matters: the backend GEPs fields by index and never needs element types — that's
purely front-end metadata, so it stays in sema. Same instinct that kept typed
pointers out of `IrType` last sprint.

### The new primitive worth naming: `expr_struct_id`

To resolve `obj.buf`'s field, I need *which struct* `obj` is — by **static
type, with no code emitted**. That's distinct from `struct_base`, which emits a
load to get the runtime body pointer. `expr_struct_id` walks `this` / locals /
nested fields purely at the type level. Having both — one that answers "what
type?" without side effects, one that produces "where's the body?" with them —
is a clean split I'll lean on again (it's the start of a real
expression-typer).

### The milestone: a hand-written `String` core runs

`string_buffer.bf` is the whole point of the last few sprints in one program: a
class owning a `char8*` buffer, allocated in its constructor, indexed through
`this.data[i]` in `Set`/`Get` instance methods, and freed in its destructor —
`delete b` runs `~this` then `free`. Classes + ctors + dtors + instance methods +
field-pointer indexing + FFI, all composing, JIT-run to **131**. That's the
manual-memory OO pattern real `String`/`List` are built from. `lvalue` carried
indexed field read/write/`+=`/`++` from one arm — its sixth sprint paying off.
verify-corpus held 152/152 (sema-only).

## 10. A String-shaped class runs (commit aaca912)

Milestone A's *pattern*, proven. `mini_string.bf` is a `MiniString` class written
in Beef — a `char8*` buffer (manual `new`/`delete`), `Append(char8)` writing via
`this.buf[this.len]`, `CharAt`/`Length` reading back — built and JIT-run to

from sprints 9–14 composing into the manual-memory String shape.

The enabling fix was small and overdue: `Expr::Char` was lowering *every* char

Honest framing: this is a corpus `MiniString`, not the actual corlib `String`
compiled and linked. It proves the *shape* runs end to end; wiring the real
`newbf-corlib` is its own integration.

## 11. A real growable String + constructor overloading (commit b9b9724)

Depth, chosen over breadth. Two findings.

**The growable String just worked — no compiler change.** `string_grow.bf`
builds a `Str` from a literal (`new Str("Hi")`), measures its own length and
copies it with index loops, and *grows* its heap buffer when `Append` outgrows
capacity. I'd expected to need a `memcpy`/`strlen` binding (the journal said so);
instead, doing strlen and the copy as plain `while`/`for` loops over
`s[i]`/`this.buf[i]`/`nb[i]` needs nothing new — param-, field-, and
local-pointer indexing all already work, and `new Str("…")` passes the literal
`char8*` straight to a `this(char8*)` ctor. The "I expected to need X" that turns
out unnecessary is itself the signal: the object-model foundation is solid.

**Writing it surfaced one real gap — single constructors.** A class recorded
only its *first* ctor, so `this()` and `this(char8* s)` couldn't coexist
(`new Str("x")` would arity-mismatch the recorded `this()` and silently run no
ctor → garbage buffer). Fixed: **constructor overloading by arity** — `ctors`
became a `Vec` per type (one sig per distinct arity, mangled `$ctorN`), and
`new` picks the overload matching the argument count (`ctor_for`). `ctor_overload`
proves it: the same `Str` built via both `new Str()` and `new Str("Yo")`.

This is the test-driven-discovery loop working as intended: write real
corlib-shaped Beef, let it run as far as it can, fix exactly what it hits. (Still
arity-only — same-arity different-type overloads don't disambiguate; that's true
of methods too.)

## 12. Stdlib organization + prelude plumbing (STDLIB.md, commit b99a538)

A deliberate breather, then the first real stdlib step.

**The design (`docs/STDLIB.md`).** Decided the stdlib is *source*, not a binary:
corlib `.bf` is prepended to the user's program and lowered as one module
(compose-at-the-AST, the pinned principle; JIT-first whole-program lowering).
Distinguished the two corlibs (the upstream `corlib-slice` we port *from* vs.
`newbf-corlib/bf` we *write*), the bottom-up layering (`Internal` → `Object` →
`String` → collections → IO → `GC` stub), and — the doc's most useful artifact —
a **gating-features table** showing the feature roadmap *is* the corlib roadmap
(qualified static calls + extern binding → `Internal`; literals-as-`String` →
real `String`; generics → collections).

**The plumbing (sprint 17).** `newbf-corlib::prelude()` embeds each `bf/*.bf`
via `include_str!`; `lower_program` parses them into owned units and prepends
them as `SourceFile`s before the user's, so `StructTable` registration and
lowering see corlib + user as one module. Centralising it in `lower_program`
means *every* consumer (driver, corpus, tests) gets the prelude with no caller
change — and corlib is now lowered into every program, so the verify gate holds
it continuously (a free forcing function as it grows). Proven by `prelude_probe`
→ 42: a `Probe` class defined *only* in `newbf-corlib/bf/Probe.bf`, used via
`new Probe()` from a program that never declares it.

The lifetime shape is the one wrinkle: the prelude units are owned locally in
`lower_program` and the combined file list borrows them, which is fine because
lowering returns an owned `Module` (no borrow escapes).

## 13. Qualified static calls + a corlib `Math` (commit 211e7a0)

First gating feature off the STDLIB.md list. `lower_method_call` gained a static
path: a `Member` callee whose base is an `Ident` that *names a registered type*
(and isn't a local) and whose method is `!is_instance` → call it by mangled name
with just the args, no `this`. The instance path (`struct_base`) stays the
fallback, so `obj.M()`/`this.M()` are untouched. The whole disambiguation is one
check — `lookup(name).is_none()` (it's a type, not a local) — leaning on the
this-aware method table from sprint 12. And corlib grew its first real type,
`Math` (`Abs`/`Max`/`Min`), called as `Math.Abs(-7)` from a program that never
declares it (`math → 16`). `static_call → 42` covers the same on a user type.

## 14. Extern binding → corlib `Internal` (commit f85c6ed)

The FFI floor of the stdlib tower. A body-less `[Intrinsic("sym")]`/
`[LinkName("sym")]` method now binds to its symbol: the pre-pass records such
methods (previously skipped as "no body") with the attribute's string as the
call target, so — combined with sprint 18's qualified static calls —
`Internal.MemCpy(..)` lowers to a call to `memcpy`. `extern_symbol` pulls the
string from the `Attribute`; other body-less methods (abstract/interface) still
skip. corlib gained `System.Internal` (`Malloc`/`Free`/`MemCpy` via `[LinkName]`)
— the allocation/copy floor the rest of corlib routes through. `internal_alloc →
131` allocates, `MemCpy`s, and frees entirely through `Internal`, no bare
externs in the program.

A satisfying convergence: this is the third feature that "just slotted into" the
method table (`is_instance` for dispatch, then static calls, now extern symbols
as the `full_name`). The table is quietly becoming the whole call-resolution
story.

## 15. Real `System.String` + the cast-skew bug (commits 421b2e0, ed13cbf)

The headline landed — but not before a genuine debugging detour.

**The feature.** The growable-string prototype is now corlib's actual `String`
class: `char8*` buffer + length + capacity, two ctors (`this()` and
`this(char8* s)`), a dtor, and `Length`/`CharAt`/`Append`/`Grow`, built entirely
on the `Internal` FFI floor (`Malloc`/`Free`/`MemCpy`). And a string literal in a
`String` context now *constructs* one: `coerce` wraps a `char8*` (a literal,
`IrType::Ptr`) into `new String(..)` when the target is the `String` class.
Because it lives in `coerce`, target-typed wrapping falls out for free at every
site — `String s = "hi"`, a literal passed to a `String` param, a returned
literal — with no expected-type threaded through `expr()`. `string_length → 5`,
`string_ops → 6533` (construct-from-literal, `Append`, `CharAt` loop).

**The bug.** Adding `String` to the prelude crashed `ctor_overload` — a program
that never references `String` — with `0xC0000005`. The bisect was telling: a
*trivial* `String` (one int field) passed; the full one crashed; and it needed
*both* `String` and a user class doing bare `malloc` + `char8*` indexing in the
same module. `cdb` (the Store WinDbg's `cdbX64.exe`) gave ground truth:
`mov byte ptr [rdx+rcx], al` with `rdx = baadf00dbaadf00d` — `this.buf` was
uninitialized heap. The IR showed why: `this.buf = malloc(this.cap)` lowered to
`store ptr undef`. The `malloc` *call* ran; its result was thrown away.

**Root cause — extern signature skew.** An external symbol's real LLVM signature
is fixed by its *first* declaration in the module. A bare `malloc(...)` is typed
`i64` by sema (the unresolved-extern default), but corlib's `Internal.Malloc`
(`void*`) now declares `@malloc` as returning `ptr` — and it lowers first. So the
call site yields a `ptr` while sema still believes it's an `i64` and emits the
`inttoptr` that `char8* = <i64>` needs. `as_int` got a pointer, returned
`i64 undef`, and `inttoptr undef` folded to `ptr undef`. Before corlib, bare
`malloc` was `@malloc`'s only declarer (`i64`), so the `inttoptr` was valid —
which is exactly why a brand-new, *uncalled* corlib type could break an old
program: it changed a *shared* extern's declared signature.

**The fix** (421b2e0): treat `IntToPtr`/`PtrToInt` as the identity when the
operand is already in the target representation. The IR↔LLVM boundary is where
advisory IR types meet real signatures, so the cast layer is the right place to
absorb the skew. (Alban predicted "alignment errors" — wrong on the specific
mechanism, right in spirit: a codegen-dependent fault that verifies clean and
only bites at runtime.)

Two lessons banked: extern call result types are advisory, reconciled by the
first declaration; and a feature that's never *called* can still change a
module's machine code by changing a symbol everyone shares.

## 17. Overload resolution by argument type — and the skew it dredged up (commits 0375396, 9657d73)

Depth on strings runs straight into a compiler feature: a real `String` API is
all *same-arity* overloads — `Append(char8)` **and** `Append(String)`, later
`==`/`Concat` — but resolution was arity-only (first-wins). So the headline is
**overload resolution by argument type**.

**The feature.** Method tables became `HashMap<String, Vec<MethodSig>>` (every
overload kept). `pick_overload` scores arity-matching candidates by exact
type-match count, gated so a `this`-less site (a bare or `Type.M` call) never
selects an instance method. The subtle part is *mangling*: two `Append`s mangled
to one symbol `@String.Append`, so only one body survived and a `ptr`-arg call
hit an `i8`-param function (10 verify failures, plus `append_overload → 4388`).
Fix: a later overload of a name gets a param-type suffix (`…$R3`/`…$i8`); the
first keeps the plain symbol; and `lower_type` now *reuses* the table's
`full_name` by matching params, so definition and call sites can't disagree.
`String` gained `Append(String)` (chosen over `Append(char8)` by type) and value
`Equals`. `append_overload → 5427`, `string_equals → 10`.

**The skew it dredged up.** First-wins had been silently *dropping* the corlib
`Math` fixture's float/double overloads — so they never lowered. Registering them
all finally lowered them, and the verifier caught `fcmp float … , double …`. The
root (found by reading the IR): `Abs(float)` and `Abs(double)` both bind the C
symbol `@abs`, whose LLVM signature is fixed by its first declaration — so a call
sema-typed `f64` returned an `f32`, poisoning the downstream `fmul`/`fcmp`. This
is exactly the `malloc` int↔ptr skew, one tier up. I first patched the *compare*
(widen mixed floats), watched the next op fail, and recognized the whack-a-mole:
the cure belongs at the **call boundary**. `reconcile_to` coerces a call result
to the type the IR expects (fpext/fptrunc, int ext/trunc, int↔ptr) — one place,
all downstream uses well-typed, and it *subsumes* the earlier int↔ptr cast fix.
Restored 152/152.

Two side-lessons: (1) a process-global like `SetStdHandle` can't have two
parallel `#[test]`s — libtest threads them, so both stdout checks live in one
test. (2) String *literals* are `Ptr`-typed at resolution, so a literal passed
straight to an overloaded `Append` would prefer a `char8*`/exact param; pass a
`String` local (or construct explicitly) to hit the `String` overload — true
literal-as-`String` overloading waits on target-typed argument resolution.

## 18. Tagged handles for Beef? — studied, declined as GC, kept as a safety primitive (commits 426ba6a, f95f1cf)

Alban: *"take a look at the simplified tagged-handle approach in `E:\RNIM` — would it fit Beef instead of fucking malloc?"* So I read Locus's `gc.md`.

**The scheme.** The program holds a **handle** — an index into one table — never a
raw pointer. The table *is* the GC root set (compiler GC-oblivious, like us), and
because a handle is an index, objects **move freely**: the collector rewrites
`table[h]`, the index is stable. Conservative scanning becomes *sound* (a stray
int that looks like a handle over-retains, never corrupts). Handles run on a stack
discipline (V8 `HandleScope` / JNI / OCaml `CAMLparam`), so a 600-node walk peaks
at a live-root depth of two. Genuinely elegant.

**The verdict (now `GC.md` §12): no as a `malloc` replacement, and the reason is
structural.** The whole payoff is *free relocation*, bought with one invariant —
*no raw pointer to a heap object ever escapes*. Beef is the opposite: `char8* buf
= s.Ptr(); WriteFile(buf, …); buf[i]`, raw `T*` in fields, interior pointers
(`&array[i]`), C layout for Win32. Beef would pin nearly everything → pay the
indirection, get no compaction. The clincher: **real Beef's own collector is
conservative + non-moving for this exact reason** — the Beef team hit the same
wall `GC.md` already chose to sit behind. The handle scheme exists to *enable
moving*; Beef can't move regardless; it solves a problem Beef isn't allowed to
have. Locus and NewBF are correctly using *different bindings of the same NewGC*.

**What we kept.** The handle *idea*, demoted from collector to optional manual
primitive: **`Pool`** — a generational slotmap. A handle is a pointer-sized `int`
packing `(gen << 16) | idx`; `Get` resolves it live and returns **null** once
`Free` bumps the slot's generation. Use-after-free becomes a *detectable null*,
not a dangling pointer — the safety win over raw `malloc`, with no GC and without
giving up raw pointers anywhere else. It's the slotmap/generational-index pattern
game engines (Beef's audience) already use. `pool_handle → 142`: live `Get` reads
`b.v` (42), then `Free`, then the stale `Get` returns null (+100). Stored as
parallel `int` arrays over `Internal`; the object pointer round-trips through
`ptrtoint`/`inttoptr` at the store/return boundary (which the §15 reconcile work
makes routine). Non-generic for now — reuse, growth, and a typed `Handle<T>`
wrapper wait on generics.

## 19. Generics, first slice — monomorphized value structs (commit 6e8b102)

Generics is the gating dependency three threads were waiting on (`Handle<T>`,
`List<T>`, `String` SSO), so: take it on. The chosen mechanism is
**monomorphization** — one concrete type per (generic, concrete-args) pair,
substituting the type parameters — not type-erased generics. It rides the
existing concrete-type pipeline rather than adding a parallel one.

The shape: a pre-pass over the program (1) indexes generic declarations by name,
(2) walks type positions — field/param/return types and local declarations —
for instantiations, and (3) registers one monomorph per distinct pair, mangled
by the **overload type codes** from §17 (`Pair<int>` → `Pair$i64`, `Pair<int32>`
→ `Pair$i32`). `lower_ty` grew exactly two cases: a bare type-parameter resolves
through a per-monomorph **env** (`T` → the concrete type it was instantiated at),
and a generic instantiation `Name<Args>` resolves to its registered monomorph.
Field-filling factored into one env-aware `fill_fields_at` shared by ordinary
types (`env = &[]`) and monomorphs — so a monomorph is *just a concrete type with
T pre-substituted*, and everything downstream (layout, `sizeof`, field access)
is unchanged.

`pair_generic → 42`: `Pair<int>` (i64 fields) and `Pair<int32>` (i32 fields) are
**distinct layouts from one template** — the proof that the type parameter is
really substituted per instantiation, not erased. 52 programs, verify 152/152.

This slice is deliberately value-structs-only — no `new`, no methods, no body
type-env. Those are the **generic class** slice: collecting instantiations from
expression position (`new Box<int>()`), lowering each monomorph's methods/ctors
with the env (threading it through `lower_type`/`lower_method`/`Lowerer`), and
resolving `new Box<int>` in `lower_new`. The infrastructure (env, mangle,
collection, registration) is now in place for it.

## 20. Generic classes — `new`/methods/`this` per monomorph (commit 19cf654)

The value-struct slice built the scaffolding (env, mangling, collection,
registration); this slice makes generic **classes** real, which is where the
type parameter actually has to flow through *code*, not just layout.

`fill_members_at` became env-aware end to end: a monomorph's ctors, dtor, and
method table are registered with their parameter/return types substituted and
their symbols mangled (`Box$i64.$ctor1`, `Box$i64.Get`). `lower_type` split into
a wrapper that **skips generic templates** (a `Box<T>` is never lowered directly
— lowering it would emit a broken `Box.Get` with `T → Ptr` and `this = None`) and
an env-threaded `lower_type_at` that `lower_program` drives once per recorded
monomorph (`StructTable.monos`) at the mono's id/prefix/env. The env rides down
through `lower_method` into the `Lowerer`, so a body's `T` locals resolve
concretely. `lower_new` learned the generic case (`generic_new_parts` +
`new_class_id`): `new Box<int>()` resolves to `Box$i64` and its ctor; instance
calls land on the monomorph's own method table.

`box_generic → 42`: `Box<int>` (i64 payload) and `Box<int32>` (i32 payload) are
distinct heap types from one template — `new`, ctor, and `Get()` each compiled
with `T` substituted, then `a.Get() + b.Get()`. **53 programs, verify 152/152.**

The design choice that kept this tractable: a monomorph is *just a concrete type
with `T` pre-substituted*, so it reuses the entire concrete-type pipeline
(layout, method table, dispatch, `new`/`delete`) unchanged — the only new
machinery is the env (one lookup in `lower_ty`) and the collection pre-pass.
Monomorphization, not erasure: zero runtime cost, the right call for a
value-types-matter language.

One honest edge: instantiations are still collected from **type positions**
(field/param/return types, local declarations). An instantiation that appears
*only* in expression position — `var a = new Box<int>(…)` with no typed local —
isn't collected yet; the test uses `Box<int> a = …`. Expr-position collection is
a small follow-up.

## 21. `Console.WriteLine(int)` — the machine prints numbers (commit f61d017)

A small capstone on the overload + generics work: print *computed* values. A
typed `WriteLine(int)` overload sits beside `WriteLine(String)` and is chosen by
argument type; it renders decimal via a recursive `AppendDigits` (emit the
high digits first, append the low one last) over `String.Append(char8)` and the
empty `String()` ctor — negatives get a `'-'`, zero prints `"0"`.

This surfaced a gap in overload ranking: `pick_overload` scored only *exact*
type matches, so `WriteLine(12345)` (arg `int32`) tied `WriteLine(int)` (`i64`)
and `WriteLine(String)` (`Ref`) at zero and fell to first-registered — which
would have coerced an int into a `String` pointer and crashed. Fixed with a
**`type_affinity`** tier: exact (2) > same category — int↔int of any width,
float↔float, pointer↔pointer (1) > unrelated (0). So an `int` argument prefers an
`int` parameter over a `String` one regardless of width. (Bonus: this also
sharpens the literal-vs-overload edge from §17 — a `char8*` literal now prefers
a `String` parameter over a `char8` one.)

`print_int → 12345`; the capture test asserts `"12345\n-7\n0\n"`. **54 programs,
verify 152/152.** The machine that learned to say "Hello, world!" in §16 now does
arithmetic and tells you the answer.

## 22. `List<T>` — a growable generic collection (commit c1c5cdc)

The headline generic collection, and the proof that monomorphized generics +
the `Internal` floor + typed-pointer indexing compose into something genuinely
useful. corlib `List<T>` holds a raw `T*` buffer with count/capacity, and
does `new`/`Add`/`Get`/`Count`/grow/`delete`.

The one compiler gap it exposed: **pointer-element types weren't env-aware**.
A `T*` field's element type drives the indexing stride (`ElemAddr`'s GEP), and
`pointer_elem` resolved `T` with no env → `Ptr` (8 bytes) for every monomorph.
`List<int>` (i64, 8) survived by coincidence; `List<int32>` (i32, 4) would have
strided by 8 and read every other slot. Fix: `pointer_elem_env`, threaded
through field layout, method params, and local declarations — so the element
type flows from the monomorph env into both field- and local-pointer indexing.
(The `lower_ty`/`pointer_elem` zero-env wrappers were now dead — everything uses
the `_env` form — so they're gone.)

Two pragmatic v1 choices, both noted in the source: the buffer over-allocates
8 bytes/slot (max scalar width) because there's no `sizeof(T)` operator yet, and
`Grow` copies **element-by-element through the typed pointer** rather than a byte
`memcpy` — which sidesteps needing `sizeof(T)` for the copy entirely.

`list_generic → 42`: `List<int>` grows past its initial capacity of 4 (one
`Grow`) and sums to 21; `List<int32>` is a distinct 4-byte-stride instantiation
summing to 21. **55 programs, verify 152/152.** A from-scratch compiler with a
working generic dynamic array, allocating through its own stdlib.

## 26. Inheritance — field + method reuse, static dispatch (commit ec0b2ba)

"BF has some deep syntax" — Alban's right, so: dive into the OO deep end. The
strategic frame from the survey held — the breadth is *parsed* (the corpus gate
is 100% clean), so this was a **lowering** slice, sema-only, no IR/LLVM change.

`StructTable` gained a `bases` vector; `fill` resolves the first base that's a
class. A new `apply_inheritance` pass runs once every type's own layout is
filled and composes each derived type — **recursive + memoized so a chain
(`C : B : A`) composes base-first**. The derived gets the base's fields right
after its own `$header`, the matching field-element types, and the base methods/
destructor it doesn't itself declare. Two facts make static dispatch fall out:

- **Prefix-compatible layout.** A derived's fields are `[$header, base fields…,
  own fields…]`, so a base method's field GEPs (typed at the base struct) hit
  the same byte offsets on a derived pointer. A base method reads inherited
  fields correctly with no adjustment.
- **Inherited sigs point at the base impl.** An inherited method in the
  derived's table keeps the base's symbol and `this` type; the call passes the
  derived `this` reinterpreted to the base ref (both are `ptr`).

So up-casting is a plain reinterpret, and inherited methods Just Work.
`inherit_basic → 42` (Dog : Animal — inherited field, inherited method, own
method over both); `inherit_chain → 42` (C : B : A — multi-level + upcast +
a method inherited through two levels). **60 programs, verify 152/152.**

Honest scope: dispatch is **static**. `virtual`/`override` — a real vtable in
the reserved `$header` slot — is the next, bigger slice, and it needs back-end
work this didn't: an IR **global** (the per-class vtable) and an **indirect
call** (call a function-pointer value, not a symbol). That's where `$header`
finally earns its keep.

## 27. Virtual dispatch — real vtables (commit ffa3d83)

The OO payoff, and the first back-end work since the generics env: `$header` —
stored as null since the heap-objects sprint, always *reserved* for this —
becomes a live class-vdata pointer. "vtables on MCJIT were hell" (Alban); ORC
was kind, because vtable globals and indirect calls ride the same
`LLJITAddLLVMIRModule` path the externs already use — no manual symbol/memory
juggling.

The cut, by crate:
- **IR.** `Module.vtables` (per class: an ordered list of impl symbols);
  `InstKind::GlobalAddr` (address of a named global) and `CallIndirect` (call a
  function-pointer *value*, not a symbol); builder + printer support.
- **LLVM.** Each vtable emits as a constant global array of function pointers
  (`@Dog$vtable = constant [1 x ptr] [@Dog.Speak]`); `GlobalAddr` lowers to the
  global pointer and `CallIndirect` to `build_indirect_call`.
- **Sema.** Record `virtual`/`override` methods; `apply_vtables` composes
  vtables base-first (inherit the base's slots, an `override` replaces a slot's
  impl, a new `virtual` appends a slot) — so **slot indices are stable from base
  to derived**. `new` stores the class vtable into `$header`. An instance call
  whose method occupies a slot dispatches indirectly: resolve the slot from the
  *static* type, load the vtable from the object's `$header`, load the slot,
  call it. The slot comes from the static type; the impl from the runtime
  object — which is exactly polymorphism.

`virtual_basic → 42` (`Animal a = new Dog(); a.Speak()` runs Dog's override, not
Animal's `7`); `virtual_poly → 49` (an Animal and a Dog behind one static
`Animal` type dispatch to `7` and `42`). **62 programs, verify 152/152.**

## 28. Ternary `?:` — short-circuiting conditional (commit 6d26c07)

A quick pick off the "deep syntax" list that doubled as a *bug fix*: `cond ? a :
b` had been lowering to `undef` (it fell through `expr`'s catch-all), and corlib
`Math`'s rounding quietly depended on it — verify-clean but wrong. Now it lowers
like an `if`/`else` joined by a phi: evaluate the condition, branch, evaluate
**only the taken arm**, and merge the two results at a join phi — each coerced to
the common type *in its own predecessor block* (the subtlety: the phi needs both
arms at one type, but the coercion has to live in each arm's block, so the arms
evaluate first, then the common type is computed, then each block gets its
coerce + branch). Short-circuit is structural — the untaken arm's block never
executes.

`ternary_basic → 42`: ordinary selection, plus a short-circuit proof — the
else-arm yields `0` while its sibling `100 / x` (with `x == 0`, a divide-by-zero
that would crash if evaluated) is skipped. A SIGFPE would have failed the run, so
green *is* the proof. **63 programs, verify 152/152.**

## 29. `==` / `!=` as value equality for classes with `Equals` (commit e1fcaab)

Lambdas are parser-blocked (the parser parses and *discards* lambda params —
`Expr::Lambda { span, body }`), and `sizeof(T)` drops its type the same way;
both need an AST change that ripples into the editor team's `format.rs`, so both
stay deferred. `operator==` likewise isn't a distinct member variant (operators
parse as ordinary methods), so a fully faithful reference-vs-`operator==` `==`
isn't cheap either. What *is* cheap and visible: make `==`/`!=` on two class
references **call the type's `bool Equals(Self)`** when it has one.

The whole feature is a guarded arm in `binary()`: if both operands are `Ref`
*and* the left type defines `Equals`, lower `a == b` to `a.Equals(b)` (and `a !=
b` to its negation, `Equals == false`); otherwise fall through to the existing
pointer/scalar compare. The guard matters twice — a `Ref`-vs-`Ref` requirement
means `obj == null` (right side is a `Ptr`, not a `Ref`) keeps **reference**
identity and never calls `Equals(null)`, and the `Equals`-exists check means
classes without one keep pointer identity too. A deliberate kernel pragmatism:
Beef's `==` is reference-default unless an `operator==` exists, but mapping to
`Equals` matches the everyday intent (`String`) with zero operator-method
machinery.

`string_eq_op → 42`: two distinct `"hello"` objects compare equal (`a == b`,
+40), `a != c` holds (+2), `a == c` is false. **64 programs, verify 152/152.**

## 30. `abstract` methods — body-less virtual slot a derived override fills (commit b4ccff4)

The natural follow-on to vtables, and it turned out to need **zero IR/LLVM
work** — the whole feature is one relaxation in method registration. Previously
a body-less, non-extern method was `continue`d (skipped: no sig, no slot). Now
an `abstract` instance method (body `None` + the `abstract` modifier) is treated
almost like a real one: it gets a mangled name `{prefix}{name}`, a callable sig
(so `a.Speak()` resolves), and a recorded vtable slot — but because its body is
`None`, `lower_method` still emits no function for it. The base vtable entry
therefore names a symbol that's never defined, and `emit_vtables` *already* maps
a missing function to a null pointer. So the base slot is null for free:

```
@"Animal.$vtable" = constant [1 x ptr] zeroinitializer      ; abstract: null
@"Dog.$vtable"    = constant [1 x ptr] [ptr @Dog.Speak]     ; override fills slot 0
@"Cat.$vtable"    = constant [1 x ptr] [ptr @Cat.Speak]
```

A derived `override` reuses the *inherited* slot index (the slot map is cloned
base→derived in `compose_vtable`), so its real function lands at the same index
the abstract method reserved. Dispatch is the existing virtual path verbatim — a
call through the abstract base type loads the slot from the object's concrete
vtable, reaching the override. The null base slot is never executed because an
abstract class is never instantiated (not enforced yet — just never written).

`abstract_method → 12`: `Animal a = new Dog(); Animal b = new Cat();
a.Speak() + b.Speak()` = `4 + 8`, each resolved through its own override.
**65 programs, verify 152/152.**

## 31. Computed (get-only) properties — `obj.Name` calls `get_Name` (commit 0ad4056)

A property with a body-having `get` accessor now lowers cleanly. Registration
(pre-pass) adds a `get_{Name}` instance method to the type's method table;
`lower_type_at` lowers each computed getter through the *existing* `lower_method`
(accessor body, `this` in scope, no params — a getter is just a nullary instance
method). Reading is the elegant part: a member access `obj.Name` first tries
`lvalue` (a field); when there's no field by that name, it falls through to
`try_property_get`, which looks for `get_Name` on the receiver's type and emits
the call. Because getters live in the *inherited* method table, derived classes
see base properties for free — no new `StructTable` field, no inheritance plumbing.

The instructive bug: the first cut took the receiver with `self.expr(base)`.
For a class that's the `Ref` (already a pointer — fine), but for a **value
struct** it's the aggregate *value*, while the getter's `this` param is a
pointer — so verify dropped to **144/152**, and tellingly the eight failures
were all the real-beef *value-type* property tests (Structs, Unions, Variant,
DateTime, DateTimeOffset, TimeZoneInfo, Anonymous, Properties). The fix is to use
`struct_base` (a pointer to the body) exactly as `lower_method_call` does, and
push it only when the getter `is_instance`. The lesson — logged before, re-proven
here — is that a green hand-written test is *not* the gate; the real-corpus
verify is, and it caught a value-vs-pointer receiver bug the toy `Rect` (a class)
could never surface. Now the verify corpus exercises real property getters
end to end.

`property_get → 52`: block-bodied `Area { get { return w*h; } }` = 30,
expression-bodied `Perimeter { get => 2*(w+h); }` = 22. **66 programs, verify
152/152.**

## 32. Property setters — `obj.X = v` calls `set_X` (commit 40f76b2; first supervised-agent slice)

The write half of properties, and the first feature built by a **supervised
sub-agent**: I specced the design, the agent edited `lower.rs`, and I reviewed
the diff + re-ran the full gate before committing (the agent's code wasn't
`cargo fmt`-clean — caught and fixed at the gate, exactly why the gate is mine
and not the agent's). A body-having `set { this.raw = value; }` accessor
registers a `set_{Name}` instance method (void, property type as its `value`
param); plain `obj.X = v` — when X is not a field — lowers to that call.

The one non-obvious bit: a setter's `value` parameter has no source span, so
`lower_method` (which names params from spans) couldn't bind it. Rather than a
bespoke setter-lowering, `lower_method` gained an `extra: &[(&str, IrType)]`
tail argument that appends pre-named params after `this`/the AST params and
binds them by literal string — getters/methods/ctors/dtors pass `&[]`, the
setter passes `&[("value", pty)]`. Write interception sits in `assign` after the
field/local lvalue path, guarded to plain `=` (compound `+=` on a property needs

receiver passes a pointer (the §31 lesson, applied up front this time).

`property_set → 99`: `c.Value = 99` then read back. **67 programs, verify 152/152.**

## 33. Auto-properties — `{ get; set; }` synthesizes a backing field + accessors (commit 4768643; supervised agent)

## 35. Generic methods — monomorphized per explicit type-arg (commit e5a8db5; supervised agent)

The last big generics frontier: a static `T Identity<T>(T x)` called `Identity<int32>(5)`.
It mirrors the generic-*type* pipeline exactly. A new `gen_method_sigs` (mangled
symbol → sig) and `gen_method_monos` (mangled, name, env) sit on `StructTable`;
`index_generic_methods` indexes generic methods by name; a `gmethods` index is
threaded through the *existing* `collect_insts_*` chain so the `Expr::Generic`
arm — which already records generic-type instantiations — now also calls
`record_method_inst` (deduping via `gen_method_sigs.contains_key`, so no extra
seen-set). `lower_program` emits each monomorph as a static free function via
`lower_method(mangled, …, None this_ty, env, &[])`, and a `Expr::Generic`-callee
fast path in the `Expr::Call` lowering resolves the mangled symbol and calls it.
The keystone correctness step: generic methods are *skipped* in the ordinary
`Member::Method` registration/lowering (both `fill_members_at` and
`lower_type_at` `continue` on non-empty `generic_params`) — otherwise their `T`
lowers with no env and emits bogus IR.

Two honest limitations, both handled by **arity guards** the supervising review
kept (the agent added them after the first cut *panicked* on the real verify
corpus): generic methods are mangled by a *global* name, and the kernel's coarse
`lower_ty_env` can map distinct instantiations to the same key — so a real call
site's arg count can disagree with the recorded mono's param count. `record_method_inst`
bails when `argtys.len() != generic_params.len()`, and the call fast path only
fires when `sig.params.len() == args.len()`, otherwise falling through to normal
handling. So same-named generic methods across classes degrade gracefully rather
than crash; a properly type-aware mangling is future work. (This is exactly the
"a green hand-written test isn't the gate, the real corpus is" lesson — the toy
`Identity` never collides; the corpus does.)

`generic_method → 12` (`Identity<int32>(5) + Identity<int32>(7)`). **71 programs,
verify 152/152, fmt + clippy clean** (a `GenMethodMono` type alias quiets clippy's
complex-type lint on the mono tuple).

## 40. Static fields — a class `static` field is a mutable module global (commit 742ec48; supervised agent, 3 crates)

## 41. foreach over `List<T>` — `for (var x in xs)` (commit 3f7745b)

## 45. Lambdas — anonymous-function emission `function R() f = () => expr` (commit dfbad7f)

Lambdas proper, on §44's function-pointer foundation. The Lowerer can't add
functions to the module mid-expression, so lambdas are gathered + emitted like
monomorphs: a `collect_lambdas` pre-pass in `lower_program` finds paramless
lambdas assigned to a `function R()` local — the target type supplies the
signature, sidestepping inference *and* capture — assigns each a `$lambdaN`
symbol (recorded by span in `StructTable.lambda_names`), and queues its body.

`lower_method` (an expression body `=> e` becomes `MethodBody::Expr(e)`, i.e.
`return e`); `Expr::Lambda` lowers to `global_addr` of that symbol, and the §44
`fn_sigs` + `CallIndirect` path calls it. `Span` gained `Hash` so the span→symbol
map keys cleanly (its `FileId` field makes a span globally unique).

Scope: paramless, non-capturing, target-typed. Typed params need the parser to
keep the params it currently drops (`Expr::Lambda { span, body }` carries no
params); closures need a heap-captured environment. Both are follow-on phases;
lambdas in other positions degrade to undef exactly as before.

`lambda_basic → 42` (`function int32() f = () => 42; f()`). **85 programs, verify
152/152.**

args to the `function R(P)` param types), and `lower_method`'s existing `extra`
mechanism (built for setters' implicit `value`) binds pre-named, target-typed
params. So the only real work was the parser keeping lambda param **names** (it
dropped them): `Expr::Lambda` gains `params: Vec<Span>`, and the call sites capture
bare-ident params — `x =>`, `(x) =>`, `(x, y) =>` (via a `lambda_param_names`
helper). A typed `(int x)` mis-captures into valid-but-unused IR (a noted limit —
the parser parses paren params as exprs then discards them). Sema zips each
captured name with the target type's param **types** (lambda params are untyped —
target-typed) and emits the free function with those as `extra` params.

`lambda_params → 30` (`dbl = x => x*2; add = (a,b) => a+b; dbl(5) + add(8,12)`).
**87 programs, verify 152/152.** Closures (capturing outer locals) are the next
lambda phase.

## 48. Closures — lambdas capture outer locals 🐄λ (commit 6583f79; 2 research agents)

Beefy-Lisp. Third research-agent round (one agent on capture detection, one on
representation), reconciled + implemented. A lambda that references enclosing
locals is now a closure. At the creation site — where the enclosing scope is live
— `detect_captures` walks the body for free identifiers (names not bound as
params or inner-locals that `lookup` resolves to an outer local), and the lambda
allocates a heap environment `[code_ptr | cap0 | cap1 …]` (raw `malloc`, uniform
8-byte slots addressed via `elem_addr(env, Ptr, i)`), storing the code pointer at
slot 0 and each captured value after. The lambda *value* is that env pointer. The
captured `(name, type)` list is recorded by symbol in a `RefCell` on
`StructTable` — the bridge from the immutable-`&structs` Lowerer (detection, during
lowering) to the emit pass (body). `emit_closure` emits `$lambdaN($self, params…)`
and binds each capture to its env address `$self[i+1]` (reads/writes flow through
it — `bind` already accepts any address as a slot). A `function R(P)` local whose
initializer is a capturing lambda is flagged in `Lowerer.closures`; a call through
it loads the code pointer from env slot 0 and passes the env back as the hidden
`$self`. Bare function pointers + non-capturing lambdas are unchanged.

The reconciliation that kept it tractable: `malloc` takes a raw byte count (no
synthesized env struct needed), `bind` already works with a computed address (so
captures need no new binding kind), and the param/`CallIndirect` machinery was
reused. New: capture detection, the env alloc, the closure-call branch.

Scope: by-value capture (the env holds a copy taken at creation). `closure_basic
→ 57` (`get = () => n` captures n=42; `addB = a => a + b` captures b=10 with a
param). **88 programs, verify 152/152.**

## 49. Higher-order functions — Map/Filter/Fold over List<T> (commit 0f5340f; 3 research agents, 1 ran ahead)

The payoff of lambdas: a function value passed *into* a method that calls it. Two
parallel agents scoped this — one the uniform function-value representation, one
the `Map`/`Filter`/`Fold` corlib — and a third ran ahead, in the background, on
the *task-after-next* (algebraic data types; see below). The reconciliation found
the minimal slice. The only kernel gap was that a `function R(P)`-typed
**parameter** wasn't callable: `fn_sigs` was populated for function-typed *locals*
(§44) but never for *params*, so a callee couldn't invoke its own `f`. One loop in
`lower_method` now registers each function-typed param (under the monomorph env,
so `function R(T)` resolves concretely) — and a `f(args)` in the body lowers to
the existing `CallIndirect` path.

With that, the HOFs are pure corlib: free static generics in `List.bf` —
`Map<T,R>(self, f)`, `Filter<T>(self, pred)`, `Fold<T,A>(self, seed, f)` — each a
plain element walk applying the function parameter. *Free* (not `self.Map<R>(f)`)
because a generic *method* on a generic *type* still isn't supported (§35's
global-name mangling); called bare and resolved through the global generic-method
index, which is exactly what makes a cross-class corlib call Just Work.

`list_hof → 18`: `[1,2,3,4]` --Map(×2)--> `[2,4,6,8]` --Filter(>3)--> `[4,6,8]`
--Fold(+,0)--> `18`. The honest edges (now queued as follow-ons): the function
*argument* is a bare code pointer, so non-capturing lambdas and method references
work — passing a **closure** would need the value to self-describe, i.e. the
uniform function-value repr (every value an env, method-refs thunked, one call
shape). And the lambdas are bound to `function`-typed locals first (target-typed
today); lambda-directly-in-call-arg needs `collect_lambdas` to read the callee's
param types. **89 programs, verify 152/152.**

The ADTs agent (run ahead, in the background) returned a full design for the
task-after-next — payload enums / sum types + pattern matching. Verdict: the
**parser is 100% ready** (`Member::EnumCase` already carries `payload: Vec<Param>`;
`switch` arms parse `.Some(let v)`), but lowering is **0%** — value enums only. The
work: distinguish binding patterns in the AST (today `let v` collapses to a plain
`Ident`, losing the binding), lay a tagged union out as a synthetic struct
`{ $disc, payload }` reusing the field/`SizeOf` machinery, then lower construction
(set the discriminant + store the payload) and `match` (test the discriminant,
bind payload fields in the arm scope). ~1 sprint; banked for when we get there.

## 50. Algebraic data types — payload enums + match with binding 🧬 (commit 1d465c4; 2 research agents)

The ML/Rust leg. A payload-bearing `enum` (`enum IntOpt { Some(int32), None }`)
is now a tagged union you can construct and destructure. Two parallel research
agents mapped the two halves — value-struct *value* mechanics (alloca → store
fields → load the aggregate) and the switch/enum-flow extension points — and I
reconciled + implemented.

Representation: a payload enum is reclassified as a **value struct**
`{$disc:i32, $p0, $p1, …}` — field 0 the discriminant, payload slots sized to the
widest case. This reclassification runs in `StructTable::build` *before* the
member/signature fill, which is the bug that cost the most to find: register it
after, and `Switch1(Shape)`'s recorded param sig is still the `i32` int-backed
fallback while the function *definition*, lowered later, uses the struct — so the
call coerces a `%Shape` arg down to `i32` and the LLVM verifier rightly rejects
`call …(i32 undef)`. Registering before the fill makes call site and definition
agree (`fill_type_struct` skips enums, so it never clobbers the synthetic fields).

Construction: `IntOpt.Some(5)` (qualified) and target-typed `.Some(5)` / bare
`.None` (resolved to the unique enum owning that case) → alloca, store the
discriminant + each payload field, load the aggregate. `match`: when the
scrutinee's type is a payload-enum struct, `lower_enum_match` spills it, loads
`$disc`, tests each arm's case discriminant, and binds its payload fields to the
pattern's `let` names in a fresh arm scope (the §a1a6b1e parser fix keeps the
bound name's span, so `case .Some(let v)` carries `v`). It mirrors the value
switch's block/`break` structure; a patternless arm is the default.

The honest scope (the guard that held verify at 152/152): only **homogeneous**
payload layouts become structs. Real-beef enums with a *heterogeneous* slot —
`ETest { A(int), B(float) }`, `EnumA { B(float,float), C(String) }` — can't share
one typed slot, so they stay int-backed (exactly as before). A real size-based
union (a byte blob reinterpreted per case), generic payload enums (`Opt<T>`), and
`when` guards are the follow-ons.

`intopt_match → 55` (`Some(42)` binds v=42; `None` → +13). **90 programs, verify
152/152** — including real-beef `Switches.bf` / `ConstEval.bf`, whose payload
enums now exercise the new path.

## 51. Generic payload enums — Option<T> monomorphized (commit 4f0ce9d; 2 research agents)

`enum Option<T> { Some(T), None }` now monomorphizes per instantiation into a
tagged-union struct `{$disc:i32, $p0:T→concrete}` — the §50 machinery threaded

no methods, no bases — are reclassified. So corlib `Result<T> : IDisposable`
(method-bearing, and `Err(void)` heterogeneous) and friends stay int-backed,
exactly as before. `generic_option → 42` (`Option<int32>.Some(40)` binds v=40;
`.None` → +2). **91 programs, verify 152/152.**

Honest scope: construction is *qualified* (`Option<int32>.Some`) — idiomatic
target-typed `.Some(x)` (where the receiver type disambiguates `Option<int>` from
`Option<bool>`) needs target-type propagation, the same bidirectional-typing

## 52. The `case` operator + methods on payload enums (commit ec2b568)

Two follow-ons from §50–51 land together because the second needs the first.

**The `case`-test operator.** `x case .Some(let v)` was parsed (`BinOp::Case`,
since §04's pattern work) but unhandled in lowering — it fell through to `undef`.
It now lowers as a *boolean expression that also binds*: one arm of
`lower_enum_match` turned inside-out. `case_test` resolves the scrutinee to an
(enum id, body-address) — a value scrutinee (`Struct(id)`) spills to a slot, a
pointer one (`Ref(id)`) is already the address — then compares `$disc` to the
named case and binds each payload field to its `let`-name *in the current scope*,
so the guarded branch reads them: `if (opt case .Some(let v)) { use v; }`. The
binding is a stable slot address, valid even on a miss; reading it is only
meaningful when matched (Beef's contract). Slightly broader than Beef (the bind
outlives the `then` block), harmless because the value is garbage unless matched.

**Methods on payload enums.** Previously a method-bearing enum stayed int-backed
(`enum_is_simple` = cases-only), stranding its methods. Now `enum_is_layoutable`
(no base; members are cases *or* methods) gates the *non-generic* reclassification,
while `enum_is_simple` still gates the *generic* monomorphization path — so
corlib `Result<T> : IDisposable` (has a base) stays int-backed exactly as before.
The reclassified enum's synthetic `{$disc, $p0…}` layout must survive method
registration, so `fill_members_at` grew a `fill_layout` flag: the new
enum call passes `false` (register ctors/methods, skip the field/​base refill that
would clobber the union). Body emission was already kind-agnostic
(`lower_type_at` emits any `Member::Method`), so `this` flows in as a `Ref(enum
id)` and the `case`-test's pointer arm destructures it — which is exactly how
`Option`'s `GetValueOrDefault` is written.

`if_case_bind` (the operator in an `if`) and `enum_method` (`a.GetOr(0) +
b.GetOr(5)` → 47, body `if (this case .Some(let v)) return v;`) both run.
**93 programs, verify 152/152.** Honest scope: `not case` still drops its
negation (parser §04), and properties-on-enums aren't layoutable yet (methods
only). Target-typed `.Some(x)` and `Result<T,E>` remain the open ADT follow-ons.

## 59. Binary operator overloading — `a + b` → `operator+` (commit 8c02a9c)

`a OP b` with a user-type operand now dispatches to a static `operator <sym>`.
`binary` evaluates both sides (as always), and if either operand is a
`struct`/`class` with a matching two-arg static operator, coerces the operands to
its parameter types and calls it; otherwise the scalar kernel paths run unchanged.

keys off the bucket name in the type's method table, using the found sig's own
mangled `full_name` for the call — so the symbol (e.g. `Vec.operator+`, `+` and
all) always agrees with the emitted definition; the JIT resolves it fine.
Covered: `+ - * / % & | ^ << >> == != < <= > >=`. Notably a `Ref == Ref` without
an `operator==` still falls through to §29's String-`Equals`/identity path.
`operator_overload` defines `+`/`*` on a `Vec` and checks them next to scalar
arithmetic → 46. **101 programs, verify 152/152.** Follow-on: *unary* operator

## 61. Compound assignment via operator overloading — `v += w` (commit 16c40d2)

## 64. Heterogeneous payload enums + generic `Result<T,E>` (commit b3d5090)

## 65. `??` null-coalescing — and a latent pointer-comparison bug it exposed (commit 81b9f11)

`a ?? b` lowers as a short-circuiting select on `a == null`: evaluate `a` once,
branch on null to either the `a`-arm (coerce `a`) or the `b`-arm (evaluate +
coerce `b`), join via a phi — the `?:` shape, keyed on a null test instead of a
bool. Intercepted in `expr` before `binary` (which would eagerly evaluate `b`).

Wiring it up surfaced a real backend bug: the LLVM `as_int` helper returned `i64
undef` for a *pointer* operand, so **every** reference/pointer comparison
(`a == null`, `a != b`) had been folding to `br i1 undef` and branching at random
— it just hadn't been exercised, since class value-equality goes through the
`Equals` path (§29). `as_int` now `ptrtoint`s a pointer to its address, so pointer
`icmp` is correct everywhere. `null_coalesce` → 12 and `null_check` (`!= null` /
`== null`) → 109 lock both in. **110 programs, verify 152/152.** A good reminder
that an untested primitive can be quietly wrong until something leans on it.

## 67. Transitive generic monomorphization — `List<T>` inside `Stack<T>` (commit 7a5e3a7)

A user `Stack<T>` backed by a corlib `List<T>` returned garbage: its `items` field
had monomorphized to **`List$p`** — `List<T>` with `T` unresolved → the `Ptr`
fallback — so it stored `int32`s through a pointer-element list. The instantiation
collector only ever scanned source with an *empty* env; it never re-scanned a
mono's own body with that mono's concrete env, so a generic nested inside another
generic never got the outer's `T`.

Fix: thread a type-param `env` through the whole collector family
(`collect_insts_*`, `use_in_type`, `record_inst`, `record_method_inst`). Type args
now resolve through it (`lower_ty_env(.., env)`), and the moment `record_inst`
registers a fresh mono it re-collects *that declaration's body* under the new env —
so `Stack<int32>` drags in `List<int32>` (`List$i32`), which in turn drags in
whatever *its* body needs, fully transitively. `seen` is marked before the
recursive scan, so a self-referential generic terminates. `generic_stack`
(push/pop/count over `Stack<int32>`) → 150. **112 programs, verify 152/152.** This
is a foundational generics fix — any generic container of a generic now works.

## Known limitation — JIT can't resolve FP constant-pool symbols

Found while attempting `Console.WriteLine(float)`: the ORC + `RTDyldObjectLinkingLayer`
JIT **fails to materialize `__real@<hex>` symbols** — the COFF constant-pool entries
the x86 backend emits for FP constants that aren't immediates (which on x86 is
essentially all of them). Error: `Failed to materialize symbols: { __real@... }`.
Simple float corpus code passes only because LLVM *constant-folds* its FP ops to
integers (e.g. `numeric_cast`); code whose float math depends on a runtime value
can't fold → real pool constants → JIT failure. **AOT is unaffected** (it runs a
real `TargetMachine` + `link.exe`). Fixing the JIT path needs dedicated work
(RTDyld/COFF mergeable-constant handling, a code-model tweak, or a custom ORC
definition generator that synthesizes `__real@<hex>` from the value encoded in its
name). Until then, JIT/run-corpus features avoid non-foldable runtime FP constants
(so float *printing* and float-heavy `Math` wait). Recorded so it isn't
rediscovered the hard way.

## 71. `ref`/`out` parameters — pass-by-reference mutation (commit 8d0e885)

`ref int32 a` was *parsed* (the `ParamModifier` rode along in the model) but never
lowered — the modifier was dropped, so `Swap(ref x, ref y)` silently copied. Now a
by-ref param is a **pointer to the caller's storage**, end to end:

- **Call site.** `arg_value` spots an `Expr::Prefix { kw: Ref|Out, operand }` argument
  and, instead of evaluating the operand's *value*, takes its **lvalue address**
  (`self.lvalue(operand)` — the same slot pointer locals/fields/indices already
  resolve to) and passes that as a `Ptr`. Wired into all three call shapes: bare-name
  same-type calls, qualified static (`Mut.Double(ref a)`), and instance
  (`c.Bump(ref a)`).
- **Callee.** The param binds **straight to `Value::Param(i)`** (which is now the
  pointer) with no entry spill — its bound value-type is the *pointee*, so an ordinary
  `Expr::Ident` read `load`s through it and an assignment `store`s through it. That's
  the whole trick: reuse the existing slot-is-a-pointer convention, but the "slot" is
  the caller's address rather than a fresh `alloca`.
- **Consistency.** `param_ir_ty` centralizes the by-ref→`Ptr` rule so the IR param
  type, the overload-mangling type-codes, and the call-site coercion all agree (the
  symbol `Swap$...` must hash the same in the index pass and the body pass, or the
  call resolves to a defaulted external).

`ref` and `out` are identical at the IR level; `out`'s definite-assignment rule isn't
enforced (an uninitialized `int32 r;` already gets an `alloca`, so `out r` just writes
into it). The width caveat from the `int32 x = 0` stack-overrun bug applies here too —
a `ref int` param over an `int32` arg would `load`/`store` 8 bytes through a 4-byte
slot; the corpus tests match widths exactly, and a real type checker would reject the
mismatch. **119 run-corpus programs** (`ref_param`, `out_param`, `ref_param_methods`);
AOT `.exe` of the method test exits 100.

**Struct- and class-typed by-ref came for free** (added `ref_struct_param`,
`out_class_param` → 121). I'd flagged struct `ref` as a follow-on, expecting
`struct_base` to need work — but it already has a `Struct(id)` place arm that returns
the place *as* the body pointer, which is exactly the caller's struct address my
binding hands it, so `p.x = …` writes through to the caller. A class `ref`/`out` binds
a `Ref`-typed place, so reading `c` loads the pointer and `c = new Box()` stores the
fresh reference back into the caller's variable — i.e. `out Box b` lets a callee
allocate and hand the object back. The one real remaining gap is `ref` args in
**generic-method** calls (the `Name<T>(args)` path still evaluates args by value).

## 72. Tuples — `(A, B)` value types as synthetic structs (commit 5d52f18)

A tuple type was parsed (`TyTuple`) and modeled (`TypeRef::Tuple`) but never lowered.
The realization that made this small: **name the synthetic struct's fields "0", "1",
…** and `t.0`/`t.1` member access, assignment, by-value param passing, and return all
fall out of the *existing* struct machinery — `lvalue`'s `Member` arm already finds a
field by `f.name == fname`, and `fname` for `t.0` is literally `"0"`. No new use-site
code; the only feature-specific lowering is building a literal.

The architecture wrinkle is *when* to register the structs. Lowering reads an immutable
`StructTable`, and `m.structs = structs.defs.clone()` snapshots it before bodies lower —
so tuple structs must exist by the end of `StructTable::build`, via a pre-pass, not lazily.
`register_tuples` walks every type position (fields, method params/returns, ctor params,
property types, and — recursing statement bodies — local declarations), keying each
distinct shape by element `type_codes` so `(int32,int32)` everywhere is one `Struct(id)`.
Nested tuples register inner-first so the outer's field type is already a concrete struct.

**The placement bug, and the fix.** My first cut ran the pass at step 4d, *after*
`fill_members_at` built the method-signature table. Result: `MakePair`'s `(int32,int32)`
return resolved to the `Ptr` fallback when its sig was built, so the IR was
`call ptr @MakePair`, Main stored the result nowhere (`store undef, %p`), and
`Sum(undef)` got a junk argument — the value silently vanished across both the return
and the argument. Moving the pass to **step 3b** (after monos are collected, before any
field/member fill) fixed both at once: element IR types need only names/kinds (step 1)
and monomorphs (step 3), never filled fields, so 3b is early enough yet has everything it
needs. Lesson reinforced: a type that flows through a *signature* has to be registered
before signatures are built, full stop.

A literal `(a, b)` builds the struct by storing each element into field i: **target-typed**
from a tuple-typed local (so the i64 literals in `(int32,int32) t = (3,4)` coerce to the
i32 fields) or **shape-inferred** from the element types otherwise. **123 run-corpus
programs** (`tuple_basic`, `tuple_param_return` — tuple param + return + field mutation);
AOT exe exits 23. Follow-ons: tuples inside **generic templates** (their elements carry
unresolved `T` — needs per-monomorph registration) and **destructuring** `(a, b) = pair;`.

(Aside: tuple **destructuring** `(int a, int b) = …` / `var (a, b) = …` can't land yet —
the *parser* drops the binding names (the target comes back as a `Tuple` of type-idents,
`var (a,b)` as a local literally named `"("`). It needs parser work first, so it's parked.)

## 73. Heap arrays — `T[]` with `new T[n]`, indexing, `.Count` (commit 11f5c23)

Arrays were parsed (`TyArray`) and modeled but never lowered. The whole feature turns on
one representation choice: a **length-prefixed block where the array value points at the
elements, not the header**. `new int32[n]` is `malloc(8 + n·sizeof(T))`, the count goes in
the first 8 bytes, and the returned pointer is `block + 8`. The payoff is that the count
lives *behind* the data, so:

- **Indexing is free.** `a[i]`, `a[i] = v`, `a[i] += v`, and even `arr[i].field` all reuse
  the existing typed-pointer index lvalue — I just taught `pointer_elem_env` that `T[]`
  records element type `T` (same as `T*`), and `a[i]` indexes from offset 0 like any
  pointer. Zero new index code. (`arr[i].field` works because `lvalue(Index)` already
  returns a `Struct`-typed place that `struct_base` GEPs into.)
- **`.Count`/`.Length`** load `ptr[-1]` (i.e. `elem_addr(ptr, u8, -8)` then `load i64`).
- **`delete a`** frees `ptr - 8`, the real allocation base.

The only thing needing a marker is distinguishing an array from a raw `T*` (both are `Ptr`
with a recorded element type) for `.Count`/`delete` — a `HashSet<String> array_locals`,
in the same spirit as `fn_sigs`/`closures`. Element size comes from `size_of_ty`, factored
out of the `sizeof(T)` lowering so a `Pair[]` sizes its elements through the IR `SizeOf`
(LLVM DataLayout) — confirmed by `array_struct` (a `Pair[]` with per-element `ps[i].a = …`
writes, plus an `int64[]` with `+=`). **125 run-corpus programs**; AOT exe exits 90.
Follow-ons: arrays as **params/returns/fields** (the `array_locals` marker is per-body, so
an array crossing a call boundary loses `.Count` until the length-prefix convention is made
type-carried), multi-dim (`T[,]`), and initializer lists `new T[]{…}`.

## 74. `defer` — scope-exit statements, LIFO (commit 2b06d26)

`defer stmt;` (run this at scope exit) was parsed but landed in `stmt()`'s `_ => {}`
catch-all — silently dropped. The model is a **per-block stack of deferred bodies**,
parallel to the existing variable-scope stack:

- A `Stmt::Block` pushes a defer scope on entry and, on **normal fall-through exit**, runs
  that scope's defers in **reverse declaration order** (LIFO) — *before* popping the
  variable scope, so the deferred code can still read the block's locals.
- A `return` evaluates its value **first** (captured as an SSA value, so a deferred
  mutation can't alter what's returned), then unwinds **every** pending defer across all
  open scopes — innermost scope first, reverse within each — before the `ret`. So an early
  `return` still fires every cleanup.
- The terminator bookkeeping does the rest: once `return` has run the defers and set
  `terminated`, each enclosing block sees `terminated` on the way out and skips its own
  fall-through defer-run, so nothing double-fires.

Deferred bodies are **cloned** (`Stmt` is `Clone`) rather than borrowed — the alternative,
storing `&'ast Stmt`, would put an AST lifetime on the `Lowerer`, and the clone is cheap
next to the IR it expands to. Verified by `defer_order` (an inner block's three defers fire
3→2→1 building `321`, exercising both `defer { … }` and bare `defer stmt;`) and
`defer_return` (a method-scoped `defer b.v += 1` fires past an early `return`; a second
method returns its value *before* its defer bumps the box, proving the capture-before-defer
ordering). **127 run-corpus programs.** Known gap: `break`/`continue` don't yet run the loop
body's defers (they branch without unwinding) — the next increment if `defer` inside loops
becomes load-bearing.

## 75. `foreach` over heap arrays (commit eb74602)

Arrays (§73) landed after `foreach` (§41–42), so an array `iter` fell through to the

## 76. Array initializers — `new T[](v…)` / `new T[N](v…)` + a constant-pool gotcha (commit 120af04)

Array initializers parse as `new (Call{callee: T[size?], args: values})` — distinct
from the class ctor `new T(...)` (callee is an `Ident`, not an `Index`), so the
discriminator is "callee is an `Index` whose base names a type". The lowering shares
`alloc_array` (factored out of §73's `new T[n]`): allocate the length-prefixed block
(count = explicit size, else the value count), then store each value into its slot.
Trivial — *except* it surfaced a new face of the JIT constant-pool limit.

**The bug hunt.** `array_init` passed under AOT (exit 96) but the JIT failed
`Program.Main resolves`. Bisecting: `new int32[](10,20,30)` passed, `new int32[4](1,2,3,4)`
failed — then `new int32[](10,20,30,40)` *also* failed while the explicit `[3]` form
passed. So it wasn't explicit-vs-inferred at all: it was **element count ≥ 4 of
all-constant ints**. That's LLVM's store-merging turning four consecutive `store i32 <const>`
into one `store <4 x i32> <constant>`, whose vector constant lives in a pool as `__xmm@…` —
the very symbol class (`__real@…`'s cousin) the RTDyld JIT can't materialize. Confirmed by
swapping to runtime-derived values (`new int32[](k, k+1, k+2, k+3)`), which forms the
vector with inserts (no pool) and passes. The corpus test now uses runtime values for its
4-element array; the all-constant path is AOT-verified. Memory note
[[jit-fp-constant-pool-limit]] broadened from `__real@` to "any constant-pool symbol incl.
`__xmm@`". **129 run-corpus programs.** The lesson: when a JIT-only failure tracks the
*number/constancy* of values rather than the code path, suspect constant-pool merging, not
your lowering.

## 77. Arrays as parameters and return values (commit 73e2b51)

§73 flagged arrays-across-call-boundaries as a follow-on, expecting it to need the
length-prefix convention made "type-carried". It didn't — the realization was that the
value crossing the boundary is *already* just the elements pointer (a `Ptr`, passed and
returned by value, no special handling), and the *receiving* end already knows it's an

`Stmt::Local`), and the only missing piece was marking a `T[]` **parameter** the same way.
One `array_locals.insert` in the param-binding loop and `.Count`/indexing/`foreach`/`delete`
all work on array params and returns. `array_param` (a `Make`→`int32[]` and a `Sum(int32[])`
using `.Count`+foreach) → 130; AOT exit 85. The `array_locals` set is still per-body, so an
array stored in a *struct field* and pulled back out loses `.Count` — that case (and a
type-carried array repr generally) is the remaining follow-on.

## 78. Null-conditional member access — `a?.field` (commit 2944383)

`a?.field` parsed as `Member{conditional:true}` but lowered identically to `.` — so a
null base would dereference and fault. Now it's a null-guarded load: evaluate the base
**once**, `icmp eq null`, branch to a non-null block that does the `field_addr`+`load`,
and phi-join with the field's default (`zero_of`) on the null path. For a reference-typed
field this is exactly Beef's semantics (the default is `null`), and chains compose —
`a?.next?.val` short-circuits to `0` at the first null link because each `?.` independently
guards. A *value*-typed field yields `0` rather than wrapping in `T?` (the nullable-result
type is the documented simplification; most `?.` use is reference navigation where the plain
result is already right). `null_conditional` covers a live read, a null-base default, a
null ref field, and the short-circuiting chain → 131; AOT exit 47. Follow-ons: `a?.M()`
(conditional call) and `a?[i]` (conditional index), plus true `T?`-valued results once
nullable value types exist. Note this stayed sema-only — no RTTI needed, unlike `is`/`as`,
which still fall through `binary()`'s `_ => undef` and want a per-class type descriptor
(an IR + emission change deferred as too broad for one increment).

## 79. `is`/`as` runtime type tests — vtable as the type tag (commit 61bc44e)

§78 parked `is`/`as` as needing a per-class type descriptor (an IR + emission change).
Turned out to be sema-only after a reframing: a class with virtual methods *already*
stores its vtable global in `$header` (field 0), and that pointer is a unique-per-class
runtime tag. So `obj is T` loads the header and emits an **OR-chain of pointer equalities**
against the vtable of `T` and of every vtable-bearing subclass of `T` — and that whole set
is known at compile time by walking the `bases` graph (`is_subtype_of`). `obj as T` is the
same test feeding a `select(test, obj, null)`. No new globals, no header/layout change, no
vtable-slot shift — exactly why it's low-risk where a real type-descriptor RTTI wasn't.

The honest limitation: the tag only exists for classes that emit a vtable (i.e. have a
`virtual`/`override`/`abstract` method), so `is`/`as` against a *non-virtual* class returns
false/null. That's the opposite of the usual worry — `is`/`as` are almost always used
polymorphically, exactly where vtables exist — so it covers the real cases and degrades
safely otherwise. `is_as` (a `Dog` seen as `Animal` tests true for `Animal`/`Dog`, false
for `Bird`; `as Dog` yields a reference usable for a virtual call, `as Bird` yields null) →
132; AOT exit 31. Follow-on: a proper type-id in the header would extend this to non-virtual
classes and to value-type/`struct` tests.

## 80. Local (nested) functions — non-capturing (commit 82c3a0b)

A function declared inside a method body (`int32 Square(int32 x){ return x*x; }`) parsed
as `Stmt::LocalFunction` but fell through `stmt()`'s catch-all. Implemented by **mirroring
the lambda pipeline**: a pre-pass (`collect_local_fns`) walks every method body, assigns
each local function a unique `$localfn{N}` symbol keyed by its name span, and queues it;
the emit loop in `lower_program` lowers each as a plain free function with its own typed
params (unlike lambdas, local functions carry real `AstParam`s, so they pass straight to
`lower_method` — no `extra` dance). During body lowering the `Stmt::LocalFunction` arm just
brings the name into scope (name → symbol/ret/param-types), and a bare-name call resolves to
a **direct** call ahead of the sibling-method/external fallbacks.

Scoped to **non-capturing, non-recursive**: the emitted body is a free function with a fresh
lowerer, so it can't see the enclosing method's locals (capture = a closure follow-on) nor
its own symbol (recursion would need self-registration in that fresh lowerer). Generic local
functions are skipped. Resolution is declaration-order (the name registers when its stmt
lowers), not hoisted. These cover the common "small private helper" use; `local_function`
(two independent helpers with nested calls) → 133; AOT exit 45.

## 81. Null-conditional method call — `a?.M()` (commit 554d60b)

Completes the `?.` family started in §78. Same null-guard shape as `a?.field` but around a
whole call: evaluate the receiver once, `icmp eq null`, branch; the non-null block runs the
ordinary `lower_method_call`, phi-joined with the method's default on the null path (a
`void` method skips the phi). The wrinkle is that `lower_method_call` re-resolves the
receiver from the AST, so it would *double-evaluate* the base — fine for `b?.Get()` (the
base is a load) but wrong for `Make()?.Get()`. Guarded by a new `is_idempotent` predicate
(identifier / `this` / member-or-index chain over those); an effectful base falls back to a
plain call (which faults on null, as before — but that combination is rare and was never
supported anyway). `null_conditional_call` → 134; AOT exit 33. Still outstanding from the
family: `a?[i]` and true `T?`-valued results.

## 83. `scope T()` — attempted, reverted (design note)

Tried implementing `scope T()` (Beef's scope-bound allocation) as **alloc-like-`new` +
auto-delete at block exit**, reusing the §74 defer block-exit machinery: a `scope_dtors`
stack parallel to `defers`, each `scope T()` pushing its `(ref, type-id)` onto the current
block's list, freed (dtor + free, LIFO) on block fall-through and before `return`. Straight-
line tests passed (a dtor bumping a static counter showed correct LIFO cleanup → 23), but it
**regressed the LLVM verifier from 152/152 to 148/152** — `String.bf`, `Runtime.bf`,
`NumberFormatter.bf`, `Params.bf` all hit *"Instruction does not dominate all uses"*.

Root cause: I captured the **raw SSA ref value** at the `scope` expression and emitted
`free(v)` at the block-exit point. When `scope` is used under control flow that isn't a
braced sub-block — e.g. `if (c) Foo f = scope Foo();`, or a `scope` temporary after a
branch — the alloc's basic block doesn't dominate the cleanup block, so `free(v)` references
a value that isn't live on all paths. Reverted to hold the line at 152/152.

The correct design needs the cleanup to reload through a **stack slot that dominates the
exit**, with a per-object "was it allocated?" guard (null-init the slot in a dominating
block; `free` only if non-null) so a conditionally-taken `scope` doesn't free garbage — i.e.
real scope-cleanup bookkeeping, not raw-value capture. Parked until that's built. Lesson:
the sema **corpus LLVM-verify gate** (`llvm_lowering_verifies_on_real_beef`, 152 real Beef
modules) catches exactly this class of "valid on my toy, invalid on real control flow" bug —
run it alongside the run-corpus after any lowering change, not just the JIT corpus.

## 85. Generic array element types + `List<T>.ToArray()` (commit b1188b4)

`new T[n]` with a *generic* element worked by accident only for pointer-sized `T`: the
element resolver (`array_elem_ty`) consulted registered types + primitives but not the

## 86. `params` arrays — variadic parameters (commit 602b18a)

`void Print(params int32[] xs)` called as `Print(1, 2, 3)`. The callee side was free — a
`params T[]` is just a `T[]` param, already marked an array (§77) so `xs.Count`/foreach work
— so the work is all at the *call site* + overload resolution. `MethodSig` gains
`variadic: Option<elem-type>` (set when the last param has the `Params` modifier, via
`pointer_elem_env` for the element type). `pick_overload` treats a variadic candidate as
matching any arg count ≥ its fixed-param count, scoring only the fixed leading params with a
**flat penalty** so an exact non-variadic overload wins a tie (e.g. `Foo(int)` beats
`Foo(params int[])` for `Foo(5)`). The call site (`pack_variadic_args`, wired through all
three call paths) coerces the fixed args and packs every overflow arg into a fresh `T[]` via
`alloc_array`, passed as the single trailing argument — including the zero-overflow case
(`Sum()` → an empty `T[]`).

The risk was the overload-resolution change touching real code, so I gated on the **sema
corpus LLVM-verify** (still 152/152, and the suite's `Params.bf` is exactly this feature) as
well as the run corpus. Used runtime-derived element values in the test to dodge the
`__xmm` constant-pool trap (§76) on the packed stores. `params_array` (pure-variadic, empty
pack, fixed+variadic) → 138; AOT 7313.

## 87. String `operator+` (concat) — and a latent operator-overload bug it exposed (commit 06c7096)

Added `String operator+(String,String)` and `operator+(String,char8)` (build a new owned
string via `Append`). Straightforward — except adding a *second* overload of the same
operator was the first time the kernel had to **choose** between operator overloads, and it
couldn't: `try_operator_overload` did `sigs.iter().find(|s| s.params.len() == 2)` — the
*first* 2-arg `operator+` by name, no operand-type check. So `c + '!'` (String + char8)
resolved to `operator+(String,String)`, `coerce`d the `char8` (33) into a `String` *reference*,
and `Append`'d it — dereferencing pointer `0x21` → **JIT `STATUS_ACCESS_VIOLATION`**. The
run corpus caught it as a hard segfault (the only feature this session to crash rather than
return a wrong value). Fixed by scoring both params with `type_affinity` and taking the best
fit, same as `pick_overload` — so `String+String` and `String+char8` each land on their own
overload. Existing single-overload operator tests are unaffected (affinity trivially picks
the lone candidate). `string_concat` → 139; verify 152/152; AOT 1333. Lesson: a feature that
*adds a second overload* of anything is really a test of the resolver, not the feature.

## 89. `base.Method()` — chain to the base-class implementation (commit 5e2b0c7)

An `override` calling `base.M()` to extend its parent was unhandled — `struct_base` didn't
know `Expr::Base`, so it fell to undef/garbage. The fix is small given the inheritance infra
(§"State"): from inside a method, walk up `bases` from the enclosing `this` type, resolve
`M` on the nearest base that declares it, and emit a **direct** call (by the base impl's
symbol, *not* through the vtable) with the current `this` as receiver. Non-virtual is the
whole point — a virtual `base.Sound()` would re-dispatch to the derived `Dog.Sound` and
recurse forever; the direct call runs `Animal.Sound`. Layout-compatible single inheritance
means the `Ref(Dog)` receiver is already a valid `Animal*`, no adjustment needed.
`base_call` (Dog.Sound = base 10 + 20, entered via a virtual call through an upcast) → 141;
verify 152/152; AOT 30.

## 90. Implicit base-constructor chaining (commit 4de45d4)

`new Derived()` ran only `Derived`'s ctor — the base ctor never fired, so inherited fields
sat at malloc garbage (a `Derived` whose base set `a = 5` read `a` as noise). Now `lower_new`
walks the `bases` chain from the new'd type, collects the ancestors, and calls each one's
**parameterless** `$ctor0` **root-first** (object as `this`, valid because single inheritance
is layout-prefix-compatible) before the derived ctor. Calling every level directly, root-down,
reproduces the implicit chain in one pass — and since explicit `: base(args)` chaining isn't
parsed yet, every ctor *is* implicit, so there's no double-init to worry about. `ctor_chain`
(3-level `Leaf:Mid:Base`, each ctor reading the field its parent just set → 124) → 142;
verify 152/152; AOT 124. Pairs naturally with §89's `base.Method()`. Follow-on: explicit
`: base(args)` / `: this(args)` once the parser keeps the initializer.

## 91. Destructor chaining on `delete` (commit 8d160c6)

The mirror of §90: `delete derived` ran only the derived dtor, so base dtors never fired

base has a dtor still runs it exactly once). `dtor_chain` (3-level, each dtor logs its tag →

## 92. Struct / object initializers — `.{ f = v }` and `new T() { f = v }` (commit 680b0d9)

This one started as a bug hunt. Testing `operator<` overloading (which I assumed was the
gap), the IR showed `store undef` for a `Money x = .{ cents = 100 }` — the operator was
fine; the **`.{ }` initializer was being dropped**. The parser had a deliberate
`consume_initializer` that parsed-and-discarded object/collection initializers ("a later
sprint"), so every `.{ }`/`new T(){}` silently left fields uninitialized. `ref_struct_param`
(§71) had passed only because it *overwrote* the fields afterward — a false green that hid
this for 20 sections. So this is the first session feature that required a **parser/AST**
change, not just sema.

- **Parser/AST:** a new `Expr::Initializer { base, entries }`; `consume_initializer` now
  returns the captured entry exprs (`field = value` → `Expr::Assign`) and the postfix loop
  wraps the base. A nice fallout: since `new`'s operand is itself a postfix expression,
  `new T(args) { … }` is captured for free (it parses as `new (init (call T) …)`).
- **Sema:** target-typed `.{ }` (DotIdent base) allocas the declared struct and stores each
  field; `new T(){}` runs the ctor on the inner base then stores fields through the ref;
  a shared `assign_init_fields` does the stores. `lower_new` gained an `Initializer`-operand
  case so the new-form keeps working.

Gated hard because it touches the parser corpus (updated the `object_initializers` golden
test to the captured shapes) *and* the 152-module LLVM-verify (still 152/152 — real Beef
leans on `.{ }`/`new T(){}` heavily, and they now initialize rather than no-op, while still
verifying). `struct_initializer` (`.{ }` + `operator<` over initialized fields → 87) and
`object_initializer` (`new T(){}` over a ctor → 73) → 145. Follow-on: array collection-init
`new T[N]{ … }` (captured but its bare entries aren't applied to elements yet — the paren
form `new T[]( … )` remains the working array initializer). Lesson: a passing test that
*mutates before observing* can mask a dead initializer indefinitely — prefer observing the
feature under test in isolation.

## 94. Collection initializers — `new List<T>() { v0, v1, … }` (commit 546182e)

## 95. Comptime, first real source — JIT-evaluating `[Comptime]` Beef at compile time (commit 41b8a70)

The headline feature begins. `newbf-comptime` already had the *engine* — `eval_const_i64`
JITs an IR module and calls a nullary `i64()` function mid-compile (its design doc states the
thesis outright: "**Comptime runs on the JIT, not an interpreter** — a deliberate reversal of
Beef, whose `CeMachine` is a bytecode VM"). But that test hand-builds IR; nothing connected it
to real `.bf`. This closes that gap for the **evaluation half**: a `[Comptime]` function
(metadata-only today) lowers to an ordinary `@Program.Sum() -> i64` symbol, so feeding a
fully-lowered program straight into `eval_const_i64` evaluates real Beef source at compile
time. A 1..=100 accumulator folds to `5050`; a `Fib()` calling a recursive `Rec(20)` →
`6765`.

Two things fell out that de-risk the rest of comptime:
- **The dependency-closure worry was moot.** I'd assumed slice 1 had to restrict comptime
  functions to self-contained bodies (no calls into other user code) to avoid gathering a
  lowering closure. But `eval_const_i64` JITs the *whole* module, so any callee (`Rec`) is
  already present — the recursion test proves a comptime function driving a real call graph
  works with zero extra plumbing.
- **The wiring constraint is now pinned.** `newbf-sema` lists `newbf-llvm` only under
  `[dev-dependencies]`, so production `lower_program` genuinely cannot reach the backend — the
  design's "sema invokes comptime via a callback" isn't aspirational, it's required. The
  **fold** (replacing a comptime call site in the compiled program with its literal) therefore
  needs an `eval` callback threaded through `lower_program` + a pre-pass that evaluates comptime
  functions before lowering their call sites. That's the next increment; this one nails
  evaluation end-to-end first, on real source incl. recursion. Integer-only for now (the
  `__real@`/`__xmm` JIT constant-pool gap still bounds what can be JIT-evaluated).

## 96. Comptime, the fold — call sites become literals, the function disappears (commit dcb6165)

The §95 follow-on, and the half that makes comptime *observable* in the compiled program:
not just evaluating a `[Comptime]` function, but **replacing its call site with the value and
deleting the function**. After this, `Main() => (int32)Sum()` compiles to
`add i64 5050, 0; trunc` with `Program.Sum` *absent* from the module entirely.

I'd planned (per §95) to thread an `eval` callback through `lower_program` and re-lower call
sites with values known. I rejected that: it meant a two-pass refactor of the 6 KLOC
`lower_program` and a double full-lowering. The cleaner shape turned out to be a **post-lowering
IR pass** living in `newbf-comptime` (which already depends on `newbf-llvm`, so it can JIT) —
sema stays backend-free, exactly as the dev-dependency constraint demands. Sema's only job is
to *mark*: a new `Module.comptime: Vec<String>` carries the mangled symbols of `[Comptime]`
methods (recorded in `lower_type_at`, the single funnel for non-generic method bodies).

`fold_comptime(&mut module)` then: (1) JIT-evaluates each foldable comptime function (nullary,
`i64`, non-extern — the `eval_const_i64` shape) against the *whole* module, so callees are
present; (2) rewrites every nullary `call @sym` to it; (3) drops the comptime functions.

The one real design choice was **how to materialize the constant without breaking SSA**. A
`call`'s result is `Value::Inst(id)` and other instructions reference that id. Deleting the
call would orphan those uses; rewiring every use means walking all operands. Instead I rewrite
the instruction *in place* to an identity `add v, 0` of the same `i64` type — the id (and every
use of it) stays valid, no operand surgery, and LLVM constant-folds the `+0` away. The IR has
no const-materializing instruction (constants are inline `Value::Const`), so the identity-add is
the minimal honest way to turn an instruction-with-result into a constant-with-result.

**The test is the interesting part.** Folding is behaviorally invisible — a folded call and a
real call both return 5050. The proof that it *folded* is that, post-fold, `Program.Sum` is
gone from the module yet `OrcJit::from_ir` still resolves `Program.Main` and it returns 5050:
an un-folded call would be a dangling symbol that fails to JIT-link. So the assertion is "the
symbol is absent **and** the program still runs" — absence + liveness together. Wired into the
driver's compile/dump-ir/dump-llvm paths; a no-op (early return) for any program without
`[Comptime]`, so the 147-program corpus is untouched. AOT confirms it: `ct_demo.exe` exits 186
(= 5050 & 0xFF). 160 tests green (+3). Follow-on: non-`i64` and non-nullary comptime folds
(needs arg marshalling), and comptime that *emits* types/members back into resolution (the
fixpoint worklist — the genuinely hard part still ahead).

## 97. Comptime fold with constant arguments — `Factorial(5)` at compile time (commit a8b359f)

The §96 follow-on, the "non-nullary" half. A `[Comptime]` call whose arguments are all integer
constants now folds: `(int32)Factorial(5)` compiles to `add i64 120, 0` with the recursive
`Factorial` gone from the binary.

The neat part is **arg marshalling without an FFI calling convention**. `eval_const_i64` only
calls nullary `i64()` symbols. Rather than teach the JIT boundary to pass arguments, I synthesize
a wrapper — `$ct_eval() => Sym(c0, c1, …)` — into a *clone* of the module and evaluate that. The
wrapper is a verbatim copy of the original call site (constant args and their IR types included),
so it's type-correct by construction: if the real call compiles, so does the wrapper. Values are
memoized per `(symbol, args)` so repeated identical sites evaluate once.

The subtle bug was in **dropping**. §96 dropped any comptime function "nothing calls"; a flat
call-scan. But `Factorial` is recursive — after folding `Main`'s `Factorial(5)`, the *only*
remaining caller of `Factorial` is `Factorial` itself, so a flat scan keeps it forever (and the
test caught exactly this: "comptime function should be removed" failed). The fix is to drop on
**reachability from ordinary code**, not on call-count: seed a worklist with the symbols
non-comptime functions call directly, walk the call graph, and keep a comptime function only if
reached. Self-recursion never makes a function its own root, so a fully-folded recursive comptime
function falls out of the reachable set and is dropped — while one still reached from an unfolded
call site (a comptime call with a *runtime* argument, which we deliberately leave as an ordinary
call rather than erroring) stays, with no dangling reference. Three unit cases pin the three
behaviours (const-arg folds / runtime-arg kept / nullary still works) plus the recursive
`Factorial(5)→120` end-to-end (function absent, program runs). AOT: `ct_fact.exe` exits 120.
163 tests green (+3). Still integer-only and still pure-leaf-or-comptime-closure (no comptime
*emitting* types yet — the fixpoint worklist remains the hard part ahead).

## 98. String interpolation — `$"…{expr}…"` (commit 846129d)

The first feature this session to need both a **parser AST change and sema lowering**, and another
"silently wrong" gap like §92's initializers: `$"…"` was *lexed* (the lexer even balances braces
and nested strings inside holes) but the parser flattened it to `Expr::Str`, so `$"x={n}"` emitted
the literal seven characters `x={n}` instead of interpolating. Now it works end-to-end.

**The parser problem was spans.** The lexer captures the whole `$"…{expr}…"` as one token, so the
hole expressions never reach the main token stream. To parse a hole I sub-lex its source slice —
but a standalone lex produces spans relative to the slice, and sema reads every identifier via
`span.text(src)` against the *whole* file. The fix: lex the slice, then shift each token's span by
the hole's absolute byte offset before handing them to a sub-`Parser` over the full `src`. The
produced `Expr` spans are then absolute and sema is none the wiser. New `Expr::Interp { parts }`
with parts alternating `Lit(String)` (escapes decoded, `{{`/`}}` un-doubled) and `Hole(Box<Expr>)`.

**The subtle bit was the format specifier.** Beef holes allow `{value:X}` (format) and `{value,5}`
(alignment), so a naive "parse the whole hole as an expression" hits the corpus gate — `:`/`,` are
trailing tokens. First cut: split on the first top-level `:`/`,`. That *regressed* `Enum.bf`, which
has `{caseHasPayload ? "!" : ""}` — an **unparenthesized ternary**, whose `:` I was eating as a
format separator (C# forces parens here; Beef doesn't). So the real rule, encoded in
`interp_expr_len`: a top-level `:` ends the expression only when no ternary `?` is pending (with
`?.`/`??` not counted as openers). Top-level `,` always ends it. That restored 152/152 and the
format spec is simply dropped for now (no width/precision rendering yet).

**Lowering** rides on §96b's `String.Append`. `lower_interp` does `new String()` then, per part:
a literal run appends byte-by-byte as `char8` (the corlib String is byte-based, so multibyte text
round-trips like a `char8*`); a hole appends through the type-matched `Append` overload —
`Append(String)` for a `String`, `Append(char8)` for a `char8`, `Append(int)` (widened) for any
integer; other types (bool, float) are evaluated for effect and skipped (no overload yet). The
result is an owned `String`, identical in lifetime to a target-typed string literal. `string_interp`
(`$"hi {who}! n={n}"` → length + last char) → 149; parser s-expr tests pin the shapes incl.
format-spec stripping; AOT prints `count = 7`. 165 tests green (+2 parser). Follow-on: format
specifiers (`{x:X2}`), `bool`/float holes, and `scope`-allocated interpolation (today's leaks to
the caller like every other `new`).

## 98b. `bool` holes in interpolation (commit ef38a08)

## 99. Multi-value switch case labels — `case a, b, c:` (commit 8be9c10)

A `switch` bug found by *probing* — compiling small feature programs through the AOT path to see
what silently misbehaves. `case 1, 2, 3:` returned the `default` arm for a scrutinee of `2` or `3`.
The cause was blunt: the parser had a comment-documented loop that consumed the extra
comma-separated case values **and threw them away**, keeping only the first. So every multi-value
case matched only its first value — silently, since it still parsed and ran.

`SwitchArm` now carries `extra: Vec<Expr>` beside the primary `pattern` (kept as `Option` so
`None` still means `default`). The integer-switch lowering folds the per-value `==` into one
`or`-chain (`sv==a || sv==b || sv==c`), reusing the existing equality/coercion path per value;
case-label expressions are constants, so evaluating all of them unconditionally is free of
surprises. Enum-pattern arms still ignore `extra` (comma patterns that bind payloads are rare and
want different handling). Parser print/`sx` render the extra values so `dump-parse` stays complete;
the parser corpus holds 152/152. `switch_multi_label` (a `Classify` over a multi-value switch,
summed across inputs) → 151; parser AST test pins `(case 1 2 3 …)`. 165 tests green. Lesson
restated: the cheapest bug-finder here is to *run real-shaped programs* — this and §92's dead
initializers were both "parses fine, runs, wrong answer", invisible to type-level tests.

## 100. Bare `.Case` labels in an int-backed enum switch (commit 61803c2)

The probing from §99 kept paying out. `switch (Color.Red) { case .Green: … case .Blue: … }`
*ran the wrong arm*. `dump-ir` showed why, starkly: `icmp eq %2, undef`. An int-backed enum
(`enum Color { Red, Green, Blue }`) lowers to `int32`, so by the time a switch arm's bare `.Green`
pattern is lowered, the scrutinee's IR type is just `int32` — the enum identity is gone.
`self.expr(.Green)` routes a bare `DotIdent` to `try_enum_construct_dot`, which only knows *payload*
enums, so it returned nothing and the pattern fell through to `undef`. The comparison against undef
is UB; LLVM picked a branch, so it sometimes "worked" (that's why §99's single-label probe looked
fine) and sometimes silently ran the wrong case. Qualified `Color.Green` labels were never affected
— they go through `try_enum_const`, which has the enum name in hand.

The fix mirrors the language rule: **the scrutinee determines the enum.** The lowerer now tracks
each local/param whose declared type is an int-backed enum (`enum_locals: name → enum name`,
populated where locals and params are bound), and a new `lower_case_value` resolves a bare `.Case`
label against the scrutinee's enum (when the scrutinee is such a tracked local/param) by looking up
`structs.enums[enum][case]` for the constant — otherwise it lowers the label as an ordinary
expression. Because int-backed enum switches go through the *integer* path, this composes with §99:
multi-label enum cases (`case .Red, .Blue:`) now work via the same `or`-chain. Limitation: the enum
is recovered only for a scrutinee that's a plain local/param (not yet a field or call result) — the
common shape; the rest fall back to the old expression path. `enum_switch_dotcase` (a `Classify`
mixing bare, multi-value, and `default` arms, with a wrong-arm case that previously slipped through
the UB) → 152. 166 tests green. The `dump-ir` `undef` was the tell — when a value "sometimes works",
look for undef in the IR before suspecting the logic.

## 101. Value-struct constructor shorthand — `Vec2 v = .(2, 3)` (commit c5c5bdc)

Probing again: a `struct` with a `this(int32, int32)` constructor, built with the idiomatic
`.(args)` shorthand, produced garbage — `Vec2 a = .(2, 3); return a.x` gave nonsense. The
constructor never ran. Value structs could only be built via the §92 `.{ field = value }`
initializer; the *constructor* path for value types was simply missing. (`Vec2 a = Vec2(2,3)`
didn't even compile — it parses as a call to a non-existent function `Vec2`.)

`.(args)` parses (per the leading-`.` shorthand in `primary`) as a `Call` whose callee is a
bare-`.` `DotIdent` — `name.text(src) == "."`, distinct from a *named* `.Case(args)` enum
constructor. `try_target_typed_ctor` recognizes that shape against a value-struct target and
`construct_value_struct` allocas a slot, runs the arity-matched `$ctor{N}` through it (the ctor's
`this` is a pointer into the body, so it writes the fields in place), and loads the value. The
ctor symbols already existed — value-struct ctors register and lower like any other; only the
*invocation* was unwired.

Wired into the two target-typed sites where the target type is known: local init and `return`.
The `return` hook is what makes value-struct-returning **operators** work —
`operator+(Vec2 a, Vec2 b) { return .(a.x+b.x, a.y+b.y); }` now constructs its result. So three
things that were all broken — `.(…)` locals, value-struct operator results, and reading their
fields — fixed by one path. `value_struct_ctor` (ctor + `operator+` + a `Dot` method) → 153.

The remaining hole: `.(args)` as a **call argument** (`obj.Method(.(1, 1))`). It needs target-
typed argument resolution, which fights the current eager evaluation — overloads are picked from
arg *types*, but a `.(…)` arg has no type until its target (the parameter) is known. That two-
phase resolution (pick overload structurally, then lower args against param types) is the broader
"target-typed argument" work flagged since §57; deferred. 166 tests green.

## 102. Target-typing the RHS of an assignment (commit 1c2c999)

Probe found the §101 sibling immediately: a constructor body that builds value-struct fields,
`Outer.this(x, y) { this.a = .(x); this.b = .(y); }`, gave garbage. §101 wired `.(args)`
target-typing into local-init and `return`, but **assignment** evaluated its RHS *before* the
target type was known — same eager-evaluation trap, different statement. `self.expr(.(x))` with no
target produced an undef-ish struct, stored into the field.

Fix: a plain `=` to a resolvable place now resolves the **place first**, then target-types the RHS
against the place's type through the same chain a typed local-init uses
(`enum → ctor → tuple → initializer → plain expr`). This also shifts evaluation of `a.b = rhs` to
place-then-RHS — left-to-right, matching Beef/C# — which is the correct order anyway. Compound
(`+=`), `??=`, indexer, and property-setter assignments keep their existing paths untouched (they
have their own read-back / setter logic). One behaviour, three constructs unblocked: `.(args)`,
`.Case`, and `.{ }` all now construct correctly on an assignment RHS, so value-struct field
construction in constructors works. `struct_field_ctor_assign` (Outer ctor building Inner fields)
→ 154. (A payload-enum reassignment `o = .Some(42)` followed by `if (o case .Some(let v))` still
trips a *parser* diagnostic — a separate, pre-existing issue, not this lowering path.) 166 green.

## 103. Field default initializers, and `scope` allocation (commits c100c35, 8f32233)

Two probe-found gaps in one sitting, both core.

**Field defaults.** `class Foo { int32 v = 9; }` dropped the `= 9` outright — the layout builder
read the type and name but never the `init` expr, so the field stayed zero. Now a *constant*
default (int/float/bool/char/null literal, incl. negated/parenthesized) is evaluated at
layout-build time — keeping the `StructTable` lifetime-free, since storing the borrowed `Expr`
would need a lifetime parameter — and recorded **by field name**. At construction
(`emit_field_inits`) it's applied before the ctor body; keying by name (not index) is what lets it
survive inheritance's field reindexing — the helper walks the type and its bases and resolves each
name in the composed layout. Wired into `new`, the `.(args)` value-struct ctor, and the `.{ }`
initializer (defaults first, explicit entries override). `field_default_init` (base + derived
defaults read by a ctor) → 155.

**`scope`.** `scope T(args)` was entirely unhandled — it fell through to undef (garbage). It now
heap-allocates like `new` and registers the instance for an automatic destructor + free at the
enclosing block's exit (no manual `delete`). The dtor-chain+free logic was factored out of
`lower_delete` into `emit_destroy`, shared by both. Cleanup runs on a block's normal exit and on
every open frame before a `return`, mirroring `defer` (and sharing its `break`/`continue` gap).

The instructive bug: my first cut tracked *every* `scope` value and freed it at block exit — and
4 corlib modules (String, Runtime, NumberFormatter, Params) failed LLVM verify with **"Instruction
does not dominate all uses."** A `scope` inside a *conditional sub-expression* (a ternary/short-
circuit branch) defines its value in a branch block that doesn't dominate the block's merge/exit,
so emitting `free(v)` there is an SSA violation. The fix is a conservative dominance guard: track a
`scope` allocation only when it's made in its frame's **entry block** (recorded when the lexical
block is entered) — a value defined there provably dominates the block's exit. Conditional `scope`
allocs aren't tracked and leak for now (a real lifetime analysis is the follow-on). `scope_alloc`
proves the auto-free with a dtor-decremented live counter that reads 0 after the block → 156. The
verifier earned its keep again: "does not dominate all uses" pinned the exact unsoundness. 166 green.

## 104. Multi-variable C-style `for` loops (commit 2f926a4)

Probe: `for (int i = 0, j = 10; i < j; i++, j--)` — a two-pointer walk — produced ~10 parse
diagnostics and a wrong result. Two `for` clauses were single-valued: the init parsed one
declarator (so `, j = 10` cascaded into the diagnostics), and the update kept only the first
expression (`i++`), silently dropping `j--` — the same "keep first, discard rest" anti-pattern as
the §99 switch labels and §101's call args.

`Stmt::For` gains `init_extra: Vec<Stmt>` and `update_extra: Vec<Expr>`. The init's extra
declarators each **re-use the leading type** (`int i=0, j=10` → both `int`), parsed by cloning the
first `Local`'s `ty` — they can't be a `Block` because the loop's init/cond/update/body share one
scope, and a block would scope the vars away. Sema lowers the extra inits in that shared loop scope
and runs every update in the continue block. The extras are threaded through the three `for`-walking
pre-passes too (tuple registration, generic-instantiation collection, closure-capture detection) so
a generic/tuple/closure introduced only in an extra declarator isn't missed. `for_multi_var` → 157.
166 green. Pattern worth internalizing: any comma-list the grammar allows (case labels, ctor args,
for clauses, …) is a place the parser might be silently keeping only the head — worth auditing.

## 105. Multi-declarator local statements (commit 36e44b5)

The §104 audit-note paid off on the very next probe: `int a = 1, b = 2, c = 3;` declared only
`a` — `try_typed_local_init` literally parsed the first declarator and `let _ = self.expr()`-ed
the rest away. So `b` and `c` were undefined identifiers (garbage). The fourth instance of "keep
first, discard rest" this session.

Unlike the for-loop, a statement-level declarator group can't be a `Stmt::For`-style extra vec on
one node — and it can't be a `Block` (that would scope the names away). So: a new
`Stmt::Locals { decls }`, a **scope-transparent** group. Sema lowers each decl in the *current*
scope (no `scopes.push`). The cost of a new `Stmt` variant is breadth — every stmt walker that
handles `Local` must also recurse into `Locals`: lowering, the tuple-registration, generic-
instantiation, lambda-collection, and closure-capture pre-passes, and both AST printers. Missing
any one is a silent gap, so I walked the `Stmt::Local ` call sites exhaustively (grep) and covered
each. A single declarator still yields a plain `Stmt::Local` (no group), keeping the common case
and all existing golden output unchanged. `multi_declarator_local` (5 vars in one decl) → 158;
parser test pins `(locals (var a 1) (var b) (var c 3))`. Parser corpus stays 152/152. 167 green.

That's four comma-list fixes (§99 case labels, §101 ctor-call args, §104 for clauses, §105 locals)
— the audit list from §104 is now essentially cleared for declarations and control flow.

## 106. Empty initializers `.{}` / `new T() {}` (commit db90612)

A one-line probe found it: `Pt p = .{};` returned garbage. The postfix initializer loop only wrapped
a base in `Expr::Initializer` when `!entries.is_empty()` — an *empty* brace pair was consumed and

## 107. Qualified generic-method calls — `Type.Method<T>(args)` (commit c49d4aa)

Probe: `Util.Pick<int32>(3, 9)` returned garbage, while the §35 bare same-class form
`Identity<int32>(x)` works. Both the instantiation collector and the call lowering matched a generic
callee only when its base is a bare `Expr::Ident`; a qualified `Type.Method<Args>` has a
`Expr::Member` base, so it was skipped — no monomorph emitted, call resolved to nothing. (The
debugging was briefly muddied by naming the probe method `First`, which *collides* with corlib
`List.First` under global-name mangling — a real demonstration of the §35 caveat. Renaming to
`Pick` isolated the actual bug.)

Since generic methods mangle by name only (no owner), a qualified and a bare call map to the same
monomorph — so the fix is just to extract the method name from either base shape. A
`generic_callee_name` helper returns the name for an `Ident` or a `Member` base; both the collector
(`collect_insts_expr`) and the lowering use it. `generic_method_qualified` → 160. 168 green. The
remaining §35 work is *type-aware* mangling (by owner + arg types) to kill the global-name collision
risk and unlock instance generic methods (`obj.Map<R>(f)`) — still a follow-on; this just makes the
qualified *static* call resolve.

## 109. Generic-methods core wave — owner-mangling, instance methods, guards (commits 7dae6dc, 47248a3, acb9e23, 4b5ade7)

- **GM-A1 (re-key, verified no-op).** Generic methods now key on a composite `(Option<StructId>,

  hardcoded `None` so emitted symbols stay byte-identical (`@Pick$i32` confirmed via dump-ir). Added
  `Lowerer.cur_type` (threaded, unread). Pure data-structure reshape to make A2 possible.
- **GM-A2 (owner determination — fixes the §107 collision).** Decls key under both `(Some(owner),

## 110. fn-values Slice A — uniform Func$ values, the §49 segfault fixed (commits d3a1fdb, 56d35f4)

The second sprint-plan feature, and the one that fixes an actual **crash**. Since §49 a *capturing*
closure passed into a generic HOF (`Map<int,int>(xs, a => a + b)`) segfaulted: closure-ness was
tracked by *local name* in the `Lowerer`, so once the value crossed the call boundary into `Map` the
convention was lost — the callee's `f(x)` took the non-closure branch and called the env pointer as
code with the args shifted. Verify-clean (LLVM builds an indirect call's type from its *args*), so
only the run corpus could ever catch it.

- **FV-T1 (foundation, no-op).** Registered `$Func = {code:Ptr, target:Ptr}` as `StructId(0)` with a
  default-id-hazard assertion, and added a position-gated `lower_value_ty` helper (unused yet).
  `lower_ty_env(Function)` stays `Ptr`, so C-ABI function-pointer fields/casts/externs are untouched.
- **FV-T2+T3+T4 (atomic).** The design (and a discarded T2-alone attempt that *reproduced the
  segfault on `list_hof`*) proved these can't separate. A function value is now a two-word value
  struct called through **one** shape — `load code+target; call_indirect(code, [target, args…])` — so
  the convention travels *with the value*, which was the whole bug. Every `$lambdaN` is `$self`-leading
  (captures at `$self[i]`, slot-0 code pointer dropped); static method refs get a `$mref$` thunk that
  drops `$self`; `coerce` auto-wraps a bare `Ptr` to `Func${code, target=null}` (and `null` →
  `Func${null,null}`, so `function f = null` is defined; `f == null` is a code-field compare). The
  entire name-keyed `closures` mechanism was deleted.

Two judgement calls worth recording. **Position gating is load-bearing:** `Func$` applies only to
`function`-typed *param/local/return* positions, and only when `!is_delegate` — the parser unifies
`function` and `delegate` into one `AstType::Function{is_delegate}`, and widening a `delegate`-typed
position (or a C-ABI `function`-typed *field* like `BfRtCallbacks`'s ~28 callback slots) would break
the ABI/layout and the verify corpus. Gating kept Array/Lazy/Platform delegates and BfRtCallbacks
bare. **Run-corpus is the authoritative gate here**, not verify — the new `closure_arg.bf` (the §49
crash, now → 36), `closure_capture_two→21`, `mref_static_arg→30`, `fn_null→7` are the real proof, plus
a `Module.funcs` unit test asserting every `$lambda*`/`$mref*` leads with a `Ptr $self`. The §49 fix
confirmed end-to-end: `closure_arg` AOT-runs to 36, and the IR shows `@Map$i32i32(&s10, %s0)` — the
function param is the `$Func` struct, dispatched uniformly.

Gates throughout: verify 153/153, parser 153/153, run-corpus 169, suite 182/0; no `newbf-llvm` dep in
sema; no dead code. This unblocks the marquee: **GM-B1** (instance generics on a generic owner,
`List<int64>.Map<R>`) → **GM-B2** (migrate the corlib HOF to `xs.Map<R>(f)`) now sit on a correct
function-value convention. Deferred to later FV slices: bound method-refs (T5), inline-lambda-as-arg
(T6, needs the targeted-args arg-loop settled first), `closure_returns_fn` (T7), the Delegate bridge
(T8); by-reference capture and >8-byte captures remain out of scope (§10).

## 111. The marquee — `xs.Map<R>(f)` on a real `List<T>` (commits de623f9, 7815053)

The generic-methods feature's capstone, and the convergence proof for the whole sprint wave.

- **GM-B1 (generics on generic owners).** An instance generic method whose owner is *itself* generic

## 112. Dynamic interface dispatch — the itables feature, complete (commits 89f1d54, a1b96b1, ef240dc, 700d7bf, 68dde61, 62f2544, IT-T8)

The third sprint-plan feature, and the one that closes the single biggest *distinctive* language gap:
calling a method through an interface-**typed** value. Before this, only the generic-constraint form
(`F<T>(T s) where T : IShape`) worked (monomorphization erases the interface); `IShape s; s.Area()`
returned `undef`. The completeness estimate (§prior) put interface dispatch at ~40% — it's now
functional for the common case.

The chosen representation (per the design doc, validated by review) is **object-header vdata, not fat
pointers**: an interface value stays a plain 8-byte `Ref(iface_id)`, and dispatch goes through the
receiver's existing `$header` vtable at a globally-fixed per-interface slot base — identical in shape
to a virtual call, no new `IrType`, no new IR instruction, **zero `newbf-llvm` changes** across the
entire feature. Seven tasks:

- **T1** — register interfaces as `StructKind::Interface` (→ `Ref(iface_id)`, empty StructDef). The
  make-or-break hazard: once interfaces are `Ref`, a `class X : IFace, Base` would record the interface
  as `X`'s inheritance base — corrupting `apply_inheritance`. The base-routing guard
  (`matches!(kind, Ref)`) shipped atomically. The agent also caught three out-of-scope paths the doc
  hadn't enumerated (generic interfaces must stay `Ptr`; interface members must not leak into
  `methods[iface]`; default-method emission stays off until T6) — exactly the judgment delegation
  needs.
- **T2** — populate `imethods` (per-interface slot signatures, base-first, static/generic filtered),
  `idefaults`, `iface_bases` (transitively flattened), `explicit_impls`.
- **T3** — `apply_itables`: assign each interface a **global slot base** `N + cumulative` (N = max
  vtable length over ALL ids incl. monos), resolve each class's per-slot impl (explicit →
  `pick_overload` incl. inherited → default → null), ABI-check it, and grow the class vtable
  bounds-safely. The slot-base bounds are the keystone — an off-by-one is a silent runtime OOB in T5.
- **T4+T5** — the milestone: a SEPARATE dispatch branch *before* the methods-keyed block (an abstract
  interface method is in `imethods`, never `methods`, so the old block would miss it), sourcing the
  sig from `imethods`, reading `$header` via a **raw** `elem_addr(body, Ptr, 0)` (interfaces have an
  empty StructDef — no `field_addr`), indexing `vtable[iface_slot_base + midx]`, `call_indirect`. 8
  programs incl. the tricky coexistence cases (a method that's both a vtable slot *and* an interface
  slot; an interface impl satisfied by an inherited base method).
- **T6** — default interface methods: emit the default bodies with `this : Ref(iface_id)` (reconciled
  to the exact `idefaults` symbol — also fixing a latent ABI bug where the slot pointed at a
  `this`-less function); a bare sibling call inside a default dispatches through `this`'s interface
  vtable. Override precedence falls out of T3's resolution order.
- **T7** — `is`/`as` against an interface: `type_test` reads the header via the raw GEP (so an
  interface-typed *source* works) and uses the `iface_bases` membership set for an interface target
  (transitive, so `x is IA` matches a class implementing only `IB : IA`); the class path is unchanged.
- **T8** — `InterfaceDispatch.bf` verify-corpus pin (→154/154) + this entry.

13 new run-corpus programs, all run-corpus-verified (the authoritative gate — a wrong slot index is a
verify-clean miscompile). Gates throughout: verify **154/154**, parser **154/154**, run-corpus **184**,
suite green; no `newbf-llvm` dep. Deferred (per doc §10): boxing value structs to interfaces, generic
interfaces/methods (needs the GM machinery to extend to interfaces), `delete`/`GetType` through an
interface value, per-class slot offsets (the global-base scheme reserves the union; fine for v1).

Three of the four design-wave features are now landed: **generic-methods (complete)**, **fn-values
(Slice A — the §49 fix)**, **itables (complete)**. Remaining: **targeted-args** (two-phase overload
resolution) and the later **fn-values** slices (T5 bound method-refs, T6 inline-lambda args, T7
returns, T8 Delegate bridge).

## 113. Target-typed call arguments — two-phase overload resolution, complete (commits cf9f10f, f980370, 65b63f9, 7b4cf56, 8bcfc2e, TA-9)

The fourth design-wave feature, and the one that finishes the long §57→§101→§102 target-typing arc:
a target-typed dot-form — `.(args)` (value-struct ctor), `.{ … }` (initializer), `.Case`/`.Case(payload)`
(enum) — now works as a **call argument**, not just in local-init / return / assignment-RHS. Before
this, `obj.M(.(3,4))` lowered the `.(3,4)` to `undef` (no target type known) and silently
mis-resolved/miscompiled.

The obstacle was structural: every call site lowered args **eagerly** to `(Value, IrType)` *before*
overload resolution (`pick_overload` needs arg types), but a dot-form has no type until its target
(the parameter) is known — a chicken-and-egg. The fix is **two-phase resolution**:

- **TA-1 (foundation).** A pure `arg_is_pending` classifier; a shared `lower_arg_targeted` (the
  canonical enum→tuple→ctor→initializer order, unifying the three previously-divergent inlined
  chains); and a latent-bug fix — `try_target_typed_initializer` no longer returns a silent
  `(undef(Ref),Ref)` for a `.{ }` against a class target.
- **TA-2 (machinery).** `ArgShape`/`PendingKind`; `pick_overload_partial` with the **shape gate** — a
  pending slot scores +1 if its kind is compatible with the formal (`.(args)`/`.{}` ↔ `Struct`,
  `.Case` ↔ the payload enum holding that case) and **disqualifies the candidate** if not (the
  wrong-pick guard — `M(Vec2,int)` vs `M(int,int)` on `M(.(…),5)` can't pick the primitive overload);
  `ctor_for_partial`; and `lower_args_phase1`/`finish_args` (phase 1 lowers concrete args in source
  order into holes; phase 2 lowers each pending arg against its *resolved* param type). `pick_overload`
  became a thin all-Concrete wrapper (identical picks — the behavior-preservation proof).
- **TA-3 (milestone) + TA-5/6 + TA-4/7/8 (the rest).** A `has_pending` fork at each call site — method
  calls, `new T(args)`, value-struct `.(args)` (incl. **nested** `.( .(…), .(…) )` via reentrancy),
  bare/free-fn/fn-ptr calls, generic-method calls, enum-payload args, variadic, ref/out. The
  invariant everywhere: **non-pending → the existing eager path verbatim** (so the 184 prior programs
  are untouched); pending → `phase1` once, resolve, `finish_args` once, all in the current block,
  exactly one construction per pending arg.

Two things worth keeping. **Evaluation order** is a documented caveat, not a false equivalence claim:
concrete args emit in source order, pending args after them — pinned by `targ_eval_order` (a
side-effect counter) rather than asserted equivalent. And **TA-7's collection recursion** is the one
genuinely new mechanism: a generic monomorph referenced *only inside a pending arg's body* (e.g.
`M(.{ v = Pick<int32>(9) })`) must be collected or it's a dangling symbol — proven load-bearing
(`targ_generic_nested_arg` returns 9 with the `Expr::Initializer` collector arm, 1/dangling without).

~16 new run-corpus programs, all run-corpus-verified (the authoritative gate — a wrong pick or
double-emit is a value error the verifier can't catch). Gates throughout: verify **154/154**, parser
**154/154**, run-corpus **200**, suite green; no `newbf-llvm` dep. Deferred (doc §10): full
single-pass eval order, `.{ }`-against-class semantics, comptime/default/named args.

**All four design-wave features are now landed:** generic-methods, fn-values (Slice A), itables, and
targeted-args. The remaining sprint-plan work is the later **fn-values** slices — T5 bound
method-refs, T6 inline-lambda-as-arg (now unblocked, since TA-3 settled the arg loop), T7
`closure_returns_fn`, T8 the Delegate bridge.

## 114. fn-values tail — inline-lambda args, bound method-refs, returned closures (commits b4fbecb, c3d281d)

Closes the fn-values feature on top of Slice A (§110). Three slices, completing the natural
function-value surface:

- **T6 — inline lambda as a call argument.** `xs.Map<int32>(x => x*10)` now works directly, no
  intermediate `function`-typed local. The obstacle was the lambda *collection* pre-pass
  (`collect_lambdas`) only matched the local-init shape; T6a extends it to walk every expression
  position (`collect_lambdas_expr` over call/generic/index/member/binary/ternary/initializer/tuple
  args + lambda bodies), assigning each inline `Expr::Lambda` a `$lambdaN` by span. T6b sources the
  lambda's param *types* from the resolved callee param — `MethodSig.param_fn_sigs` carries each
  function-typed param's `(ret, ptys)`, and `record_inline_lambda_sigs` keys the resolved sig by the

  inline lambda is its own `Func$` producer, so it lowers as a normal arg + `coerce(Ptr→Func$)`.
  Capture works (the creation site is the call expr, inside the live enclosing scope) —
  `lambda_direct_arg→90`, `lambda_direct_arg_capture→69`.
- **T5 — bound instance method-refs.** `obj.M` in a value position becomes `Func${code=$mrefb$<full>,
  target=receiver}`; the `$mrefb` thunk forwards `$self` as the method's `this` (a class `this` and
  `$self` are the same opaque ptr, so the cast is free). Distinguished from the static `$mref$` by
  whether the base names a type vs an instance value. Class receivers only; value-struct/`mut`/`ref`
  receivers and virtual-through-bound-ref are deferred (the test uses a non-virtual method).
  `mref_bound_arg→50`.
- **T7 — returning a closure.** `static function int32(int32) MakeAdder(int32 n) { return x => x+n; }`
  then `add5(10)→15`. The `Func$` return type already worked, but T6a exposed a latent bug — a lambda
  in a *return* position was collected as inline yet had no target sig, emitting **paramless**. Fixed
  by threading `ret_fn_sig` through `lower_method` and a `record_return_lambda_sig` at the `Return`
  site (the return-type analogue of T6b's call-arg mechanism). `closure_returns_fn→15`.

Lifetime caveats (documented, not "bugs"): by-value capture survives a return **only because the env
is heap-`malloc`'d and leaked**; a captured mutable outer local does **not** observe later writes (an
*observable* divergence from Beef's by-ref capture); only ≤8-byte scalar captures (capturing a `Func$`
is unsupported); an in-loop closure builds/leaks an env per iteration. By-reference capture, wide
captures, value-struct bound receivers, and virtual-through-bound-ref are the fn-values follow-ons.
T8 (the `System.Delegate` bridge) stays design-only until a `Delegate` type exists.

Gates: build clean, suite green, verify 154/154, parser 154/154, run-corpus 204; no `newbf-llvm` dep.

**The four-feature sprint wave is complete.** generic-methods, fn-values, itables, and targeted-args
— designed in §108, executed across §109–§114 — all landed behind green gates, the manage-the-loop way
(each task delegated, then diff-reviewed, gate-re-run, IR/AOT-spot-checked, and committed by the
reviewer). Run-corpus grew 160 → 204; verify/parser 152 → 154. The only deferred items are the
explicitly out-of-scope follow-ons noted in each design doc's §10.

## 115. Wave 2 design — the distinctive Beef frontier gets design docs + a sprint plan (docs/design/)

With the dispatch/representation wave done, the next frontier is the features the language *isn't
really Beef without* — so, same process as §108: a multi-agent design wave (26 agents, ~2.3M tokens; 5
research → per-feature design → 3 adversarial lenses → revise → synthesized plan) producing
implementation-ready docs for four features, NOT code. Targets (PLAN phases 4/7/8/9):

- **memory-safety** — Beef's signature: a runtime debug guard (a *quarantining* stomp allocator +
  tombstone ledger in newbf-runtime, deliberately diverging from upstream's page-recycle so a UAF
  faults deterministically for the whole debug run) hooked into `new/delete → newbf_alloc/newbf_free`
  for JIT *and* AOT, **plus** a conservative compile-time delete-flow pass (provable double-free / leak).
- **comptime-breadth** — comptime that *emits* members/types feeding back into resolution, via an outer
  fixpoint loop in newbf-comptime (emit Beef `extension Owner {…}` source → reparse → re-analyze →
  re-lower until fixpoint), keeping the dependency direction comptime→sema (no circular dep).
- **reflection** — runtime type metadata: a Beef-faithful `ClassVData {i32 mType, [N×ptr] vtbl}` header,
  `typeof(T)`/`GetType()` via an *in-module LLVM-emitted* `__newbf_type_by_id` accessor (no Rust runtime
  crate — the verified fix for the symbol-resolution blocker), policy-gated (`[Reflect]`) metadata tables.
- **mixins** — hygienic AST→IR splices reusing the live `Lowerer` (so control-flow escape and SSA
  dominance are free), unblocking `Try!`/`Result`.

This wave is materially riskier than wave 1 (which was four sema-dispatch refinements): it touches the
**runtime crate, the JIT/AOT symbol-resolution seam, and the object `$header` ABI**. The adversarial
review earned its keep again, catching code-specific flaws the drafts had wrong: comptime's
"wrap-in-`class Owner` and re-lower" plan trips the duplicate-type check and `fill_fields_at` clobbers
fields → replaced with `extension` + a dedicated substrate task; reflection's `typeof` currently
*discards* its type (parser.rs:1136) and `newbf-tests` has **no** `newbf-runtime` dep (so AOT can't link
a Rust shim) → an in-module LLVM accessor instead; memory-safety's symbols are host-EXE Rust symbols not
PE exports → ORC absolute symbols, not the unsound process-export. I spot-checked every load-bearing
file:line against the tree (jit.rs:113/152 process-generator-only, the six malloc sites + the `−8`
frees, parser.rs:1136 typeof-discard, `Mixins.bf` in the verify corpus, the missing runtime dep) —

Output: [`docs/design/memory-safety.md`], [`comptime-breadth.md`], [`reflection.md`], [`mixins.md`],
and [`SPRINT-PLAN-2.md`] (9 review-batches, ~33 tasks). The plan's spine: **MS-T0 first** — the ORC

assign one task to an agent, review/test/commit, repeat.

## 116. Wave 2 execution begins — MS-T0: the ORC absolute-symbol seam (jit.rs, newbf-runtime)

First task off SPRINT-PLAN-2, and the one the plan demands first: the JIT seam that lets JIT'd Beef
code call a Rust `fn` living in the host EXE. This neutralizes the wave's #1 risk (R1) *before* anything
depends on it — purely additive, no alloc-path rename, no behavior change. The problem: `OrcJit`
resolved externals only through `DynamicLibrarySearchGeneratorForProcess` (PE exports — the CRT's
`malloc`/`free`), which can never find `newbf_alloc`/`newbf_free` (Rust fns, not PE exports). Memory-
safety needs `new/delete → newbf_alloc/newbf_free`; comptime emission needs `__newbf_ct_emit`. The fix
is **ORC absolute symbols**.

Landed:
- **newbf-runtime** now exports the two stable C-ABI thunks `newbf_alloc(i64 size, i32 type_id, i32
  site_id) -> *mut u8` and `newbf_free(*mut u8)` (MS-T0 bodies are malloc/free wrappers — a *resolution
  seam* only; MS-T1 swaps in the stomp allocator without touching the ABI), re-exports
  `newbf_install_crash_handler`, and sets `crate-type = ["rlib","staticlib"]` (rlib so newbf-llvm can
  take the runtime fn addresses; staticlib for the MS-T3 AOT link).
- **newbf-llvm** gains a clean `newbf-llvm → newbf-runtime` edge (newbf-runtime is a pure leaf — no
  cycle) and `OrcJit::add_absolute_symbol(name, addr)`: mangle+intern the name with the JIT's *own*
  mangler (`LLVMOrcLLJITMangleAndIntern` — sidesteps the platform global-prefix/underscore question by
  construction, vs the doc's `…SessionIntern` sketch), build a one-entry `LLVMOrcCSymbolMapPair`
  (`Exported|Callable`), `LLVMOrcAbsoluteSymbols` → MaterializationUnit → `LLVMOrcJITDylibDefine`, with
  the consume/dispose ownership handled on each path. `from_ir` registers `newbf_alloc`/`newbf_free`/
  `newbf_install_crash_handler` *before* adding the IR module; no duplicate-definition error because
  these names aren't PE exports and an explicit JITDylib definition wins over the on-demand process
  generator. (The refactor constructs the `OrcJit` early so the existing `Drop` disposes the LLJIT on
  every error path — verified `impl Drop for OrcJit` at jit.rs:293, so no double-dispose.)
- **The load-bearing proof:** a newbf-llvm smoke test JITs `newbf_alloc(16,-1,0)`, writes+reads a
  sentinel through the returned buffer, asserts `(readback==sentinel) && (p!=null)`, then `newbf_free`s
  — runs fault-free, returns 1. A second test binds an arbitrary host fn under a custom name (the
  generic mechanism CB-T2 will reuse for `__newbf_ct_emit`).

Gates: newbf-llvm 24/24 (2 new), newbf-runtime 6/6, parser 154/154, verify 154/154 (14235 fns lowered
clean), run-corpus 204/204 — all unchanged, confirming the seam is additive. One edit, all hosts (run-
corpus, comptime, driver all go through `from_ir`). R1 retired.

## 117. MS-T1 — the deterministic memory guard engine (newbf-runtime/src/guard/, runtime-only)

The marquee Beef feature's *engine*, landed entirely inside the leaf `newbf-runtime` crate (zero new
deps): a quarantining stomp allocator + a tombstone ledger, behind a runtime mode flag. ~1300 lines of
new guard code + 20 tests. The MS-T0 malloc/free thunk bodies now route through `guard::route_*`.

- **`vm.rs`** — a `Vm` trait (reserve/commit/decommit/release) so the allocator is unit-testable with a
  `MockVm` (real page-backed memory, no OS faults) and runs on a `Win32Vm` (`VirtualAlloc`/`VirtualFree`
  with `MEM_RESERVE`/`MEM_COMMIT`/`MEM_DECOMMIT`, declared via `extern "system"` matching crash_dump.rs
  — no `windows-sys` dep) in production; a POSIX `mmap` impl keeps the crate cross-buildable.
- **`stomp.rs`** — the quarantining allocator: each alloc commits its own page region (header at
  page-start, front-aligned user ptr); the range cursor only advances — **freed pages are decommitted
  but never released or recycled**, so a later read/write to a freed pointer faults *deterministically*
  (the UAF property). Size-0 and page-multiple sizes get a `+1` slack page (the StompAlloc bump).
- **`ledger.rs`** — a side-table keyed by user-ptr with **persistent tombstones**: a free of an
  already-tombstoned ptr = double-free; a free of a never-seen ptr = wild-free. The verdict is computed
  purely from the side table — the possibly-decommitted page header is **never dereferenced**
  (review-correctness #7).
- **`mod.rs`** — the `Stomp|Thunk` mode (default **Thunk** = malloc/free passthrough, line 53), the
  `Guard<V: Vm>` (stomp+ledger under one Mutex), and the **ledger-first lock-free abort path**:
  `check_free` returns `Err(DoubleFree|WildFree)` *without aborting* (the testable seam), and only
  `stomp_free` aborts — after releasing the lock; the crash dump reads only the lock-free `crash_dump`
  atomics. Lifecycle: `newbf_set_guard_mode`, `newbf_guard_reset` (release ranges + clear ledger between
  programs), `newbf_guard_report_leaks`, `newbf_guard_enter/exit_comptime` (a nesting depth; comptime
  allocs are tagged and excluded from leak reports). Metrics publish through the existing
  `update_guard_metrics`/`note_free`.

The load-bearing design choice: **the default mode is passthrough**, so the guard is inert until
MS-T3's host wiring calls `set_guard_mode(Stomp)`. That keeps MS-T0's JIT smoke test and the entire
run-corpus green with the new code present but dormant. The two correctness properties that matter most
— quarantine (freed pages never recycle) and ledger-first detection (never deref decommitted memory) —
are each pinned by a dedicated test (`quarantine_never_recycles_freed_address`,
`guard_check_free_classifies_without_aborting`).

Gates: newbf-runtime 26/26 (+20), newbf-llvm MS-T0 smoke test green, parser 154/154, verify 154/154,
run-corpus 205/205 — all unchanged (default passthrough). `newbf-runtime` is still a zero-dep leaf. The
live UAF-faults/double-free-aborts observation is MS-T3 (the `guard_corpus` child-process harness).

## 118. MS-T2 — the alloc path becomes guard-routable (heap_alloc, lower.rs)

The rename that makes the guard reachable: every heap allocation now goes through `newbf_alloc` and
every free through `newbf_free` (MS-T0's absolute symbols), via a shape-aware `heap_alloc(size,
AllocKind, ret_ty)` — `AllocKind { Object(StructId), Array{header_bytes}, Raw }`, emitting
`newbf_alloc(size:i64, type_id:i32, site_id:i32)` (`type_id = StructId.0` for objects, `-1` for
array/raw; `site_id = 0` until MS-T7). All five `malloc` sites (closure env→Raw, array→Array, object/two
String paths→Object) and all three `free` sites are converted. Because MS-T0 pre-registered the
symbols, there is **no red window** — the renamed calls resolve in every JIT host.

**A design-doc correction worth recording.** The doc's §A2-array said to *drop* the array's `−8` base
reconstruction on the free side ("free the elements pointer; the ledger maps it"). That is **wrong as
sequenced**, and the run-corpus would have caught it as a heap-corruption crash: (1) at MS-T2 the
run-corpus still runs in **default Thunk mode**, where `newbf_free`→CRT `free`, so the freed pointer
must be *exactly* the `malloc`-returned base — `free(elements)=free(base+8)` corrupts the heap; and
(2) the already-landed MS-T1 ledger keys each allocation by the pointer `newbf_alloc` returns (the
base), and the 3-arg C-ABI carries no header offset, so even in Stomp mode free must pass the base. So
the `−8` reconstruction is **kept** (only the symbol name changed) — correct in both modes, consistent
with the landed ledger. This is exactly the kind of verify-clean miscompile the plan flagged (R5): the
IR verifies either way; only **run-corpus** distinguishes them. It stayed green at 205/205, the
authoritative proof.

A minimal AOT bridge (`/ALTERNATENAME:newbf_alloc=malloc` / `=free` in `aot.rs`) keeps the hello_world
AOT link green during the MS-T2→MS-T3 window — a link-time *fallback* mirroring the guard's own
default-Thunk equivalence, ABI-safe on x64 (the extra `type_id`/`site_id` sit in RDX/R8, which `malloc`
ignores), automatically superseded when MS-T3 links the real runtime staticlib.

Gates: run-corpus 205/205 (authoritative — arrays+closures+strings+objects, Thunk mode), verify
154/154, parser 154/154, newbf-llvm 24/24 + newbf-runtime 26/26 — all green. The alloc path is now a
single seam the guard owns.

## 119. RF-T2 — the ClassVData $header ABI (the slot-shift, atomically) (ir+sema+llvm)

The highest-risk task in the wave, and the one cross-feature ordering constraint (it must precede
MS-T4). The object `$header` now ALWAYS points at a `%ClassVData = { i32 mType, [N×ptr] }` global —
retiring the bare `vtable_name` array and the empty-vimpls `Null` branch. That **shifts every vtable
slot by one word**, so this is a verify-clean miscompile hazard (R2): `corpus.rs` and the itable
invariant harness can't see a physical slot-shift — the IR verifies but would dispatch to the wrong
function. The defense is two-fold and both are green: a **NAMED run-corpus green list** (is_as→31,
virtual_poly→49, virtual_basic→42, abstract_method→12, base_call→30, all 13 `iface_*`) and a **non-JIT
emission unit test** asserting the `%ClassVData = {i32,[N×ptr]}` shape *and* that dispatch GEPs **field
1** (the only IR-level slot-shift detector).

Done atomically across three crates so no half-applied state can drift:
- **newbf-ir** — a new `VtableBase{hdr}` inst (struct-GEP `%ClassVData` field 1) + `load_type_id`
  builder. `VtableBase` lets sema express "the vtable lives at field 1" as a *struct-typed* GEP without
  newbf-sema naming `%ClassVData` (preserving sema ⊥ llvm) — the clean alternative to a hand-rolled
  byte offset, mirroring RF-T1's `LoadTypeId`.
- **newbf-sema** — register a ClassVData for **every `StructKind::Ref` id** (entries = `vimpls`, empty
  ⇒ `{i32,[0×ptr]}`), so every object has a non-null header; the `new`-site (and the two String
  allocation paths) always store `&classvdata_name(id)` (`"{prefix}$cvdata"`); `is`/`as` compare against
  it; **all three dispatch sites** (eager virtual, pending virtual, interface) route through a single
  `load_vtable_base(obj)` = `load $header → VtableBase` (field 1). Slot indices are unchanged — now
  correctly relative to the vtable array, not the header. (`load_type_id` added too, `#[allow]` until
  RF-T5.)
- **newbf-llvm** — `emit_vtables` → `emit_classvdata`: each `VtableDef` emits a `constant { i32, [N×ptr]
  }` (mType = `type_id`, **0 for all at RF-T2** — RF-T3 assigns real ids); a canonical
  `%ClassVData = {i32,[0×ptr]}` named type is the GEP base (field 1 sits at offset 8 for any N, so one
  GEP type reaches every class's vtable). `LoadTypeId` lowering implemented now (field 0 as i32) as the
  natural complement — unused until RF-T5, behavior-preserving.

Behavior is **identical** to before (type_ids all 0; dispatch results unchanged) — RF-T2 only reshapes
the header. Reflection behavior arrives in RF-T3/T4/T5. Gates: the named green list all pass, run-corpus
205/205, verify 154/154, parser 154/154, newbf-ir 9/9, newbf-sema 46/46, newbf-llvm 26/26 (+2 emission
tests), full suite green. MS-T4's delete path can now be written against the final header shape.

## 120. MS-T3a — the guard goes live in the JIT; UAF faults are observable (guard_corpus)

The marquee milestone: the deterministic memory guard is now **real and observable** in the JIT host.
This is the JIT half of MS-T3 (the AOT staticlib link is MS-T3b).

- **The guard lifecycle is re-exported from newbf-llvm** (`set_guard_mode`/`guard_reset`/`live_count`/
  `report_leaks`/`install_crash_handler`/`GuardMode`) — so the run-corpus harness and the child runner
  reach it through the crate they already depend on (no new external Cargo dep; the seam is MS-T0's
  absolute symbols). The driver sets `set_guard_mode(Stomp)` at startup under `#[cfg(debug_assertions)]`.
- **The full value run-corpus now runs under Stomp** — `install_crash_handler()` + `set_guard_mode(Stomp)`
  once, `guard_reset()` between programs, leak report suppressed (it's a value gate, not a leak gate, R4).
  The sequencing risk (Stomp exposing pre-existing scope-cleanup double-frees that MS-T4 fixes) **did
  not materialize**: all **205/205** pass under the real quarantining stomp allocator + ledger —
  objects, arrays, closures, strings all routed and value-correct. So MS-T2's alloc rename + the −8
  decision are validated against the live guard, not just CRT free.
- **The `guard_corpus` child-process harness** (R4 — a fault/abort kills the process, so the in-process
  value harness can't observe it): a `guard_runner` binary JIT-runs one `.bf` in Stomp mode; the parent
  `guard_corpus` test spawns it and classifies the child's Windows exit code. Three programs prove the
  three behaviors, with empirically-observed codes:
  - `uaf_after_delete.bf` → **`0xC0000005` ACCESS_VIOLATION** (read a field after `delete`; the page is
    decommitted + permanently quarantined → hardware fault).
  - `double_free.bf` → **`0xC0000409`** (Rust's `abort()` fail-fasts via `__fastfail` on Win64) — the
    ledger tombstone is hit; stderr `newbf guard: double free of <ptr> -> abort`. Ledger-first: the
    freed page is never dereferenced.
  - `no_leak_balanced.bf` → **clean exit 0**, `live_count()==0` (the no-false-leak side).

This **closes the JIT side of the MS first slice**: a Beef program that uses-after-free faults
deterministically, a double-free aborts with a named verdict, and a balanced program leaves the ledger
at zero — the signature Beef debug experience, working. Gates: run-corpus 205/205 (Stomp), guard_corpus
3/3, verify 154/154, parser 154/154, newbf-runtime 26/26, newbf-llvm 26/26, full suite green. Forward
notes: MS-T4 should add the conditional-scope/break-continue/early-return programs and re-run them under
the guard (those edges aren't exercised by the current corpus); MS-T3b links the runtime staticlib into
AOT (`/ENTRY:newbf_entry`) and adds the debug+release AOT parity tests.

## 121. RF-T3 + RF-T4 — reflection becomes observable: typeof(T) is real (sema+llvm+corlib)

Two tasks, one milestone. **RF-T3** (sema) computed each type's `ReflectPolicy` from its attributes

- **newbf-llvm `emit_metadata`** — from `module.type_meta`: a `%struct.Type` constant global per type

## 122. CB-T3 + CB-T4 — comptime emission that feeds back into resolution (newbf-comptime)

The comptime marquee: a `[Comptime, EmitGenerator]` method emits Beef source text that is spliced into
a type and re-resolved, so the emitted member is callable. **CB-T3** (sema) recorded an `EmitJob{
owner_qual_name, symbol}` per generator and rewrote its `Compiler.EmitTypeBody("…")` body to
`__newbf_ct_emit(<owner StructId.0>, text.Ptr, text.Len)` (the shim signature CB-T2 defined), keeping
the corpus inert (no program uses `[EmitGenerator]`, so `emit_jobs` stays empty → the no-op fast path).
**CB-T4** made it real — the fixpoint loop in `emit.rs` (the signature moved to source-in, adding the
legal `newbf-comptime → newbf-sema` edge):

1. analyze + `lower_program` the current sources; **empty `emit_jobs` → strip + return verbatim** (every
   corpus program — the fast path keeps the corpus untouched).
2. Build a per-round `StructId.0 → owner_qual_name` map (ids shift each round), JIT a sandbox clone with
   `__newbf_ct_emit` bound via MS-T0's `add_absolute_symbol`, run a `$ct_emit_run` wrapper calling every
   generator → `drain_emit_sink()` yields `(owner_id, text)`.
3. Resolve `owner_id → owner_qual_name`, **normalize + dedup** against a `seen` set (the termination
   guarantee — idempotent emission can't loop; a `MAX_EMIT_ROUNDS` backstop too), splice each new
   emission as `extension <Owner> { <text> }` (CB-T0's append-not-replace composition), re-lower. No new
   emission → fixpoint.
4. **Strip before return** (R6, load-bearing): reverse-reachability removes every `[EmitGenerator]`
   generator *and* the `__newbf_ct_emit` extern from the final module — the app JIT/AOT never registers
   that shim, so a survivor would fail to link. The emitted members stay.

The greens: **`comptime_emit_member → 42`** (a generator emits `Sum() => this.mX + this.mY` reading
*pre-existing* fields; `Main` calls it — emission genuinely feeds back into resolution),
`comptime_emit_then_call_twice → 42` (idempotent), `comptime_emit_dead_member → 7` (an uncalled emitted
member still links clean — the strip works). A dump-ir golden confirms the generated symbol is present
while the generator + `__newbf_ct_emit` are absent; the final module **JIT-links AND AOT-links** clean
(verified through a real driver AOT compile → exit 42). Gates: run-corpus **213/213** (+3), verify
154/154, parser 154/154, newbf-comptime 21/21 + comptime integration 7/7, guard_corpus 3/3 (stable),
full suite green; independently re-verified. CB-T5 hardens the caps (byte cap + diagnostics-not-`Err`);
CB-T6 brings widened-int const-eval breadth.

## 123. MX-T3 — mixins splice for real; all four wave-2 features now have marquee behavior

The mixins marquee: `expand_mixin` splices a mixin body at its call site by **reusing the live
`Lowerer`** — so a `return`/`break`/`continue` in the body targets the caller and SSA values dominate,
both *for free* (the design's core bet). Built on MX-T2's registry + MX-T2.5's strict gate:

- A `MixinFrame`/`mixin_stack` on the Lowerer (reset per method); `expand_mixin` runs
  `mixin_expandable`, and on `Err(FreeNameInBody)` **re-admits only when every body free name resolves
  in the live caller scope** (a caller local/param) — the widening MX-T2.5 anticipated. An owner field /
  static bare name doesn't resolve (bare `Ident` ≠ implicit `this.field`), so those stay declined.
- **Param-bind-once**: each arg is lowered exactly once in caller context, stored to an alloca, and

## 124. MS-T4 — scope cleanup correct on every exit edge (the riskiest behavior fix)

`scope`-lifetime cleanup is now correct on fallthrough / return / break / continue — each
`scope`-allocated object's dtor+free fires **exactly once**. This was the wave's riskiest
behavior-changing fix, and MS-T3a made it *directly* testable: the value run-corpus runs under the Stomp
guard, so a cleanup double-free **aborts** the harness and a use-after-free **faults** it — the
exactly-once property is enforced by live hardware, not just inspection.

- **The R9 trap and its fix.** A `scope` alloc inside an `if` branch doesn't dominate the block exit, so
  freeing the `new` SSA value at exit is "instruction does not dominate all uses." Fix: a new
  newbf-ir entry-block-alloca API (`entry_null_slot` — alloca + `store null`, both emitted into block 0
  so the slot dominates every path); the taken branch does `store ptr → slot`; each exit does `load
  slot; if != null { emit_destroy }`. Only the slot pointer and the loaded-ptr-or-null cross blocks —
  the `new` value never does. Verify stays 159/159 (verify *is* the R9 detector).
- **Dominating vs non-dominating, each in exactly one mechanism.** A `ScopeAlloc::Direct{val}` (alloc at
  the frame's entry block — always executes) frees the value directly; a `ScopeAlloc::Slot{slot}`
  (alloc in a nested/`if` block) uses the null-guarded slot. Each alloc registers exactly one entry — the
  value-list XOR the slot — so nothing frees twice. (The Stomp guard is the disjointness proof: a
  double-track would abort run-corpus.)
- **break/continue** record the scope-frame depth at loop entry (`loops` carries it) and run a
  depth-bounded `free_scopes_down_to(loop_depth)` before branching — freeing exactly the frames being
  exited, once; the `terminated` flag then suppresses the enclosing fall-through cleanup.
- **delete de-registration**: a `scope`-bound local tagged with its storage slot; `delete x` removes a
  `Direct` entry / null-stores a `Slot` so the scope exit doesn't free it again (no double-free).
- **Interface-typed `delete`** takes the bare `newbf_free` branch (an interface ref's concrete dtor
  chain isn't statically known); `emit_destroy` now `debug_assert!`s a concrete (non-interface) class id.

Gates: run-corpus **226/226 under the Stomp guard** (the authoritative exactly-once detector — +6 value
programs: fallthrough/return/break(3)/continue(4)/in-if/delete-dereg, each proving "fired" by count and
"once" by no abort), verify+parser **159/159** (+5 pins incl. the bare-if-body `Scope*` files that
genuinely hit the non-dominating Slot path, + `InterfaceDelete`), guard_corpus 3/3, full suite green;
independently re-verified. Documented conservative edges (out of scope): a conditional `delete` of a
`Direct`-tracked binding leaks the not-deleted path rather than double-freeing; defer-on-break/continue
is a pre-existing gap. The MS first slice's behavior correctness is now complete; MS-T5/T5.5/T6 add the
compile-time delete-flow diagnostics, MS-T3b the AOT guard link.

## 125. MS-T5 + MS-T5.5 + MS-T6 — compile-time delete-flow (Track B complete)
