# 2026-05-31 ‚Äî Control flow, the no-GC data-type model, and value structs

A long session spanning three arcs: finishing the control-flow surface, a
research detour into how Beef's core data types and optional GC actually work
(which set our memory-management direction), and then the value-struct layout

## 1. Finishing control flow (calls ‚Üí loops ‚Üí switch)

### Intra-program call resolution: the two guards (commit 3e04c58)

**Why:** bare `Foo(args)` calls were lowering to a defaulted external `i64`
call. To get recursion/mutual-recursion, a call must resolve to the sibling
method's real signature. So I pre-collect a per-type `MethodSig` table *before*
lowering any body (forward refs + recursion resolve for free).

**The challenge:** the naive version regressed the LLVM verify gate from 152/152
to **127/152**. Two distinct causes, two guards:

1. **Arity gate.** Beef has overloads; my table keyed by name (first wins). A
   call whose arg count didn't match the resolved method's arity emitted a call
   with the wrong number of arguments ‚Üí verifier error. Fix: resolve *only* when
   `arg_count == sig.params.len()`. Coercion then makes every arg exactly the
   declared param type, so the call always matches the signature; an arity
   mismatch falls back to the external varargs call. Got us to 146/152.
2. **Body-less skip.** The remaining 6: I was indexing *every* same-named method
   including body-less ones (extern/abstract). A `Foo(int,int)` extern could win
   the name over the body-having `Foo(int)`, so a 2-arg call resolved to a symbol
   that was never emitted (or had a different arity). Fix: only index methods
   that actually get lowered (skip `MethodBody::None`). Back to 152/152.

**Insight worth keeping:** when lowering with *partial* information (no real
overload resolution yet), the safe move is to resolve only when the resolution
is unambiguous *and* provably type-correct (exact arity ‚Üí coercion guarantees a
match), and otherwise degrade. "Index only what you emit" is the corollary ‚Äî a
resolution table must mirror what's actually in the module.

### The loop family + ++/-- (commit 97b18ac)

Added `for`, do-while, `break`, `continue` on a shared **loop-target stack**:
`Vec<(continue_target, break_target)>`, innermost last. `for`'s `continue` must
run the update before re-testing, so it targets a dedicated `for.cont` block,
not the head.

`++`/`--` initially lowered to `undef` ‚Äî which meant a `for` update of `i++`
silently did nothing and would spin forever. Realisation: inc/dec are
*lvalue* operations (load-modify-store against a slot), not value ops. Wrote an
`incdec` helper; postfix returns the old value, prefix the new. The corpus test
`incr_decr ‚Üí 577` exists specifically to pin that old-vs-new distinction.

### switch (commit 25c0d17)

Lowered to a cmp/cond-br chain (Beef arms don't fall through ‚Üí each branches to
one exit). The subtlety: inside a switch arm, `break` exits the *switch* but
`continue` targets the *enclosing loop*. Handled by pushing
`(enclosing_continue, switch_exit)` onto the loop stack for the arm bodies ‚Äî so
the two keywords correctly diverge.

## 2. Why we stopped to research the data types

The user asked the right question: **with no GC, how are strings and arrays even
supposed to work?** Rather than guess, I fanned out agents over the actual Beef
corlib + compiler source at `E:\beef` and the official docs.

**The finding that reframed everything:** `String`/`T[]`/`List<T>`/`Span<T>` are
**not compiler built-ins** ‚Äî they're ordinary stdlib classes/structs resting on
**four compiler primitives**: the object header, `new`/`delete`, `scope`, and
**append allocation** (`[AllowAppend]`). This cleanly splits the work into
*compiler/runtime* (our Rust) vs *stdlib* (Beef we compile). No GC anywhere ‚Äî
ownership is deterministic. Captured in `docs/CORETYPES.md`.

The genuinely clever bit is `String`'s small-string optimisation: one field
`mPtrOrBuffer` is *either* an inline char buffer (chars live at the field's own
address) *or* a real heap pointer, selected by a flag bit. And append allocation
(object header + variable payload in one allocation, via a synthesized
`__CalcAppend` size function + an `__appendIdx` cursor) is the hard primitive ‚Äî
it's what makes `String` SSO and inline-element `T[]` possible.

## 3. The GC "escape hatch" ‚Äî and the direction it set

The user was intrigued by Beef's optional-collector hook, having been burned
before by "declaring precise root safepoints all over the code." Researched
Beef's actual GC, then how our own `NewGC` and its two existing clients (NOD,
NCL) integrate. Captured in `docs/GC.md`.

**The asymmetry that dissolves the safepoint pain:** people conflate "precise
GC" with "safepoints everywhere." They're separable. Safepoints buy precise
*roots* (what's live in registers/stack at a suspension point). Precise *heap*
tracing (which fields of an object are pointers) is **static type info ‚Äî free,
no safepoints**. Beef exploits exactly this: conservative *roots* (scan the
stack/registers word-by-word, validate before trusting), precise *heap* (a
compiler-emitted per-type `GCMarkMembers`). A grep of the entire Beef compiler
for `gcroot`/`statepoint`/`stackmap` returns **zero** hits.

The price of conservative roots is you can't *move* objects (you can't safely
rewrite an ambiguous root) ‚Äî hence Beef is non-moving.

**Reconciling our NCL scar** ("conservative scanning fails non-deterministically;
precise roots are the answer"): that was right *for a moving collector*. The
fork is upstream of roots ‚Äî *moving ‚áí precise roots ‚áí safepoints; non-moving (or
pin-only) ‚áí conservative roots suffice.* The safepoint nightmare is a consequence
of choosing to move, not a tax on having a GC.

**Direction decided (recorded in GC.md):** NewBF stays no-GC by default; the
*optional* GC plugs into **NewGC's `conservative-pin` mode** ‚Äî conservative
roots + precise heap + mostly-copying ‚Äî which is its *tested default*, built for
"JITs without statepoint stack maps." This **deletes NOD's entire precise-root
apparatus** (liveness fixpoint, spill slab, per-site safepoint tables, reload
plumbing, loop-header polls). NewBF would be NewGC's first pure-conservative
client. The gating constraint: NewGC's pinner rejects interior pointers, and Beef
code holds them freely (`&array[i]`, `Span.mPtr`) ‚Äî but NewGC already has the
`object_start_at_or_before` function to fix this, just unwired. NCL's deepest
lesson, banked: the *manual* root contract rots (one missed `push_root` =
use-after-GC far from the cause); conservative roots remove that whole class.

## 4. The value-struct layout sprint (commit 81dd1fe)

The IR was scalar-only by design ("aggregate types arrive with the layout
sprint"). This was that sprint: `IrType::Struct` + a field-address instruction
(IR), LLVM struct lowering, and nominal-type tracking + member access (sema).

### Decisions

- **Value structs first**, deferring the heap/`new`/memory-model fork. It's the
  smallest self-contained slice that builds the layout machinery everything else
  needs, and it's fully JIT-testable without an allocator.
- **`StructId` is a `u32` handle**, not an inline layout, so `IrType` stays
  `Copy`. The field list lives in a `Module`-level registry. This kept the
  pervasive `IrType: Copy` assumption intact across the codebase.
- **`ObjectLayout` is one contiguous pointer range, not a bitmap** (this is
  NewGC's model too). For value structs the field list is just ordered fields;
  the constraint bites later when a `BeefLayout` must group reference fields ‚Äî
  and since *we* control layout, the answer is field reordering, not a bitmap.

### Challenges + solutions

- **Nested / forward / self struct references in LLVM.** A struct field whose
  type is another struct needs both LLVM types to exist. Solution: two passes ‚Äî
  create all *opaque* struct shells first, then set each body. Standard, but easy
  to get wrong if you build bodies eagerly.
- **The keystone risk: a global verify regression.** Registering structs made
  `lower_ty` map *every* struct-typed reference in the whole corpus from `Ptr`
  to `Struct(id)` ‚Äî params, locals, returns, fields. That's a huge IR change
  across 152 modules, and ill-typed aggregate IR would fail the verifier. This
  was the moment that could cascade. It **held at 152/152** because: (a) `coerce`
  *always* yields a value of the target type (struct‚Üîscalar with no real cast
  degrades to a typed `undef`), and (b) the "skipped/unsupported ‚Üí typed undef"
  discipline keeps everything well-typed even when semantically lossy. The
  graceful-degrade design we'd been maintaining all along is what made a
  high-blast-radius change safe.
- **Distinguishing `struct` from `class`.** Critical: only value `struct`s
  become aggregates; a `class` must stay `Ptr` (reference), or class-typed
  locals/params across the corpus would flip type and likely break calls. Keyed
  on `TypeKind::Struct` + non-generic.
- **Borrow-checker choreography in `lvalue`.** Resolving `obj.field` reads the
  struct def (`&self.structs`) *and* emits a `field_addr` (`&mut self.fb`). Had
  to copy the field index + type out of the immutable borrow in an inner block
  before the mutable emit. Small thing, but the kind that stalls you if you
  fight it instead of restructuring.

### Insight

Refactoring `assign` and `incdec` onto a single `lvalue` resolver was the unlock:
once "what's a storable place?" lives in one function (local slot *or* chained
struct field via `fieldaddr`), `p.x = v`, `p.x += i`, `p.x++`, and nested
`o.inner.v` all fell out together. `lvalue` is recursive, so arbitrary chains
work. Proven by `struct_nested ‚Üí 15` (two-level GEP) and `struct_swap ‚Üí 21`.

The strategic bonus: the per-type field layout this sprint computes is *exactly*
the precise heap map a future `BeefLayout` hands NewGC. The layout sprint quietly
advanced the GC direction too.

## 5. The heap + object model sprint (commit 2c6708a)

Reference types (`class`), `new`/`delete`, and member access through references ‚Äî
the no-GC manual-memory core. Came together same-day, on top of the value-struct
machinery.

### The key decision: `IrType::Ref(StructId)`

The carrying-forward note (above, now realised) predicted the hard part: member
access on a class reference needs to recover *which class* a pointer points at,
but we'd kept class refs as opaque `Ptr`. The clean fix was a new IR type:
**`Ref(StructId)`** ‚Äî a pointer that *remembers its pointee's layout*. It lowers
to a plain LLVM `ptr` (so nothing changes at the LLVM level), but carries the
nominal class so the lowerer can `fieldaddr` through the heap body.

Why this was almost free on the verify gate: because `Ref` ‚Üí `ptr`, every
class-typed parameter/local/return that used to be `Ptr` is *still* `ptr` in
LLVM. Registering all the corpus's classes (previously `Ptr`, now `Ref(id)`)
produced **zero LLVM-signature change** ‚Äî verify stayed 152/152. The only new IR
is where member access on a reference now *resolves* (previously degraded to
`undef`). Same trick as the value-struct sprint: a change with huge surface but
no type-level blast radius.

### The unification that paid off: `struct_base`

A value struct and a class reference differ in *where the body is*:
- value struct local `p`: its lvalue place **is** the struct body (an alloca).
- class ref local `obj`: its lvalue place holds a *pointer*; the body is one
  load away.

Rather than branch member access on this everywhere, one helper `struct_base`
resolves any base to `(body_pointer, struct_id)` ‚Äî returning the place directly
for `Struct`, loading it for `Ref`. Because it's built on the recursive
`lvalue`, **chained access through reference fields just works**: `a.next.val`
loads `a`'s ref ‚Üí GEPs `next` (a `Ref` field) ‚Üí loads that ref ‚Üí GEPs `val`.
Proven by `heap_list ‚Üí 42` (a self-referential `Node { int val; Node next; }`).
This is the second time collapsing a distinction into one resolver
(`lvalue` for places, now `struct_base` for member bases) unlocked several
features at once. Worth internalising as a pattern.

### Object model + what's deferred

- A class's instance body is a registered struct with a synthetic `$header`
  (`ClassVData*`) field at index 0; real fields follow. User field lookups skip
  the header naturally (name mismatch), so member indices come out right.
- `new C()` = `malloc(SizeOf body)` + zero the header, yielding `Ref(C)`. Added
  `InstKind::SizeOf` so the *backend* computes the ABI size
  (`StructType::size_of`) ‚Äî the IR must not guess struct layout/padding.
- **Deferred on purpose:** the constructor call (`new` just allocates + zeroes
  the header for now) and the destructor (`delete` just `free`s). Both need the
  method-call + ctor-resolution machinery; fields are set explicitly in the
  corpus tests meanwhile. Also deferred: inheritance (a class body has only its
  *own* fields, so base-class field access degrades).

### Why the header is `Ptr`, and the GC tie-in

The header is the `ClassVData*` slot from `CORETYPES.md`. It's null for now
(no vtables/type-data yet). When we add the optional GC, this is also where a
type-id / mark word would live, and the per-class field layout we now compute
*is* the `BeefLayout::header_layout` NewGC consumes (`GC.md`). The object model
and the GC direction keep converging.

## 6. Constructors + destructors (commit e6b17fb)

Objects initialize themselves now: `new C(args)` runs the ctor, `delete x` runs
the dtor before freeing.

### The scoping insight that made it small

I'd flagged this as needing "instance-method calls (`this` + dispatch)" ‚Äî which
sounded like the big ABI-changing refactor (every instance method gains a
`this`; `obj.Method()` needs cross-type dispatch). But **a constructor doesn't
need dispatch**: `new C(args)` names the exact type `C`, so it calls `C`'s ctor
*directly* by mangled name. That collapsed the sprint to **sema-only, no
ABI-wide change**: a ctor is just a function with a leading `this` param, `this`
is a param, and `new`/`delete` already existed. Regular instance methods were
left untouched (`this_ty = None`) ‚Äî I explicitly did *not* add `this` to all of
them, dodging the risky cross-corpus signature change. Lesson: re-examine
whether the "obvious prerequisite" is actually required for the slice in hand.

### `this.field` came for free

The only "member access" a ctor body needs is `this.field`. And `this` is just a
`Ref` param ‚Üí so `struct_base` (built last sprint for `obj.field` on references)
already handles it: `lvalue(This)` returns the `this` slot, `struct_base` sees
`Ref(id)`, loads it to the body, GEPs the field. **Zero new member-access code.**
Third sprint running where the `lvalue`/`struct_base` resolvers paid forward ‚Äî
the pattern of routing everything through one place-resolver keeps compounding.

### The one fiddly detail: symbol names must line up

`new` resolves the ctor by a *name* recorded in the pre-pass (`{prefix}$ctor`),
but the ctor *function* is emitted later by `lower_type` under its own prefix.
Those two prefixes have to match exactly, so I threaded the same prefix-building
(`format!("{prefix}{name}.")`, types-only, namespaces pass through) into the
pre-pass and stored it per id. A mismatch would be a silent "ctor never runs"
(the call would target a nonexistent symbol ‚Üí external varargs degrade). Worth a
test if this grows.

### Deferred (and why it's fine)

Ctor `: this(...)/base(...)` chaining, implicit-`this` bare field names
(`v = 5` meaning `this.v = 5`), `obj.Method()` dispatch, and inheritance. Simple
ctors that set/compute fields from args cover the common case and unblock the
core types; the rest are their own sprints. verify held at 152/152 (ctors/dtors
are well-typed `void` functions; `Ref this` lowers to `ptr`).

## 7. Instance methods + `obj.Method()` dispatch (commit eb7467e)

The "bigger refactor" I'd twice deferred ‚Äî and it came in cleaner than feared.

### What made it tractable

The pre-pass already walked every type collecting ctor/dtor signatures; adding a
**global per-type method table** (name ‚Üí this-aware signature) was a few more
lines in the same loop. That replaced the old *per-type-local* `sigs` (built
inline in `lower_type` for sibling calls only) with one table usable both for
same-type bare calls *and* cross-type `obj.Method()`. The dispatch itself is by
static type ‚Äî `obj`'s `Ref(id)`/`Struct(id)` gives the owning type ‚Äî so no
vtable yet, no runtime dispatch.

### The arity gate paid a dividend I didn't plan

Worry: now that instance methods carry a leading `this`, would a bare `Foo()`
call (no receiver) wrongly resolve to an instance method and call it without a
`this`? No ‚Äî and for free. Instance sigs now have `params = [this, ‚Ä¶explicit]`,
so a bare call with only explicit args **arity-mismatches** and falls back to
the external path. The arity gate built way back in the call-resolution sprint
(08d) already does exactly the right thing. Bare calls resolve only to static
siblings; instance calls must go through a receiver. Zero special-casing.

### `struct_base` paid forward a *fourth* time

`obj.Method()` and `this.Method()` both resolve the receiver through the same
`struct_base` ‚Äî value struct (place is the body) or class ref (load to the
body) ‚Äî that powered `obj.field`, `this.field`, and chained access. One
receiver-resolver, every member/method form. The "route everything through one
resolver" discipline has now paid off in four consecutive sprints; it's the
single highest-leverage shape decision of the object-model work.

Verify held at 152/152 again ‚Äî same reason every object-model sprint did:
`Ref` lowers to `ptr`, so adding `this` changed no LLVM signature.

Deferred: overload-by-type (first-name-wins today), inheritance + virtual
dispatch, and implicit-`this` bare field/method names (`v` for `this.v`).

## 8. Pointer indexing ‚Äî `p[i]` (commit fa1126a)

The journaled gate to `String`. The wrinkle the carrying-forward flagged turned
out to be the whole sprint: indexing needs the **element type** behind a
pointer, but our pointers are opaque (`Ptr`) ‚Äî `char8*` and `int32*` both lower
to the same `ptr`, yet they index at different widths/strides.

### The decision: track element types in sema, not in `IrType`

The tempting move ‚Äî a `TypedPtr(elem)` IR type ‚Äî would ripple through every
exhaustive `match IrType` in three crates (`bit_width`, `mnemonic`, `coerce`,
`zero_of`, LLVM `basic_type_of`/`undef_of`, ‚Ä¶) and `IrType` is `Copy`, so the
element couldn't even be a `Box`. Instead I attached the element type as
**binding metadata in the lowerer**: the scope entry became
`(slot, ty, Option<elem>)`, set from a pointer local/param's `AstType::Pointer`
inner. **Zero IR/LLVM type-system change** ‚Äî the only IR addition is one
instruction (`ElemAddr`, a base+index GEP). The element type lives exactly where
it's introduced and is read back at the index site. Much smaller blast radius
than a new pointer type.

### `lvalue` paid off a fifth time

`Expr::Index` lowered as one `lvalue` arm ‚Äî compute `elem_addr(base, elem, idx)`,
return `(addr, elem)`. Reads `load` it; assignment and `++`/`--` already route
through `lvalue`, so `p[i]`, `p[i] = x`, `p[i] += d`, `p[i]++` *all* fell out of
that single arm. Five sprints now where the place-resolver discipline turned "add
a feature" into "add one match arm."

### Wrinkles

- **Stride correctness is the point.** `index_bytes` (char8* ‚Üí byte stride ‚Üí
  131) and `index_ints` (int32* ‚Üí 4-byte stride, variable index in loops ‚Üí 60)
  exist specifically to prove the GEP scales by `sizeof(elem)`, not a fixed 1.
- **Testing needed a real pointer.** No arrays and no `&addr-of` yet, so the
  corpus programs `malloc` a buffer (CRT `malloc`/`free` resolve through the
  JIT's process-symbol generator) and `free` it. Convenient, and it exercises
  the FFI path too.
- **Limitation:** v1 resolves the element type for pointer *locals/params* only.
  Indexing a pointer *field* (`this.buf[i]`) needs field element types ‚Äî and
  that's exactly the `String` shape, so it's the immediate follow-up.

verify-corpus held at 152/152 (`ElemAddr` is a standard GEP; the binding change
is sema-internal).

## 9. Field-pointer indexing ‚Äî `obj.buf[i]` (commit 54a0ef9)

The one gap the last sprint flagged, closed ‚Äî and it brought the
**String-buffer pattern** to life.

### Same shape as before, one level deeper

Locals/params tracked their pointer element type as binding metadata; fields now
do too, via a `field_elems` side-table parallel to `defs[id].fields`. Keeping it
*beside* the IR `StructDef` (rather than adding `elem` to the IR `FieldDef`)
matters: the backend GEPs fields by index and never needs element types ‚Äî that's
purely front-end metadata, so it stays in sema. Same instinct that kept typed
pointers out of `IrType` last sprint.

### The new primitive worth naming: `expr_struct_id`

To resolve `obj.buf`'s field, I need *which struct* `obj` is ‚Äî by **static
type, with no code emitted**. That's distinct from `struct_base`, which emits a
load to get the runtime body pointer. `expr_struct_id` walks `this` / locals /
nested fields purely at the type level. Having both ‚Äî one that answers "what
type?" without side effects, one that produces "where's the body?" with them ‚Äî
is a clean split I'll lean on again (it's the start of a real
expression-typer).

### The milestone: a hand-written `String` core runs

`string_buffer.bf` is the whole point of the last few sprints in one program: a
class owning a `char8*` buffer, allocated in its constructor, indexed through
`this.data[i]` in `Set`/`Get` instance methods, and freed in its destructor ‚Äî
`delete b` runs `~this` then `free`. Classes + ctors + dtors + instance methods +
field-pointer indexing + FFI, all composing, JIT-run to **131**. That's the
manual-memory OO pattern real `String`/`List` are built from. `lvalue` carried
indexed field read/write/`+=`/`++` from one arm ‚Äî its sixth sprint paying off.
verify-corpus held 152/152 (sema-only).

## 11. A real growable String + constructor overloading (commit b9b9724)

Depth, chosen over breadth. Two findings.

**The growable String just worked ‚Äî no compiler change.** `string_grow.bf`
builds a `Str` from a literal (`new Str("Hi")`), measures its own length and
copies it with index loops, and *grows* its heap buffer when `Append` outgrows
capacity. I'd expected to need a `memcpy`/`strlen` binding (the journal said so);
instead, doing strlen and the copy as plain `while`/`for` loops over
`s[i]`/`this.buf[i]`/`nb[i]` needs nothing new ‚Äî param-, field-, and
local-pointer indexing all already work, and `new Str("‚Ä¶")` passes the literal
`char8*` straight to a `this(char8*)` ctor. The "I expected to need X" that turns
out unnecessary is itself the signal: the object-model foundation is solid.

**Writing it surfaced one real gap ‚Äî single constructors.** A class recorded
only its *first* ctor, so `this()` and `this(char8* s)` couldn't coexist
(`new Str("x")` would arity-mismatch the recorded `this()` and silently run no
ctor ‚Üí garbage buffer). Fixed: **constructor overloading by arity** ‚Äî `ctors`
became a `Vec` per type (one sig per distinct arity, mangled `$ctorN`), and
`new` picks the overload matching the argument count (`ctor_for`). `ctor_overload`
proves it: the same `Str` built via both `new Str()` and `new Str("Yo")`.

This is the test-driven-discovery loop working as intended: write real
corlib-shaped Beef, let it run as far as it can, fix exactly what it hits. (Still
arity-only ‚Äî same-arity different-type overloads don't disambiguate; that's true
of methods too.)

## 12. Stdlib organization + prelude plumbing (STDLIB.md, commit b99a538)

A deliberate breather, then the first real stdlib step.

**The design (`docs/STDLIB.md`).** Decided the stdlib is *source*, not a binary:
corlib `.bf` is prepended to the user's program and lowered as one module
(compose-at-the-AST, the pinned principle; JIT-first whole-program lowering).
Distinguished the two corlibs (the upstream `corlib-slice` we port *from* vs.
`newbf-corlib/bf` we *write*), the bottom-up layering (`Internal` ‚Üí `Object` ‚Üí
`String` ‚Üí collections ‚Üí IO ‚Üí `GC` stub), and ‚Äî the doc's most useful artifact ‚Äî
a **gating-features table** showing the feature roadmap *is* the corlib roadmap
(qualified static calls + extern binding ‚Üí `Internal`; literals-as-`String` ‚Üí
real `String`; generics ‚Üí collections).

**The plumbing (sprint 17).** `newbf-corlib::prelude()` embeds each `bf/*.bf`
via `include_str!`; `lower_program` parses them into owned units and prepends
them as `SourceFile`s before the user's, so `StructTable` registration and
lowering see corlib + user as one module. Centralising it in `lower_program`
means *every* consumer (driver, corpus, tests) gets the prelude with no caller
change ‚Äî and corlib is now lowered into every program, so the verify gate holds
it continuously (a free forcing function as it grows). Proven by `prelude_probe`
‚Üí 42: a `Probe` class defined *only* in `newbf-corlib/bf/Probe.bf`, used via
`new Probe()` from a program that never declares it.

The lifetime shape is the one wrinkle: the prelude units are owned locally in
`lower_program` and the combined file list borrows them, which is fine because
lowering returns an owned `Module` (no borrow escapes).

## 13. Qualified static calls + a corlib `Math` (commit 211e7a0)

## 15. Real `System.String` + the cast-skew bug (commits 421b2e0, ed13cbf)

The headline landed ‚Äî but not before a genuine debugging detour.

**The feature.** The growable-string prototype is now corlib's actual `String`
class: `char8*` buffer + length + capacity, two ctors (`this()` and
`this(char8* s)`), a dtor, and `Length`/`CharAt`/`Append`/`Grow`, built entirely
on the `Internal` FFI floor (`Malloc`/`Free`/`MemCpy`). And a string literal in a
`String` context now *constructs* one: `coerce` wraps a `char8*` (a literal,
`IrType::Ptr`) into `new String(..)` when the target is the `String` class.
Because it lives in `coerce`, target-typed wrapping falls out for free at every
site ‚Äî `String s = "hi"`, a literal passed to a `String` param, a returned
literal ‚Äî with no expected-type threaded through `expr()`. `string_length ‚Üí 5`,
`string_ops ‚Üí 6533` (construct-from-literal, `Append`, `CharAt` loop).

**The bug.** Adding `String` to the prelude crashed `ctor_overload` ‚Äî a program
that never references `String` ‚Äî with `0xC0000005`. The bisect was telling: a
*trivial* `String` (one int field) passed; the full one crashed; and it needed
*both* `String` and a user class doing bare `malloc` + `char8*` indexing in the
same module. `cdb` (the Store WinDbg's `cdbX64.exe`) gave ground truth:
`mov byte ptr [rdx+rcx], al` with `rdx = baadf00dbaadf00d` ‚Äî `this.buf` was
uninitialized heap. The IR showed why: `this.buf = malloc(this.cap)` lowered to
`store ptr undef`. The `malloc` *call* ran; its result was thrown away.

**Root cause ‚Äî extern signature skew.** An external symbol's real LLVM signature
is fixed by its *first* declaration in the module. A bare `malloc(...)` is typed
`i64` by sema (the unresolved-extern default), but corlib's `Internal.Malloc`
(`void*`) now declares `@malloc` as returning `ptr` ‚Äî and it lowers first. So the
call site yields a `ptr` while sema still believes it's an `i64` and emits the
`inttoptr` that `char8* = <i64>` needs. `as_int` got a pointer, returned
`i64 undef`, and `inttoptr undef` folded to `ptr undef`. Before corlib, bare
`malloc` was `@malloc`'s only declarer (`i64`), so the `inttoptr` was valid ‚Äî
which is exactly why a brand-new, *uncalled* corlib type could break an old
program: it changed a *shared* extern's declared signature.

**The fix** (421b2e0): treat `IntToPtr`/`PtrToInt` as the identity when the
operand is already in the target representation. The IR‚ÜîLLVM boundary is where
advisory IR types meet real signatures, so the cast layer is the right place to
absorb the skew. (Alban predicted "alignment errors" ‚Äî wrong on the specific
mechanism, right in spirit: a codegen-dependent fault that verifies clean and
only bites at runtime.)

Two lessons banked: extern call result types are advisory, reconciled by the
first declaration; and a feature that's never *called* can still change a
module's machine code by changing a symbol everyone shares.

## 17. Overload resolution by argument type ‚Äî and the skew it dredged up (commits 0375396, 9657d73)

Depth on strings runs straight into a compiler feature: a real `String` API is
all *same-arity* overloads ‚Äî `Append(char8)` **and** `Append(String)`, later
`==`/`Concat` ‚Äî but resolution was arity-only (first-wins). So the headline is
**overload resolution by argument type**.

**The feature.** Method tables became `HashMap<String, Vec<MethodSig>>` (every
overload kept). `pick_overload` scores arity-matching candidates by exact
type-match count, gated so a `this`-less site (a bare or `Type.M` call) never
selects an instance method. The subtle part is *mangling*: two `Append`s mangled
to one symbol `@String.Append`, so only one body survived and a `ptr`-arg call
hit an `i8`-param function (10 verify failures, plus `append_overload ‚Üí 4388`).
Fix: a later overload of a name gets a param-type suffix (`‚Ä¶$R3`/`‚Ä¶$i8`); the
first keeps the plain symbol; and `lower_type` now *reuses* the table's
`full_name` by matching params, so definition and call sites can't disagree.
`String` gained `Append(String)` (chosen over `Append(char8)` by type) and value
`Equals`. `append_overload ‚Üí 5427`, `string_equals ‚Üí 10`.

**The skew it dredged up.** First-wins had been silently *dropping* the corlib
`Math` fixture's float/double overloads ‚Äî so they never lowered. Registering them
all finally lowered them, and the verifier caught `fcmp float ‚Ä¶ , double ‚Ä¶`. The
root (found by reading the IR): `Abs(float)` and `Abs(double)` both bind the C
symbol `@abs`, whose LLVM signature is fixed by its first declaration ‚Äî so a call
sema-typed `f64` returned an `f32`, poisoning the downstream `fmul`/`fcmp`. This
is exactly the `malloc` int‚Üîptr skew, one tier up. I first patched the *compare*
(widen mixed floats), watched the next op fail, and recognized the whack-a-mole:
the cure belongs at the **call boundary**. `reconcile_to` coerces a call result
to the type the IR expects (fpext/fptrunc, int ext/trunc, int‚Üîptr) ‚Äî one place,
all downstream uses well-typed, and it *subsumes* the earlier int‚Üîptr cast fix.
Restored 152/152.

Two side-lessons: (1) a process-global like `SetStdHandle` can't have two
parallel `#[test]`s ‚Äî libtest threads them, so both stdout checks live in one
test. (2) String *literals* are `Ptr`-typed at resolution, so a literal passed
straight to an overloaded `Append` would prefer a `char8*`/exact param; pass a
`String` local (or construct explicitly) to hit the `String` overload ‚Äî true
literal-as-`String` overloading waits on target-typed argument resolution.

## 18. Tagged handles for Beef? ‚Äî studied, declined as GC, kept as a safety primitive (commits 426ba6a, f95f1cf)

Alban: *"take a look at the simplified tagged-handle approach in `E:\RNIM` ‚Äî would it fit Beef instead of fucking malloc?"* So I read Locus's `gc.md`.

**The scheme.** The program holds a **handle** ‚Äî an index into one table ‚Äî never a
raw pointer. The table *is* the GC root set (compiler GC-oblivious, like us), and
because a handle is an index, objects **move freely**: the collector rewrites
`table[h]`, the index is stable. Conservative scanning becomes *sound* (a stray
int that looks like a handle over-retains, never corrupts). Handles run on a stack
discipline (V8 `HandleScope` / JNI / OCaml `CAMLparam`), so a 600-node walk peaks
at a live-root depth of two. Genuinely elegant.

**The verdict (now `GC.md` ¬ß12): no as a `malloc` replacement, and the reason is
structural.** The whole payoff is *free relocation*, bought with one invariant ‚Äî
*no raw pointer to a heap object ever escapes*. Beef is the opposite: `char8* buf
= s.Ptr(); WriteFile(buf, ‚Ä¶); buf[i]`, raw `T*` in fields, interior pointers
(`&array[i]`), C layout for Win32. Beef would pin nearly everything ‚Üí pay the
indirection, get no compaction. The clincher: **real Beef's own collector is
conservative + non-moving for this exact reason** ‚Äî the Beef team hit the same
wall `GC.md` already chose to sit behind. The handle scheme exists to *enable
moving*; Beef can't move regardless; it solves a problem Beef isn't allowed to
have. Locus and NewBF are correctly using *different bindings of the same NewGC*.

**What we kept.** The handle *idea*, demoted from collector to optional manual
primitive: **`Pool`** ‚Äî a generational slotmap. A handle is a pointer-sized `int`
packing `(gen << 16) | idx`; `Get` resolves it live and returns **null** once
`Free` bumps the slot's generation. Use-after-free becomes a *detectable null*,
not a dangling pointer ‚Äî the safety win over raw `malloc`, with no GC and without
giving up raw pointers anywhere else. It's the slotmap/generational-index pattern
game engines (Beef's audience) already use. `pool_handle ‚Üí 142`: live `Get` reads
`b.v` (42), then `Free`, then the stale `Get` returns null (+100). Stored as
parallel `int` arrays over `Internal`; the object pointer round-trips through
`ptrtoint`/`inttoptr` at the store/return boundary (which the ¬ß15 reconcile work
makes routine). Non-generic for now ‚Äî reuse, growth, and a typed `Handle<T>`
wrapper wait on generics.

## 19. Generics, first slice ‚Äî monomorphized value structs (commit 6e8b102)

Generics is the gating dependency three threads were waiting on (`Handle<T>`,
`List<T>`, `String` SSO), so: take it on. The chosen mechanism is
**monomorphization** ‚Äî one concrete type per (generic, concrete-args) pair,
substituting the type parameters ‚Äî not type-erased generics. It rides the
existing concrete-type pipeline rather than adding a parallel one.

The shape: a pre-pass over the program (1) indexes generic declarations by name,
(2) walks type positions ‚Äî field/param/return types and local declarations ‚Äî
for instantiations, and (3) registers one monomorph per distinct pair, mangled
by the **overload type codes** from ¬ß17 (`Pair<int>` ‚Üí `Pair$i64`, `Pair<int32>`
‚Üí `Pair$i32`). `lower_ty` grew exactly two cases: a bare type-parameter resolves
through a per-monomorph **env** (`T` ‚Üí the concrete type it was instantiated at),
and a generic instantiation `Name<Args>` resolves to its registered monomorph.
Field-filling factored into one env-aware `fill_fields_at` shared by ordinary
types (`env = &[]`) and monomorphs ‚Äî so a monomorph is *just a concrete type with
T pre-substituted*, and everything downstream (layout, `sizeof`, field access)
is unchanged.

`pair_generic ‚Üí 42`: `Pair<int>` (i64 fields) and `Pair<int32>` (i32 fields) are
**distinct layouts from one template** ‚Äî the proof that the type parameter is
really substituted per instantiation, not erased. 52 programs, verify 152/152.

This slice is deliberately value-structs-only ‚Äî no `new`, no methods, no body
type-env. Those are the **generic class** slice: collecting instantiations from
expression position (`new Box<int>()`), lowering each monomorph's methods/ctors
with the env (threading it through `lower_type`/`lower_method`/`Lowerer`), and
resolving `new Box<int>` in `lower_new`. The infrastructure (env, mangle,
collection, registration) is now in place for it.

## 20. Generic classes ‚Äî `new`/methods/`this` per monomorph (commit 19cf654)

The value-struct slice built the scaffolding (env, mangling, collection,
registration); this slice makes generic **classes** real, which is where the
type parameter actually has to flow through *code*, not just layout.

`fill_members_at` became env-aware end to end: a monomorph's ctors, dtor, and
method table are registered with their parameter/return types substituted and
their symbols mangled (`Box$i64.$ctor1`, `Box$i64.Get`). `lower_type` split into
a wrapper that **skips generic templates** (a `Box<T>` is never lowered directly
‚Äî lowering it would emit a broken `Box.Get` with `T ‚Üí Ptr` and `this = None`) and
an env-threaded `lower_type_at` that `lower_program` drives once per recorded
monomorph (`StructTable.monos`) at the mono's id/prefix/env. The env rides down
through `lower_method` into the `Lowerer`, so a body's `T` locals resolve
concretely. `lower_new` learned the generic case (`generic_new_parts` +
`new_class_id`): `new Box<int>()` resolves to `Box$i64` and its ctor; instance
calls land on the monomorph's own method table.

`box_generic ‚Üí 42`: `Box<int>` (i64 payload) and `Box<int32>` (i32 payload) are
distinct heap types from one template ‚Äî `new`, ctor, and `Get()` each compiled
with `T` substituted, then `a.Get() + b.Get()`. **53 programs, verify 152/152.**

The design choice that kept this tractable: a monomorph is *just a concrete type
with `T` pre-substituted*, so it reuses the entire concrete-type pipeline
(layout, method table, dispatch, `new`/`delete`) unchanged ‚Äî the only new
machinery is the env (one lookup in `lower_ty`) and the collection pre-pass.
Monomorphization, not erasure: zero runtime cost, the right call for a
value-types-matter language.

One honest edge: instantiations are still collected from **type positions**
(field/param/return types, local declarations). An instantiation that appears
*only* in expression position ‚Äî `var a = new Box<int>(‚Ä¶)` with no typed local ‚Äî
isn't collected yet; the test uses `Box<int> a = ‚Ä¶`. Expr-position collection is
a small follow-up.

## 21. `Console.WriteLine(int)` ‚Äî the machine prints numbers (commit f61d017)

A small capstone on the overload + generics work: print *computed* values. A
typed `WriteLine(int)` overload sits beside `WriteLine(String)` and is chosen by
argument type; it renders decimal via a recursive `AppendDigits` (emit the

**`type_affinity`** tier: exact (2) > same category ‚Äî int‚Üîint of any width,

`int` parameter over a `String` one regardless of width. (Bonus: this also
sharpens the literal-vs-overload edge from ¬ß17 ‚Äî a `char8*` literal now prefers
a `String` parameter over a `char8` one.)

verify 152/152.** The machine that learned to say "Hello, world!" in ¬ß16 now does
arithmetic and tells you the answer.

## 22. `List<T>` ‚Äî a growable generic collection (commit c1c5cdc)

The headline generic collection, and the proof that monomorphized generics +
the `Internal` floor + typed-pointer indexing compose into something genuinely
useful. corlib `List<T>` holds a raw `T*` buffer with count/capacity, and
does `new`/`Add`/`Get`/`Count`/grow/`delete`.

The one compiler gap it exposed: **pointer-element types weren't env-aware**.
A `T*` field's element type drives the indexing stride (`ElemAddr`'s GEP), and
`pointer_elem` resolved `T` with no env ‚Üí `Ptr` (8 bytes) for every monomorph.
`List<int>` (i64, 8) survived by coincidence; `List<int32>` (i32, 4) would have
strided by 8 and read every other slot. Fix: `pointer_elem_env`, threaded
through field layout, method params, and local declarations ‚Äî so the element
type flows from the monomorph env into both field- and local-pointer indexing.
(The `lower_ty`/`pointer_elem` zero-env wrappers were now dead ‚Äî everything uses
the `_env` form ‚Äî so they're gone.)

Two pragmatic v1 choices, both noted in the source: the buffer over-allocates
8 bytes/slot (max scalar width) because there's no `sizeof(T)` operator yet, and
`Grow` copies **element-by-element through the typed pointer** rather than a byte
`memcpy` ‚Äî which sidesteps needing `sizeof(T)` for the copy entirely.

`list_generic ‚Üí 42`: `List<int>` grows past its initial capacity of 4 (one
`Grow`) and sums to 21; `List<int32>` is a distinct 4-byte-stride instantiation
summing to 21. **55 programs, verify 152/152.** A from-scratch compiler with a
working generic dynamic array, allocating through its own stdlib.

## 26. Inheritance ‚Äî field + method reuse, static dispatch (commit ec0b2ba)

"BF has some deep syntax" ‚Äî Alban's right, so: dive into the OO deep end. The
strategic frame from the survey held ‚Äî the breadth is *parsed* (the corpus gate
is 100% clean), so this was a **lowering** slice, sema-only, no IR/LLVM change.

`StructTable` gained a `bases` vector; `fill` resolves the first base that's a
class. A new `apply_inheritance` pass runs once every type's own layout is
filled and composes each derived type ‚Äî **recursive + memoized so a chain
(`C : B : A`) composes base-first**. The derived gets the base's fields right
after its own `$header`, the matching field-element types, and the base methods/
destructor it doesn't itself declare. Two facts make static dispatch fall out:

- **Prefix-compatible layout.** A derived's fields are `[$header, base fields‚Ä¶,
  own fields‚Ä¶]`, so a base method's field GEPs (typed at the base struct) hit
  the same byte offsets on a derived pointer. A base method reads inherited
  fields correctly with no adjustment.
- **Inherited sigs point at the base impl.** An inherited method in the
  derived's table keeps the base's symbol and `this` type; the call passes the
  derived `this` reinterpreted to the base ref (both are `ptr`).

So up-casting is a plain reinterpret, and inherited methods Just Work.
`inherit_basic ‚Üí 42` (Dog : Animal ‚Äî inherited field, inherited method, own
method over both); `inherit_chain ‚Üí 42` (C : B : A ‚Äî multi-level + upcast +
a method inherited through two levels). **60 programs, verify 152/152.**

Honest scope: dispatch is **static**. `virtual`/`override` ‚Äî a real vtable in
the reserved `$header` slot ‚Äî is the next, bigger slice, and it needs back-end
work this didn't: an IR **global** (the per-class vtable) and an **indirect
call** (call a function-pointer value, not a symbol). That's where `$header`
finally earns its keep.

## 27. Virtual dispatch ‚Äî real vtables (commit ffa3d83)

The OO payoff, and the first back-end work since the generics env: `$header` ‚Äî
stored as null since the heap-objects sprint, always *reserved* for this ‚Äî
becomes a live class-vdata pointer. "vtables on MCJIT were hell" (Alban); ORC
was kind, because vtable globals and indirect calls ride the same
`LLJITAddLLVMIRModule` path the externs already use ‚Äî no manual symbol/memory
juggling.

The cut, by crate:
- **IR.** `Module.vtables` (per class: an ordered list of impl symbols);
  `InstKind::GlobalAddr` (address of a named global) and `CallIndirect` (call a
  function-pointer *value*, not a symbol); builder + printer support.
- **LLVM.** Each vtable emits as a constant global array of function pointers
  (`@Dog$vtable = constant [1 x ptr] [@Dog.Speak]`); `GlobalAddr` lowers to the
  global pointer and `CallIndirect` to `build_indirect_call`.
- **Sema.** Record `virtual`/`override` methods; `apply_vtables` composes
  vtables base-first (inherit the base's slots, an `override` replaces a slot's
  impl, a new `virtual` appends a slot) ‚Äî so **slot indices are stable from base
  to derived**. `new` stores the class vtable into `$header`. An instance call
  whose method occupies a slot dispatches indirectly: resolve the slot from the
  *static* type, load the vtable from the object's `$header`, load the slot,
  call it. The slot comes from the static type; the impl from the runtime
  object ‚Äî which is exactly polymorphism.

`virtual_basic ‚Üí 42` (`Animal a = new Dog(); a.Speak()` runs Dog's override, not
Animal's `7`); `virtual_poly ‚Üí 49` (an Animal and a Dog behind one static
`Animal` type dispatch to `7` and `42`). **62 programs, verify 152/152.**

That closes the OO arc: a manual-memory language with single/multi-level
inheritance, field + method reuse, and now true virtual dispatch ‚Äî all on the
keystone that `$header` was always meant to hold.

## Carrying forward

The big OO and generics axes are in. What's left of Beef's deep syntax:

1. **Generic methods** (`Identity<T>`, `Pool.Alloc<T>`) ‚Äî method-level type

3. **`abstract`/interfaces** ‚Äî abstract slots (null vtable entries) + interface

4. **`sizeof(T)`** ‚Äî retires `List<T>`'s over-alloc (parser/`format.rs` coord).

## 28. Ternary `?:` ‚Äî short-circuiting conditional (commit 6d26c07)

A quick pick off the "deep syntax" list that doubled as a *bug fix*: `cond ? a :
b` had been lowering to `undef` (it fell through `expr`'s catch-all), and corlib
`Math`'s rounding quietly depended on it ‚Äî verify-clean but wrong. Now it lowers
like an `if`/`else` joined by a phi: evaluate the condition, branch, evaluate
**only the taken arm**, and merge the two results at a join phi ‚Äî each coerced to
the common type *in its own predecessor block* (the subtlety: the phi needs both
arms at one type, but the coercion has to live in each arm's block, so the arms
evaluate first, then the common type is computed, then each block gets its
coerce + branch). Short-circuit is structural ‚Äî the untaken arm's block never
executes.

`ternary_basic ‚Üí 42`: ordinary selection, plus a short-circuit proof ‚Äî the
else-arm yields `0` while its sibling `100 / x` (with `x == 0`, a divide-by-zero
that would crash if evaluated) is skipped. A SIGFPE would have failed the run, so
green *is* the proof. **63 programs, verify 152/152.**

## 29. `==` / `!=` as value equality for classes with `Equals` (commit e1fcaab)

Lambdas are parser-blocked (the parser parses and *discards* lambda params ‚Äî
`Expr::Lambda { span, body }`), and `sizeof(T)` drops its type the same way;
both need an AST change that ripples into the editor team's `format.rs`, so both
stay deferred. `operator==` likewise isn't a distinct member variant (operators
parse as ordinary methods), so a fully faithful reference-vs-`operator==` `==`
isn't cheap either. What *is* cheap and visible: make `==`/`!=` on two class
references **call the type's `bool Equals(Self)`** when it has one.

The whole feature is a guarded arm in `binary()`: if both operands are `Ref`
*and* the left type defines `Equals`, lower `a == b` to `a.Equals(b)` (and `a !=
b` to its negation, `Equals == false`); otherwise fall through to the existing
pointer/scalar compare. The guard matters twice ‚Äî a `Ref`-vs-`Ref` requirement
means `obj == null` (right side is a `Ptr`, not a `Ref`) keeps **reference**
identity and never calls `Equals(null)`, and the `Equals`-exists check means
classes without one keep pointer identity too. A deliberate kernel pragmatism:
Beef's `==` is reference-default unless an `operator==` exists, but mapping to
`Equals` matches the everyday intent (`String`) with zero operator-method
machinery.

`string_eq_op ‚Üí 42`: two distinct `"hello"` objects compare equal (`a == b`,
+40), `a != c` holds (+2), `a == c` is false. **64 programs, verify 152/152.**

## 30. `abstract` methods ‚Äî body-less virtual slot a derived override fills (commit b4ccff4)

The natural follow-on to vtables, and it turned out to need **zero IR/LLVM
work** ‚Äî the whole feature is one relaxation in method registration. Previously
a body-less, non-extern method was `continue`d (skipped: no sig, no slot). Now
an `abstract` instance method (body `None` + the `abstract` modifier) is treated
almost like a real one: it gets a mangled name `{prefix}{name}`, a callable sig
(so `a.Speak()` resolves), and a recorded vtable slot ‚Äî but because its body is
`None`, `lower_method` still emits no function for it. The base vtable entry
therefore names a symbol that's never defined, and `emit_vtables` *already* maps
a missing function to a null pointer. So the base slot is null for free:

```
@"Animal.$vtable" = constant [1 x ptr] zeroinitializer      ; abstract: null
@"Dog.$vtable"    = constant [1 x ptr] [ptr @Dog.Speak]     ; override fills slot 0
@"Cat.$vtable"    = constant [1 x ptr] [ptr @Cat.Speak]
```

A derived `override` reuses the *inherited* slot index (the slot map is cloned
base‚Üíderived in `compose_vtable`), so its real function lands at the same index
the abstract method reserved. Dispatch is the existing virtual path verbatim ‚Äî a
call through the abstract base type loads the slot from the object's concrete
vtable, reaching the override. The null base slot is never executed because an
abstract class is never instantiated (not enforced yet ‚Äî just never written).

`abstract_method ‚Üí 12`: `Animal a = new Dog(); Animal b = new Cat();
a.Speak() + b.Speak()` = `4 + 8`, each resolved through its own override.
**65 programs, verify 152/152.**

## 31. Computed (get-only) properties ‚Äî `obj.Name` calls `get_Name` (commit 0ad4056)

A property with a body-having `get` accessor now lowers cleanly. Registration
(pre-pass) adds a `get_{Name}` instance method to the type's method table;
`lower_type_at` lowers each computed getter through the *existing* `lower_method`
(accessor body, `this` in scope, no params ‚Äî a getter is just a nullary instance
method). Reading is the elegant part: a member access `obj.Name` first tries
`lvalue` (a field); when there's no field by that name, it falls through to
`try_property_get`, which looks for `get_Name` on the receiver's type and emits
the call. Because getters live in the *inherited* method table, derived classes
see base properties for free ‚Äî no new `StructTable` field, no inheritance plumbing.

The instructive bug: the first cut took the receiver with `self.expr(base)`.
For a class that's the `Ref` (already a pointer ‚Äî fine), but for a **value
struct** it's the aggregate *value*, while the getter's `this` param is a
pointer ‚Äî so verify dropped to **144/152**, and tellingly the eight failures
were all the real-beef *value-type* property tests (Structs, Unions, Variant,
DateTime, DateTimeOffset, TimeZoneInfo, Anonymous, Properties). The fix is to use
`struct_base` (a pointer to the body) exactly as `lower_method_call` does, and
push it only when the getter `is_instance`. The lesson ‚Äî logged before, re-proven
here ‚Äî is that a green hand-written test is *not* the gate; the real-corpus
verify is, and it caught a value-vs-pointer receiver bug the toy `Rect` (a class)
could never surface. Now the verify corpus exercises real property getters
end to end.

`property_get ‚Üí 52`: block-bodied `Area { get { return w*h; } }` = 30,
expression-bodied `Perimeter { get => 2*(w+h); }` = 22. **66 programs, verify
152/152.**

## 35. Generic methods ‚Äî monomorphized per explicit type-arg (commit e5a8db5; supervised agent)

The last big generics frontier: a static `T Identity<T>(T x)` called `Identity<int32>(5)`.
It mirrors the generic-*type* pipeline exactly. A new `gen_method_sigs` (mangled
symbol ‚Üí sig) and `gen_method_monos` (mangled, name, env) sit on `StructTable`;
`index_generic_methods` indexes generic methods by name; a `gmethods` index is
threaded through the *existing* `collect_insts_*` chain so the `Expr::Generic`
arm ‚Äî which already records generic-type instantiations ‚Äî now also calls
`record_method_inst` (deduping via `gen_method_sigs.contains_key`, so no extra
seen-set). `lower_program` emits each monomorph as a static free function via
`lower_method(mangled, ‚Ä¶, None this_ty, env, &[])`, and a `Expr::Generic`-callee
fast path in the `Expr::Call` lowering resolves the mangled symbol and calls it.
The keystone correctness step: generic methods are *skipped* in the ordinary
`Member::Method` registration/lowering (both `fill_members_at` and
`lower_type_at` `continue` on non-empty `generic_params`) ‚Äî otherwise their `T`
lowers with no env and emits bogus IR.

Two honest limitations, both handled by **arity guards** the supervising review
kept (the agent added them after the first cut *panicked* on the real verify
corpus): generic methods are mangled by a *global* name, and the kernel's coarse
`lower_ty_env` can map distinct instantiations to the same key ‚Äî so a real call
site's arg count can disagree with the recorded mono's param count. `record_method_inst`
bails when `argtys.len() != generic_params.len()`, and the call fast path only
fires when `sig.params.len() == args.len()`, otherwise falling through to normal
handling. So same-named generic methods across classes degrade gracefully rather
than crash; a properly type-aware mangling is future work. (This is exactly the
"a green hand-written test isn't the gate, the real corpus is" lesson ‚Äî the toy
`Identity` never collides; the corpus does.)

`generic_method ‚Üí 12` (`Identity<int32>(5) + Identity<int32>(7)`). **71 programs,
verify 152/152, fmt + clippy clean** (a `GenMethodMono` type alias quiets clippy's
complex-type lint on the mono tuple).

## 48. Closures ‚Äî lambdas capture outer locals üêÑŒª (commit 6583f79; 2 research agents)

Beefy-Lisp. Third research-agent round (one agent on capture detection, one on
representation), reconciled + implemented. A lambda that references enclosing
locals is now a closure. At the creation site ‚Äî where the enclosing scope is live
‚Äî `detect_captures` walks the body for free identifiers (names not bound as
params or inner-locals that `lookup` resolves to an outer local), and the lambda
allocates a heap environment `[code_ptr | cap0 | cap1 ‚Ä¶]` (raw `malloc`, uniform
8-byte slots addressed via `elem_addr(env, Ptr, i)`), storing the code pointer at
slot 0 and each captured value after. The lambda *value* is that env pointer. The
captured `(name, type)` list is recorded by symbol in a `RefCell` on
`StructTable` ‚Äî the bridge from the immutable-`&structs` Lowerer (detection, during
lowering) to the emit pass (body). `emit_closure` emits `$lambdaN($self, params‚Ä¶)`
and binds each capture to its env address `$self[i+1]` (reads/writes flow through
it ‚Äî `bind` already accepts any address as a slot). A `function R(P)` local whose
initializer is a capturing lambda is flagged in `Lowerer.closures`; a call through
it loads the code pointer from env slot 0 and passes the env back as the hidden
`$self`. Bare function pointers + non-capturing lambdas are unchanged.

The reconciliation that kept it tractable: `malloc` takes a raw byte count (no
synthesized env struct needed), `bind` already works with a computed address (so
captures need no new binding kind), and the param/`CallIndirect` machinery was
reused. New: capture detection, the env alloc, the closure-call branch.

Scope: by-value capture (the env holds a copy taken at creation). `closure_basic
‚Üí 57` (`get = () => n` captures n=42; `addB = a => a + b` captures b=10 with a
param). **88 programs, verify 152/152.**

## 50. Algebraic data types ‚Äî payload enums + match with binding üß¨ (commit 1d465c4; 2 research agents)

The ML/Rust leg. A payload-bearing `enum` (`enum IntOpt { Some(int32), None }`)
is now a tagged union you can construct and destructure. Two parallel research
agents mapped the two halves ‚Äî value-struct *value* mechanics (alloca ‚Üí store
fields ‚Üí load the aggregate) and the switch/enum-flow extension points ‚Äî and I
reconciled + implemented.

Representation: a payload enum is reclassified as a **value struct**
`{$disc:i32, $p0, $p1, ‚Ä¶}` ‚Äî field 0 the discriminant, payload slots sized to the

member/signature fill, which is the bug that cost the most to find: register it
after, and `Switch1(Shape)`'s recorded param sig is still the `i32` int-backed
fallback while the function *definition*, lowered later, uses the struct ‚Äî so the

`call ‚Ä¶(i32 undef)`. Registering before the fill makes call site and definition
agree (`fill_type_struct` skips enums, so it never clobbers the synthetic fields).

Construction: `IntOpt.Some(5)` (qualified) and target-typed `.Some(5)` / bare

discriminant + each payload field, load the aggregate. `match`: when the
scrutinee's type is a payload-enum struct, `lower_enum_match` spills it, loads

pattern's `let` names in a fresh arm scope (the ¬ßa1a6b1e parser fix keeps the
bound name's span, so `case .Some(let v)` carries `v`). It mirrors the value
switch's block/`break` structure; a patternless arm is the default.

payload layouts become structs. Real-beef enums with a *heterogeneous* slot ‚Äî
`ETest { A(int), B(float) }`, `EnumA { B(float,float), C(String) }` ‚Äî can't share
one typed slot, so they stay int-backed (exactly as before). A real size-based
union (a byte blob reinterpreted per case), generic payload enums (`Opt<T>`), and

`intopt_match ‚Üí 55` (`Some(42)` binds v=42; `None` ‚Üí +13). **90 programs, verify

## 52. The `case` operator + methods on payload enums (commit ec2b568)

Two follow-ons from ¬ß50‚Äì51 land together because the second needs the first.

**The `case`-test operator.** `x case .Some(let v)` was parsed (`BinOp::Case`,
since ¬ß04's pattern work) but unhandled in lowering ‚Äî it fell through to `undef`.
It now lowers as a *boolean expression that also binds*: one arm of
`lower_enum_match` turned inside-out. `case_test` resolves the scrutinee to an
(enum id, body-address) ‚Äî a value scrutinee (`Struct(id)`) spills to a slot, a
pointer one (`Ref(id)`) is already the address ‚Äî then compares `$disc` to the
named case and binds each payload field to its `let`-name *in the current scope*,
so the guarded branch reads them: `if (opt case .Some(let v)) { use v; }`. The
binding is a stable slot address, valid even on a miss; reading it is only
meaningful when matched (Beef's contract). Slightly broader than Beef (the bind
outlives the `then` block), harmless because the value is garbage unless matched.

**Methods on payload enums.** Previously a method-bearing enum stayed int-backed
(`enum_is_simple` = cases-only), stranding its methods. Now `enum_is_layoutable`
(no base; members are cases *or* methods) gates the *non-generic* reclassification,
while `enum_is_simple` still gates the *generic* monomorphization path ‚Äî so
corlib `Result<T> : IDisposable` (has a base) stays int-backed exactly as before.
The reclassified enum's synthetic `{$disc, $p0‚Ä¶}` layout must survive method
registration, so `fill_members_at` grew a `fill_layout` flag: the new
enum call passes `false` (register ctors/methods, skip the field/‚Äãbase refill that
would clobber the union). Body emission was already kind-agnostic
(`lower_type_at` emits any `Member::Method`), so `this` flows in as a `Ref(enum
id)` and the `case`-test's pointer arm destructures it ‚Äî which is exactly how
`Option`'s `GetValueOrDefault` is written.

`if_case_bind` (the operator in an `if`) and `enum_method` (`a.GetOr(0) +
b.GetOr(5)` ‚Üí 47, body `if (this case .Some(let v)) return v;`) both run.
**93 programs, verify 152/152.** Honest scope: `not case` still drops its
negation (parser ¬ß04), and properties-on-enums aren't layoutable yet (methods
only). Target-typed `.Some(x)` and `Result<T,E>` remain the open ADT follow-ons.

## 64. Heterogeneous payload enums + generic `Result<T,E>` (commit b3d5090)

(`Ok(int32)` vs `Err(bool)`) used to stay int-backed ‚Äî the layout demanded every
case agree on each slot's type. The unlock was noticing that construction and
matching already use each case's *own* payload type for the store/load, so a slot
only needs to *reserve enough bytes*; under opaque pointers the store width
follows the value, not the slot's declared type. So a shared `payload_enum_slots`
now sizes each heterogeneous position to its widest scalar member (the homogeneous
fast-path is unchanged, and a heterogeneous *struct* payload still bails to
int-backed ‚Äî can't size it without DataLayout). Both the non-generic
(`register_payload_enum_type`) and monomorph (`register_payload_enum_mono`) paths
share it. This directly unblocks a user-defined generic `Result<T, E>`:
`result_generic` monomorphizes `Result<int32, bool>` (4- vs 1-byte payloads) with
target-typed `.Ok`/`.Err` ‚Üí 42; `enum_heterogeneous` packs int32/float/bool into
one union ‚Üí 142. **108 programs, verify 152/152.** (corlib `Result<T> :
IDisposable` is unaffected ‚Äî it has a base, so `enum_is_layoutable` still keeps it
int-backed.)

## 65. `??` null-coalescing ‚Äî and a latent pointer-comparison bug it exposed (commit 81b9f11)

`a ?? b` lowers as a short-circuiting select on `a == null`: evaluate `a` once,
branch on null to either the `a`-arm (coerce `a`) or the `b`-arm (evaluate +
coerce `b`), join via a phi ‚Äî the `?:` shape, keyed on a null test instead of a
bool. Intercepted in `expr` before `binary` (which would eagerly evaluate `b`).

Wiring it up surfaced a real backend bug: the LLVM `as_int` helper returned `i64
undef` for a *pointer* operand, so **every** reference/pointer comparison
(`a == null`, `a != b`) had been folding to `br i1 undef` and branching at random
‚Äî it just hadn't been exercised, since class value-equality goes through the
`Equals` path (¬ß29). `as_int` now `ptrtoint`s a pointer to its address, so pointer
`icmp` is correct everywhere. `null_coalesce` ‚Üí 12 and `null_check` (`!= null` /
`== null`) ‚Üí 109 lock both in. **110 programs, verify 152/152.** A good reminder
that an untested primitive can be quietly wrong until something leans on it.

## 67. Transitive generic monomorphization ‚Äî `List<T>` inside `Stack<T>` (commit 7a5e3a7)

A user `Stack<T>` backed by a corlib `List<T>` returned garbage: its `items` field
had monomorphized to **`List$p`** ‚Äî `List<T>` with `T` unresolved ‚Üí the `Ptr`
fallback ‚Äî so it stored `int32`s through a pointer-element list. The instantiation
collector only ever scanned source with an *empty* env; it never re-scanned a
mono's own body with that mono's concrete env, so a generic nested inside another
generic never got the outer's `T`.

Fix: thread a type-param `env` through the whole collector family
(`collect_insts_*`, `use_in_type`, `record_inst`, `record_method_inst`). Type args
now resolve through it (`lower_ty_env(.., env)`), and the moment `record_inst`
registers a fresh mono it re-collects *that declaration's body* under the new env ‚Äî
so `Stack<int32>` drags in `List<int32>` (`List$i32`), which in turn drags in
whatever *its* body needs, fully transitively. `seen` is marked before the
recursive scan, so a self-referential generic terminates. `generic_stack`
(push/pop/count over `Stack<int32>`) ‚Üí 150. **112 programs, verify 152/152.** This
is a foundational generics fix ‚Äî any generic container of a generic now works.

## Known limitation ‚Äî JIT can't resolve FP constant-pool symbols

Found while attempting `Console.WriteLine(float)`: the ORC + `RTDyldObjectLinkingLayer`
JIT **fails to materialize `__real@<hex>` symbols** ‚Äî the COFF constant-pool entries
the x86 backend emits for FP constants that aren't immediates (which on x86 is
essentially all of them). Error: `Failed to materialize symbols: { __real@... }`.
Simple float corpus code passes only because LLVM *constant-folds* its FP ops to
integers (e.g. `numeric_cast`); code whose float math depends on a runtime value
can't fold ‚Üí real pool constants ‚Üí JIT failure. **AOT is unaffected** (it runs a
real `TargetMachine` + `link.exe`). Fixing the JIT path needs dedicated work
(RTDyld/COFF mergeable-constant handling, a code-model tweak, or a custom ORC
definition generator that synthesizes `__real@<hex>` from the value encoded in its
name). Until then, JIT/run-corpus features avoid non-foldable runtime FP constants
(so float *printing* and float-heavy `Math` wait). Recorded so it isn't
rediscovered the hard way.

## 71. `ref`/`out` parameters ‚Äî pass-by-reference mutation (commit 8d0e885)

`ref int32 a` was *parsed* (the `ParamModifier` rode along in the model) but never
lowered ‚Äî the modifier was dropped, so `Swap(ref x, ref y)` silently copied. Now a
by-ref param is a **pointer to the caller's storage**, end to end:

- **Call site.** `arg_value` spots an `Expr::Prefix { kw: Ref|Out, operand }` argument
  and, instead of evaluating the operand's *value*, takes its **lvalue address**
  (`self.lvalue(operand)` ‚Äî the same slot pointer locals/fields/indices already
  resolve to) and passes that as a `Ptr`. Wired into all three call shapes: bare-name
  same-type calls, qualified static (`Mut.Double(ref a)`), and instance
  (`c.Bump(ref a)`).
- **Callee.** The param binds **straight to `Value::Param(i)`** (which is now the
  pointer) with no entry spill ‚Äî its bound value-type is the *pointee*, so an ordinary
  `Expr::Ident` read `load`s through it and an assignment `store`s through it. That's
  the whole trick: reuse the existing slot-is-a-pointer convention, but the "slot" is
  the caller's address rather than a fresh `alloca`.
- **Consistency.** `param_ir_ty` centralizes the by-ref‚Üí`Ptr` rule so the IR param
  type, the overload-mangling type-codes, and the call-site coercion all agree (the
  symbol `Swap$...` must hash the same in the index pass and the body pass, or the
  call resolves to a defaulted external).

`ref` and `out` are identical at the IR level; `out`'s definite-assignment rule isn't
enforced (an uninitialized `int32 r;` already gets an `alloca`, so `out r` just writes
into it). The width caveat from the `int32 x = 0` stack-overrun bug applies here too ‚Äî
a `ref int` param over an `int32` arg would `load`/`store` 8 bytes through a 4-byte
slot; the corpus tests match widths exactly, and a real type checker would reject the
mismatch. **119 run-corpus programs** (`ref_param`, `out_param`, `ref_param_methods`);
AOT `.exe` of the method test exits 100.

**Struct- and class-typed by-ref came for free** (added `ref_struct_param`,
`out_class_param` ‚Üí 121). I'd flagged struct `ref` as a follow-on, expecting
`struct_base` to need work ‚Äî but it already has a `Struct(id)` place arm that returns
the place *as* the body pointer, which is exactly the caller's struct address my
binding hands it, so `p.x = ‚Ä¶` writes through to the caller. A class `ref`/`out` binds
a `Ref`-typed place, so reading `c` loads the pointer and `c = new Box()` stores the
fresh reference back into the caller's variable ‚Äî i.e. `out Box b` lets a callee
allocate and hand the object back. The one real remaining gap is `ref` args in
**generic-method** calls (the `Name<T>(args)` path still evaluates args by value).

## 72. Tuples ‚Äî `(A, B)` value types as synthetic structs (commit 5d52f18)

A tuple type was parsed (`TyTuple`) and modeled (`TypeRef::Tuple`) but never lowered.
The realization that made this small: **name the synthetic struct's fields "0", "1",
‚Ä¶** and `t.0`/`t.1` member access, assignment, by-value param passing, and return all
fall out of the *existing* struct machinery ‚Äî `lvalue`'s `Member` arm already finds a
field by `f.name == fname`, and `fname` for `t.0` is literally `"0"`. No new use-site
code; the only feature-specific lowering is building a literal.

The architecture wrinkle is *when* to register the structs. Lowering reads an immutable
`StructTable`, and `m.structs = structs.defs.clone()` snapshots it before bodies lower ‚Äî
so tuple structs must exist by the end of `StructTable::build`, via a pre-pass, not lazily.
`register_tuples` walks every type position (fields, method params/returns, ctor params,
property types, and ‚Äî recursing statement bodies ‚Äî local declarations), keying each
distinct shape by element `type_codes` so `(int32,int32)` everywhere is one `Struct(id)`.
Nested tuples register inner-first so the outer's field type is already a concrete struct.

**The placement bug, and the fix.** My first cut ran the pass at step 4d, *after*
`fill_members_at` built the method-signature table. Result: `MakePair`'s `(int32,int32)`
return resolved to the `Ptr` fallback when its sig was built, so the IR was
`call ptr @MakePair`, Main stored the result nowhere (`store undef, %p`), and
`Sum(undef)` got a junk argument ‚Äî the value silently vanished across both the return
and the argument. Moving the pass to **step 3b** (after monos are collected, before any
field/member fill) fixed both at once: element IR types need only names/kinds (step 1)
and monomorphs (step 3), never filled fields, so 3b is early enough yet has everything it
needs. Lesson reinforced: a type that flows through a *signature* has to be registered
before signatures are built, full stop.

A literal `(a, b)` builds the struct by storing each element into field i: **target-typed**
from a tuple-typed local (so the i64 literals in `(int32,int32) t = (3,4)` coerce to the
i32 fields) or **shape-inferred** from the element types otherwise. **123 run-corpus
programs** (`tuple_basic`, `tuple_param_return` ‚Äî tuple param + return + field mutation);
AOT exe exits 23. Follow-ons: tuples inside **generic templates** (their elements carry
unresolved `T` ‚Äî needs per-monomorph registration) and **destructuring** `(a, b) = pair;`.

(Aside: tuple **destructuring** `(int a, int b) = ‚Ä¶` / `var (a, b) = ‚Ä¶` can't land yet ‚Äî
the *parser* drops the binding names (the target comes back as a `Tuple` of type-idents,
`var (a,b)` as a local literally named `"("`). It needs parser work first, so it's parked.)

## 73. Heap arrays ‚Äî `T[]` with `new T[n]`, indexing, `.Count` (commit 11f5c23)

Arrays were parsed (`TyArray`) and modeled but never lowered. The whole feature turns on
one representation choice: a **length-prefixed block where the array value points at the
elements, not the header**. `new int32[n]` is `malloc(8 + n¬∑sizeof(T))`, the count goes in
the first 8 bytes, and the returned pointer is `block + 8`. The payoff is that the count
lives *behind* the data, so:

- **Indexing is free.** `a[i]`, `a[i] = v`, `a[i] += v`, and even `arr[i].field` all reuse
  the existing typed-pointer index lvalue ‚Äî I just taught `pointer_elem_env` that `T[]`
  records element type `T` (same as `T*`), and `a[i]` indexes from offset 0 like any
  pointer. Zero new index code. (`arr[i].field` works because `lvalue(Index)` already
  returns a `Struct`-typed place that `struct_base` GEPs into.)
- **`.Count`/`.Length`** load `ptr[-1]` (i.e. `elem_addr(ptr, u8, -8)` then `load i64`).
- **`delete a`** frees `ptr - 8`, the real allocation base.

The only thing needing a marker is distinguishing an array from a raw `T*` (both are `Ptr`
with a recorded element type) for `.Count`/`delete` ‚Äî a `HashSet<String> array_locals`,
in the same spirit as `fn_sigs`/`closures`. Element size comes from `size_of_ty`, factored
out of the `sizeof(T)` lowering so a `Pair[]` sizes its elements through the IR `SizeOf`
(LLVM DataLayout) ‚Äî confirmed by `array_struct` (a `Pair[]` with per-element `ps[i].a = ‚Ä¶`
writes, plus an `int64[]` with `+=`). **125 run-corpus programs**; AOT exe exits 90.
Follow-ons: arrays as **params/returns/fields** (the `array_locals` marker is per-body, so
an array crossing a call boundary loses `.Count` until the length-prefix convention is made
type-carried), multi-dim (`T[,]`), and initializer lists `new T[]{‚Ä¶}`.

## 74. `defer` ‚Äî scope-exit statements, LIFO (commit 2b06d26)

`defer stmt;` (run this at scope exit) was parsed but landed in `stmt()`'s `_ => {}`
catch-all ‚Äî silently dropped. The model is a **per-block stack of deferred bodies**,
parallel to the existing variable-scope stack:

- A `Stmt::Block` pushes a defer scope on entry and, on **normal fall-through exit**, runs
  that scope's defers in **reverse declaration order** (LIFO) ‚Äî *before* popping the
  variable scope, so the deferred code can still read the block's locals.
- A `return` evaluates its value **first** (captured as an SSA value, so a deferred
  mutation can't alter what's returned), then unwinds **every** pending defer across all
  open scopes ‚Äî innermost scope first, reverse within each ‚Äî before the `ret`. So an early
  `return` still fires every cleanup.
- The terminator bookkeeping does the rest: once `return` has run the defers and set
  `terminated`, each enclosing block sees `terminated` on the way out and skips its own
  fall-through defer-run, so nothing double-fires.

Deferred bodies are **cloned** (`Stmt` is `Clone`) rather than borrowed ‚Äî the alternative,
storing `&'ast Stmt`, would put an AST lifetime on the `Lowerer`, and the clone is cheap
next to the IR it expands to. Verified by `defer_order` (an inner block's three defers fire
3‚Üí2‚Üí1 building `321`, exercising both `defer { ‚Ä¶ }` and bare `defer stmt;`) and
`defer_return` (a method-scoped `defer b.v += 1` fires past an early `return`; a second
method returns its value *before* its defer bumps the box, proving the capture-before-defer
ordering). **127 run-corpus programs.** Known gap: `break`/`continue` don't yet run the loop
body's defers (they branch without unwinding) ‚Äî the next increment if `defer` inside loops
becomes load-bearing.

## 75. `foreach` over heap arrays (commit eb74602)

Arrays (¬ß73) landed after `foreach` (¬ß41‚Äì42), so an array `iter` fell through to the
List-like path ‚Äî which only matches a `Ref` receiver with `Count()`/`Get()` methods ‚Äî and
