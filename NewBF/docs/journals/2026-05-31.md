# 2026-05-31 — Control flow, the no-GC data-type model, and value structs

A long session spanning three arcs: finishing the control-flow surface, a
research detour into how Beef's core data types and optional GC actually work
(which set our memory-management direction), and then the value-struct layout
sprint. Recording the reasoning, not the diffs.

---

## 1. Finishing control flow (calls → loops → switch)

### Intra-program call resolution: the two guards (commit 3e04c58)

**Why:** bare `Foo(args)` calls were lowering to a defaulted external `i64`
call. To get recursion/mutual-recursion, a call must resolve to the sibling
method's real signature. So I pre-collect a per-type `MethodSig` table *before*
lowering any body (forward refs + recursion resolve for free).

**The challenge:** the naive version regressed the LLVM verify gate from 152/152
to **127/152**. Two distinct causes, two guards:

1. **Arity gate.** Beef has overloads; my table keyed by name (first wins). A
   call whose arg count didn't match the resolved method's arity emitted a call
   with the wrong number of arguments → verifier error. Fix: resolve *only* when
   `arg_count == sig.params.len()`. Coercion then makes every arg exactly the
   declared param type, so the call always matches the signature; an arity
   mismatch falls back to the external varargs call. Got us to 146/152.
2. **Body-less skip.** The remaining 6: I was indexing *every* same-named method
   including body-less ones (extern/abstract). A `Foo(int,int)` extern could win
   the name over the body-having `Foo(int)`, so a 2-arg call resolved to a symbol
   that was never emitted (or had a different arity). Fix: only index methods
   that actually get lowered (skip `MethodBody::None`). Back to 152/152.

**Insight worth keeping:** when lowering with *partial* information (no real
overload resolution yet), the safe move is to resolve only when the resolution
is unambiguous *and* provably type-correct (exact arity → coercion guarantees a
match), and otherwise degrade. "Index only what you emit" is the corollary — a
resolution table must mirror what's actually in the module.

### The loop family + ++/-- (commit 97b18ac)

Added `for`, do-while, `break`, `continue` on a shared **loop-target stack**:
`Vec<(continue_target, break_target)>`, innermost last. `for`'s `continue` must
run the update before re-testing, so it targets a dedicated `for.cont` block,
not the head.

`++`/`--` initially lowered to `undef` — which meant a `for` update of `i++`
silently did nothing and would spin forever. Realisation: inc/dec are
*lvalue* operations (load-modify-store against a slot), not value ops. Wrote an
`incdec` helper; postfix returns the old value, prefix the new. The corpus test
`incr_decr → 577` exists specifically to pin that old-vs-new distinction.

### switch (commit 25c0d17)

Lowered to a cmp/cond-br chain (Beef arms don't fall through → each branches to
one exit). The subtlety: inside a switch arm, `break` exits the *switch* but
`continue` targets the *enclosing loop*. Handled by pushing
`(enclosing_continue, switch_exit)` onto the loop stack for the arm bodies — so
the two keywords correctly diverge.

## 2. Why we stopped to research the data types

The user asked the right question: **with no GC, how are strings and arrays even
supposed to work?** Rather than guess, I fanned out agents over the actual Beef
corlib + compiler source at `E:\beef` and the official docs.

**The finding that reframed everything:** `String`/`T[]`/`List<T>`/`Span<T>` are
**not compiler built-ins** — they're ordinary stdlib classes/structs resting on
**four compiler primitives**: the object header, `new`/`delete`, `scope`, and
**append allocation** (`[AllowAppend]`). This cleanly splits the work into
*compiler/runtime* (our Rust) vs *stdlib* (Beef we compile). No GC anywhere —
ownership is deterministic. Captured in `docs/CORETYPES.md`.

The genuinely clever bit is `String`'s small-string optimisation: one field
`mPtrOrBuffer` is *either* an inline char buffer (chars live at the field's own
address) *or* a real heap pointer, selected by a flag bit. And append allocation
(object header + variable payload in one allocation, via a synthesized
`__CalcAppend` size function + an `__appendIdx` cursor) is the hard primitive —
it's what makes `String` SSO and inline-element `T[]` possible.

## 3. The GC "escape hatch" — and the direction it set

The user was intrigued by Beef's optional-collector hook, having been burned
before by "declaring precise root safepoints all over the code." Researched
Beef's actual GC, then how our own `NewGC` and its two existing clients (NOD,
NCL) integrate. Captured in `docs/GC.md`.

**The asymmetry that dissolves the safepoint pain:** people conflate "precise
GC" with "safepoints everywhere." They're separable. Safepoints buy precise
*roots* (what's live in registers/stack at a suspension point). Precise *heap*
tracing (which fields of an object are pointers) is **static type info — free,
no safepoints**. Beef exploits exactly this: conservative *roots* (scan the
stack/registers word-by-word, validate before trusting), precise *heap* (a
compiler-emitted per-type `GCMarkMembers`). A grep of the entire Beef compiler
for `gcroot`/`statepoint`/`stackmap` returns **zero** hits.

The price of conservative roots is you can't *move* objects (you can't safely
rewrite an ambiguous root) — hence Beef is non-moving.

**Reconciling our NCL scar** ("conservative scanning fails non-deterministically;
precise roots are the answer"): that was right *for a moving collector*. The
fork is upstream of roots — *moving ⇒ precise roots ⇒ safepoints; non-moving (or
pin-only) ⇒ conservative roots suffice.* The safepoint nightmare is a consequence
of choosing to move, not a tax on having a GC.

**Direction decided (recorded in GC.md):** NewBF stays no-GC by default; the
*optional* GC plugs into **NewGC's `conservative-pin` mode** — conservative
roots + precise heap + mostly-copying — which is its *tested default*, built for
"JITs without statepoint stack maps." This **deletes NOD's entire precise-root
apparatus** (liveness fixpoint, spill slab, per-site safepoint tables, reload
plumbing, loop-header polls). NewBF would be NewGC's first pure-conservative
client. The gating constraint: NewGC's pinner rejects interior pointers, and Beef
code holds them freely (`&array[i]`, `Span.mPtr`) — but NewGC already has the
`object_start_at_or_before` function to fix this, just unwired. NCL's deepest
lesson, banked: the *manual* root contract rots (one missed `push_root` =
use-after-GC far from the cause); conservative roots remove that whole class.

## 4. The value-struct layout sprint (commit 81dd1fe)

The IR was scalar-only by design ("aggregate types arrive with the layout
sprint"). This was that sprint: `IrType::Struct` + a field-address instruction
(IR), LLVM struct lowering, and nominal-type tracking + member access (sema).

### Decisions

- **Value structs first**, deferring the heap/`new`/memory-model fork. It's the
  smallest self-contained slice that builds the layout machinery everything else
  needs, and it's fully JIT-testable without an allocator.
- **`StructId` is a `u32` handle**, not an inline layout, so `IrType` stays
  `Copy`. The field list lives in a `Module`-level registry. This kept the
  pervasive `IrType: Copy` assumption intact across the codebase.
- **`ObjectLayout` is one contiguous pointer range, not a bitmap** (this is
  NewGC's model too). For value structs the field list is just ordered fields;
  the constraint bites later when a `BeefLayout` must group reference fields —
  and since *we* control layout, the answer is field reordering, not a bitmap.

### Challenges + solutions

- **Nested / forward / self struct references in LLVM.** A struct field whose
  type is another struct needs both LLVM types to exist. Solution: two passes —
  create all *opaque* struct shells first, then set each body. Standard, but easy
  to get wrong if you build bodies eagerly.
- **The keystone risk: a global verify regression.** Registering structs made
  `lower_ty` map *every* struct-typed reference in the whole corpus from `Ptr`
  to `Struct(id)` — params, locals, returns, fields. That's a huge IR change
  across 152 modules, and ill-typed aggregate IR would fail the verifier. This
  was the moment that could cascade. It **held at 152/152** because: (a) `coerce`
  *always* yields a value of the target type (struct↔scalar with no real cast
  degrades to a typed `undef`), and (b) the "skipped/unsupported → typed undef"
  discipline keeps everything well-typed even when semantically lossy. The
  graceful-degrade design we'd been maintaining all along is what made a
  high-blast-radius change safe.
- **Distinguishing `struct` from `class`.** Critical: only value `struct`s
  become aggregates; a `class` must stay `Ptr` (reference), or class-typed
  locals/params across the corpus would flip type and likely break calls. Keyed
  on `TypeKind::Struct` + non-generic.
- **Borrow-checker choreography in `lvalue`.** Resolving `obj.field` reads the
  struct def (`&self.structs`) *and* emits a `field_addr` (`&mut self.fb`). Had
  to copy the field index + type out of the immutable borrow in an inner block
  before the mutable emit. Small thing, but the kind that stalls you if you
  fight it instead of restructuring.

### Insight

Refactoring `assign` and `incdec` onto a single `lvalue` resolver was the unlock:
once "what's a storable place?" lives in one function (local slot *or* chained
struct field via `fieldaddr`), `p.x = v`, `p.x += i`, `p.x++`, and nested
`o.inner.v` all fell out together. `lvalue` is recursive, so arbitrary chains
work. Proven by `struct_nested → 15` (two-level GEP) and `struct_swap → 21`.

The strategic bonus: the per-type field layout this sprint computes is *exactly*
the precise heap map a future `BeefLayout` hands NewGC. The layout sprint quietly
advanced the GC direction too.

## Carrying forward

Next is the **heap + object model** (`CORETYPES.md` §7 step 2): object header
(`ClassVData*`), `new`→malloc+ctor, `delete`→dtor+free. The known hard part:
member access on a *class reference* needs to recover the nominal class from a
`Ptr` (we deliberately kept class refs as opaque `Ptr` this sprint), so this
sprint introduces the first real "what class does this reference point to?"
type-tracking. Likely model: a class's instance body is a `Struct(id)`
(header word + fields), the reference is a `Ptr`, and the lowerer tracks the
nominal type alongside the `Ptr` so `obj.field` can GEP through the body.
